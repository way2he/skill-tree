## 🎯 09:00-10:00　OpenAI API 完整指南

### ① 流式输出 vs 非流式
- **流式（stream=True）**：基于 SSE 长连接，模型每生成一个 token 就 push 一个，**TTFT（首字延迟）从几秒降到几百毫秒**，用户体验像真人打字
- **非流式**：等完整生成后一次性返回，便于解析 JSON、做后处理
- **怎么选**：
  - 用户 UI / 聊天 / 写作 → 流式
  - Agent 内部链路、批量处理、要解析结构化结果 → 非流式

### ② 错误码处理速查
| 错误码 | 含义 | 应对策略 |
|--------|------|---------|
| 429 | 限流 | 指数退避 + jitter 重试 |
| 401 | 认证失败 | 立即报错，不要重试 |
| 400 | 参数错误 | 立即报错，检查 prompt |
| 5xx | 服务端故障 | 重试 3-5 次后切降级模型 |

### ③ 指数退避 + jitter（面试高频）
```python
delay = min(max_delay, initial_delay * backoff_factor ** attempt) * random.uniform(0.5, 1.5)
```

> **💡 一句话杀手锏**：「不加 jitter 就是制造请求风暴 —— 100 个请求同时 429，1s 后又同时打过去，2s、4s、8s 全同步重试，叫**惊群效应（Thundering Herd）**。加 jitter 让重试时间分散，服务才能平滑恢复。」

## 🎯 11:00-12:00　Token 计费与成本优化

### ① Token 基础常识
- **英文**：约 750 单词 = 1K Token
- **中文**：约 500 汉字 = 1K Token（中文每字约 2 Token，比英文贵 1.5×）
- **输出比输入贵 2-3 倍**（生成成本高于编码）
- **长上下文模型单价更贵**（如 1M 上下文比 128K 贵）

### ② 计费公式（必背）
```
单次成本 = 输入 tokens × 输入单价 + 输出 tokens × 输出单价
```
以 GPT-5.4 为例（$2.5 输入 / $10 输出 per 1M）：
- 输入 1000 tokens + 输出 500 tokens = $0.0025 + $0.005 = **$0.0075**

### ③ 成本优化 5 大招（按降本幅度排序）
1. **换便宜模型**（降本 50-90%）：DeepSeek V4 Flash 比 GPT-5.5 便宜 ~60×
2. **缓存重复请求**（降本 0-90%）：完全相同的 prompt 直接走缓存
3. **上下文压缩**（降本 30-70%）：只保留与当前问题相关的历史
4. **限制输出长度**（降本 20-50%）：`max_tokens` 别给太大
5. **精简 System Prompt**（降本 10-30%）：去掉客套话和废话

### ④ Token 计数代码骨架
```python
import tiktoken
encoding = tiktoken.encoding_for_model("gpt-5.4")
tokens = len(encoding.encode(text))
# 对话计数：每条消息 +3 token 开销，最后再 +3
```

## 🎯 14:00-15:00　统一接口层封装

### ① 为什么必须封装（面试必答）
1. **换模型不改业务代码**，只改配置
2. **统一**错误处理、重试、日志、监控
3. **符合开闭原则**：对扩展开放（新增模型），对修改关闭（不动业务）
4. **方便 A/B 测试**多模型效果

### ② 三大设计模式组合拳
- **策略模式**：每个模型一个策略类（OpenAILLM、DeepSeekLLM）
- **工厂模式**：`LLMFactory.create_llm(type, **kwargs)` 按配置创建
- **适配器模式**：统一不同 SDK 的接口差异

### ③ 抽象基类骨架（必默写）
```python
from abc import ABC, abstractmethod

class BaseLLM(ABC):
    @abstractmethod
    def chat(self, messages, **kwargs) -> str: ...
    @abstractmethod
    def stream_chat(self, messages, **kwargs): ...  # generator

class LLMFactory:
    @staticmethod
    def create_llm(llm_type: str, **kwargs) -> BaseLLM:
        return {"openai": OpenAILLM, "deepseek": DeepSeekLLM}[llm_type](**kwargs)
```

### ④ ABC vs interface（面试追问）
> Python 没有原生 interface，用 `abc.ABC` + `@abstractmethod` 模拟。区别：抽象类**可有默认实现**，子类可选择性覆盖；纯接口只定义签名。BaseLLM 通常只放抽象方法，强制子类实现 → 面向接口编程。

## 🎯 15:00-16:00　降级策略（模型挂了怎么办）

### ① 为什么一定要降级
- OpenAI / DeepSeek 都有过大面积宕机记录
- **单点依赖 = 单点故障**，生产系统必须多模型冗余
- 没降级 = 上游挂了下游全线挂

### ② 降级、熔断、限流三者区别（面试高频）
- **降级 Fallback**：A 模型挂了切 B，保证服务可用
- **熔断 Circuit Breaker**：连续失败 N 次 → 暂停调用一段时间避免雪崩
- **限流 Rate Limit**：QPS 太高主动拒绝，保护下游

### ③ 降级链排序策略
- **按效果**：GPT-5.5 → Claude Opus 4.7 → DeepSeek V4（高优业务）
- **按成本**：DeepSeek V4 Flash → Sonnet 4.6 → GPT-5.4（常规业务）
- **混合**：关键路径按效果，边缘场景按成本

### ④ 熔断器三态（必背）
1. **关闭 Closed**：正常调用，累计失败数
2. **打开 Open**：失败超阈 → 直接拒绝 N 秒
3. **半开 Half-Open**：超时后放 1 个请求试探 → 成功回关闭、失败回打开

> **💡 一句话杀手锏**：「只有开/关会造成**重试风暴**——超时一到全部请求同时打过去又把服务击垮。半开态只放 1 个试水，避免假性恢复造成二次雪崩。Hystrix、Resilience4j、Sentinel 都是这个套路。」

## 🎯 16:00-17:00　评估指标（分类 + 回归）

### ① 一句话区分
- **分类看对错**（是/否、多分类）
- **回归看偏差**（预测值 vs 真实值的距离）

### ② 混淆矩阵与分类四大指标

#### 🔑 公式背诵表
1. **准确率 Accuracy**
   - 公式：(TP+TN) / 全部样本
   - 一句话：整体预测对不对
   - 适用：样本均衡的场景
2. **精确率 Precision**
   - 公式：TP / (TP + FP)
   - 一句话：**别误报**，宁放过不杀错
   - 适用：垃圾邮件、司法判决
3. **召回率 Recall**
   - 公式：TP / (TP + FN)
   - 一句话：**别漏报**，宁杀错不放过
   - 适用：新冠检测、癌症筛查
4. **F1 分数**
   - 公式：2PR / (P+R)（P、R 的调和平均）
   - 一句话：P 和 R 两者都重要时看 F1

### ③ 回归五大指标

1. **MAE 平均绝对误差**
   - 公式：Σ|y-ŷ| / n
   - 一句话：平均偏了多少
   - 业务场景：要可解释、对异常值鲁棒
2. **MSE 均方误差**
   - 公式：Σ(y-ŷ)² / n
   - 一句话：重罚大误差
   - 业务场景：损失函数首选
3. **RMSE 均方根误差**
   - 公式：√MSE
   - 一句话：MSE 开根，与 y 同量纲
   - 业务场景：Kaggle、对外报告
4. **R² 决定系数**
   - 公式：1 - SSres/SStot
   - 一句话：比猜均值好多少
   - 业务场景：衡量模型解释力
5. **MAPE 平均绝对百分比误差**
   - 公式：(1/n) Σ |y-ŷ|/|y| × 100%
   - 一句话：偏了百分之多少
   - 业务场景：与业务方沟通

### ④ 面试高频 3 问（必背）

**Q1：准确率 99.9% 为什么还可能完全没用？**
> 样本不均衡。例：1000 封邮件 999 正常 + 1 垃圾，全预测为正常准确率 99.9%但垃圾邮件召回率 0%。准确率被多数类带偏，遮蔽少数类表现。

**Q2：精确率和召回率为什么是此消彼长？**
> 阈值调低→预测为正例变多→召回率上升，但 FP 也变多→精确率下降。阈值调高反之。要看 FP 和 FN 哪个代价更大来选。

**Q3：F1 为什么用调和平均而不是算术平均？**
> 调和平均会被小的那个拉低。P=0.1, R=1.0，算术平均 0.55、F1=0.18。这保证了只有 P 和 R 都高时 F1 才会高，避免「单边倒」骗 F1。

### ⑤ 对齐 Agent 场景
| Agent 评估问题 | 对应指标 | 举例 |
|--|--|--|
| 工具选择是否正确 | 分类问题 P/R/F1 | 该调搜索调了计算器→FN |
| 参数提取是否准确 | 抽取/召回 | 含日期/金额的 NER |
| RAG 检索是否命中 | 召回率@K | 前 K 个中是否含金文档 |
| RAG 生成是否准确 | 事实一致性 | 需大模型打分或人工判定 |

## 🎯 20:00-21:00　复盘 + Agent 评估方案

### ① Agent 评估四维度（必背）

1. **正确性**：任务完成率、答案准确率（用分类指标 P/R/F1）
2. **工具使用**：工具调用成功率、参数提取准确率、工具选择是否最优
3. **效率**：响应耗时（P50/P95/P99）、Token 消耗、调用次数
4. **用户体验**：人工 1-5 分评分（流畅度、自然度、有用度）

### ② 评估流程五步法

1. **定测试集** — 覆盖正常 / 边界 / 异常，每场景 20-30 例
2. **定指标** — 四维度量化，每个维度有明确公式
3. **自动化跑批** — 每次发版自动评估，输出报告
4. **Bad Case 分类复盘** — 按错误类型归类，针对性修复
5. **持续迭代** — 版本对比，看每个版本的提升

> **💡 一句话总结**：「没有量化就没有优化。说你的 Agent 好，拿出数据来！」
