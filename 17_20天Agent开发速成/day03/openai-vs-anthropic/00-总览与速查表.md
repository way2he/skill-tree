## 二、核心差异速查表

| 对比维度 | OpenAI | Anthropic |
|---------|--------|-----------|
| **SDK 包名 / 版本** | `openai` v2.37.0 | `anthropic` v0.102.0 |
| **安装方式** | `pip install openai` | `pip install anthropic` |
| **Python 版本要求** | 3.9+ | 3.9+ |
| **HTTP 底层** | httpx | httpx |
| **许可证** | Apache-2.0 | MIT |

| **核心 API 入口** | `client.responses.create()` | `client.messages.create()` |

## 三、术语映射表

从 OpenAI 切换到 Anthropic（或反向切换）时，最常遇到的概念名称差异如下：

| OpenAI 术语 | Anthropic 术语 | 说明 |
|------------|---------------|------|
| Chat Completions | Messages | 核心对话 API 的名称 |
| `developer` role | `system` parameter | 系统指令的传递方式不同：OpenAI 用消息角色，Anthropic 用顶层参数 |
| `user` role | `user` role | 用户消息，两侧一致 |
| `assistant` role | `assistant` role | 助手消息，两侧一致 |
| Function Calling | Tool Use | 工具调用的命名 |
| `tools` (function type) | `tools` (function type) | 工具定义数组，结构略有不同 |
| `tool_calls` | `tool_use` content block | 助手返回的工具调用结果 |
| `tool` role (result) | `tool_result` content block | 工具执行结果的回传方式 |
| `response_format` (JSON) | 无直接对应 | Anthropic 通过预填 assistant 消息实现类似效果 |
| Structured Outputs | 无直接对应 | Anthropic 通过 tool_choice + JSON Schema 约束实现 |
| Vision (image input) | Vision (image input) | 图片输入，base64 或 URL 方式类似 |
| Audio / Video input | 不支持 | OpenAI 独有的多模态能力 |
| PDF input | PDF input | Anthropic 原生支持 PDF 文档解析 |
| Realtime API | 不支持 | OpenAI 独有的 WebSocket 实时对话 |
| Extended Thinking | 不支持 | Anthropic 独有的深度推理能力 |
| Prompt Caching | Prompt Caching | Anthropic 的缓存机制更成熟，可显著降低成本 |
| Computer Use | Computer Use | Anthropic 独有的屏幕操控能力 |
| Batches API | 不支持 | OpenAI 独有的批量异步处理 API |
| `max_completion_tokens` | `max_tokens` | 最大输出 token 数的参数名 |
| `temperature` | `temperature` | 采样温度，两侧一致 |
| `top_p` | `top_p` | 核采样，两侧一致 |
| `stop` | `stop_sequences` | 停止序列的参数名不同 |
| `stream` | `stream` | 流式开关，两侧一致 |
| `n` (多候选) | 不支持 | OpenAI 支持一次生成多个候选回复 |

## 四、模型对应关系

| 定位 | OpenAI | Anthropic | 说明 |
|------|--------|-----------|------|
| **旗舰模型** | GPT-5 | Claude 4.6+ (Opus) | 各自最强推理与创作能力，适合复杂任务 |
| **高端模型** | GPT-4o | Claude 3.5 Sonnet | 性能与成本的平衡点，日常开发首选 |
| **中端模型** | GPT-4o-mini | Claude 3.5 Haiku | 轻量快速，适合简单分类、提取等任务 |
| **轻量模型** | o3-mini / o4-mini | Claude 3 Haiku | 极致低成本，适合大规模批量处理 |
| **推理模型** | o3 / o4 | Claude 4.6+ (Extended Thinking) | 专注于深度推理、数学、代码等任务 |
| **实时模型** | GPT-4o Realtime | 不支持 | OpenAI 独有的 WebSocket 实时语音对话 |

> **注意**：模型迭代速度较快，上表中的模型名称和定位可能随时间变化。建议查阅各官方文档获取最新模型列表。

## 五、一句话总结

### OpenAI

**核心优势**：生态最成熟、多模态能力最全面（音频/视频）、Realtime API 支持实时语音对话、结构化输出开箱即用、Batches API 适合大规模离线处理。

**最佳适用场景**：需要音频/视频多模态交互的应用、实时语音对话、需要强结构化输出的生产系统、已有 OpenAI 生态集成（如 Assistants API、Embeddings、Fine-tuning）的项目。

### Anthropic

**核心优势**：超长上下文处理能力突出、Prompt Caching 可显著降低重复请求成本、Extended Thinking 提供可观测的深度推理过程、Computer Use 支持屏幕操控、Claude 系列在长文本理解和代码生成方面表现优异。

**最佳适用场景**：需要处理超长文档（200K-1M tokens）的应用、对 API 成本敏感且请求有大量重复前缀的场景、需要深度推理和可解释思维链的复杂任务、需要 PDF 原生解析的文档处理系统。
