# 第9章：统一抽象层设计

## 概述

在实际的 Agent 开发中，我们几乎不可避免地会面对多 LLM 供应商共存的局面。你可能今天用 OpenAI 的 GPT-4o 做主力推理，明天发现 Anthropic 的 Claude 在长上下文任务上表现更优；又或者在生产环境中，需要一个 Provider 出现故障时自动切换到备用 Provider 以保证服务可用性。

**为什么需要统一抽象层？**

1. **多 Provider 切换**：不同任务适合不同的模型，统一抽象让你可以用同一套业务逻辑自由切换底层模型。
2. **成本优化**：简单任务用便宜模型，复杂任务用强力模型，统一接口让路由策略的实现变得简单。
3. **容灾与高可用**：当某个 Provider 的 API 出现故障或限流时，自动 fallback 到另一个 Provider。
4. **开发效率**：团队不需要为每个 Provider 写一套独立的调用代码，降低维护成本。
5. **A/B 测试**：同一份 Prompt 在不同模型上对比效果，统一抽象让实验框架的搭建更轻松。

本章将设计一套轻量但实用的统一抽象层，覆盖 OpenAI 和 Anthropic 两大主流 Provider，并预留扩展接口。

---

## 设计目标

| 目标 | 说明 |
|------|------|
| **统一接口** | 一套代码调用不同 LLM，业务逻辑与 Provider 实现解耦 |
| **最小抽象** | 不丢失各家特色能力，通过 `raw_response` 保留原始响应供高级用法使用 |
| **可扩展** | 方便接入更多 Provider（如 Google Gemini、DeepSeek、Mistral 等） |
| **类型安全** | 完整的类型注解，利用 Python 3.12+ 的类型系统在编码阶段捕获错误 |
| **同步/异步双支持** | 同时提供 `complete()` 和 `acomplete()` 方法 |

## 核心抽象设计

### 1. 消息格式统一

消息是 LLM 调用的核心数据结构。不同 Provider 的消息格式存在差异（例如 Anthropic 将 system 消息单独提取），我们需要一个统一的中间表示。

```python
"""
统一消息格式定义
将 OpenAI 和 Anthropic 的消息格式抽象为统一的中间表示。
支持文本和多模态内容，兼容工具调用场景。
"""

from typing import Literal, Union, Any
from dataclasses import dataclass, field


@dataclass
class Message:
    """
    统一消息类型

    Attributes:
        role: 消息角色，支持 system/user/assistant/tool 四种
        content: 消息内容，支持纯文本字符串或多模态内容列表
        name: 工具名称，仅当 role 为 tool 时使用
        tool_call_id: 工具调用 ID，仅当 role 为 tool 时使用
        tool_calls: 工具调用列表，仅当 role 为 assistant 时使用
    """
    role: Literal["system", "user", "assistant", "tool"]
    content: str | list[dict]  # 支持纯文本和多模态内容块
    name: str | None = None
    tool_call_id: str | None = None
    tool_calls: list[dict] | None = None


@dataclass
class ToolCall:
    """
    统一工具调用类型

    Attributes:
        id: 工具调用的唯一标识符
        name: 被调用的工具名称
        arguments: 工具调用的参数，JSON 字符串格式
    """
    id: str
    name: str
    arguments: str  # JSON 字符串


@dataclass
class TokenUsage:
    """
    Token 用量统计

    Attributes:
        input_tokens: 输入 token 数量
        output_tokens: 输出 token 数量
        total_tokens: 总 token 数量（部分 Provider 不返回，需手动计算）
    """
    input_tokens: int
    output_tokens: int
    total_tokens: int | None = None

    def __post_init__(self) -> None:
        """如果未提供 total_tokens，自动计算 input + output"""
        if self.total_tokens is None:
            self.total_tokens = self.input_tokens + self.output_tokens
```

### 2. 工具定义统一

```python
"""
统一工具定义
将不同 Provider 的 Function/Tool 定义格式统一为 JSON Schema 驱动的描述。
"""

@dataclass
class ToolDefinition:
    """
    统一工具定义类型

    Attributes:
        name: 工具名称，需符合标识符命名规范
        description: 工具功能的自然语言描述，供 LLM 理解工具用途
        parameters: 工具参数的 JSON Schema 定义，描述参数的类型、约束等
    """
    name: str
    description: str
    parameters: dict  # JSON Schema 格式
```

### 3. LLM 响应统一

```python
"""
统一 LLM 响应格式
将不同 Provider 的响应统一封装，同时保留原始响应供高级用法使用。
"""

from typing import Any


@dataclass
class LLMResponse:
    """
    统一 LLM 响应类型

    Attributes:
        content: 模型生成的文本内容
        tool_calls: 模型发起的工具调用列表，无工具调用时为 None
        usage: Token 用量统计
        raw_response: 原始 Provider 响应对象，用于访问 Provider 特有字段
        model: 实际使用的模型名称
        finish_reason: 结束原因（如 "stop", "tool_calls", "max_tokens" 等）
    """
    content: str
    tool_calls: list[ToolCall] | None = None
    usage: TokenUsage | None = None
    raw_response: Any = None  # 保留原始响应，供高级用法使用
    model: str | None = None
    finish_reason: str | None = None
```

### 4. Provider 适配器接口

```python
"""
Provider 适配器抽象基类
定义所有 LLM Provider 必须实现的统一接口。
"""

from abc import ABC, abstractmethod


class LLMProvider(ABC):
    """
    LLM Provider 抽象基类

    所有具体的 Provider 适配器（OpenAI、Anthropic 等）都必须继承此类
    并实现 complete() 和 acomplete() 方法。
    """

    @property
    @abstractmethod
    def provider_name(self) -> str:
        """返回 Provider 的名称标识，如 'openai'、'anthropic'"""
        ...

    @abstractmethod
    def complete(
        self,
        messages: list[Message],
        tools: list[ToolDefinition] | None = None,
        **kwargs: Any,
    ) -> LLMResponse:
        """
        同步调用 LLM

        Args:
            messages: 统一消息列表
            tools: 可选的工具定义列表
            **kwargs: 传递给底层 Provider 的额外参数（如 temperature、max_tokens 等）

        Returns:
            LLMResponse: 统一的 LLM 响应对象

        Raises:
            LLMError: LLM 调用过程中的错误
        """
        ...

    @abstractmethod
    async def acomplete(
        self,
        messages: list[Message],
        tools: list[ToolDefinition] | None = None,
        **kwargs: Any,
    ) -> LLMResponse:
        """
        异步调用 LLM

        Args:
            messages: 统一消息列表
            tools: 可选的工具定义列表
            **kwargs: 传递给底层 Provider 的额外参数

        Returns:
            LLMResponse: 统一的 LLM 响应对象

        Raises:
            LLMError: LLM 调用过程中的错误
        """
        ...
```

## OpenAI 适配器实现

```python
"""
OpenAI Provider 适配器
将统一消息格式转换为 OpenAI API 格式，并将 OpenAI 响应转换回统一格式。
"""

from typing import Any
from openai import OpenAI, AsyncOpenAI


class OpenAIProvider(LLMProvider):
    """
    OpenAI Provider 适配器

    将统一的 Message/ToolDefinition/LLMResponse 格式
    与 OpenAI SDK 的原生格式进行双向转换。

    Attributes:
        client: OpenAI 同步客户端
        async_client: OpenAI 异步客户端
        model: 默认使用的模型名称
    """

    def __init__(
        self,
        api_key: str | None = None,
        model: str = "gpt-4o",
        base_url: str | None = None,
        organization: str | None = None,
        **kwargs: Any,
    ) -> None:
        """
        初始化 OpenAI Provider

        Args:
            api_key: OpenAI API Key，为 None 时从环境变量读取
            model: 默认模型名称
            base_url: 自定义 API 基础 URL（用于代理或兼容 API）
            organization: 组织 ID
            **kwargs: 传递给 OpenAI 客户端的其他参数
        """
        # 构建客户端初始化参数
        client_kwargs: dict[str, Any] = {"api_key": api_key}
        if base_url is not None:
            client_kwargs["base_url"] = base_url
        if organization is not None:
            client_kwargs["organization"] = organization
        client_kwargs.update(kwargs)

        # 初始化同步和异步客户端
        self.client = OpenAI(**client_kwargs)
        self.async_client = AsyncOpenAI(**client_kwargs)
        self.model = model

    @property
    def provider_name(self) -> str:
        """返回 Provider 名称标识"""
        return "openai"

    def _convert_messages(self, messages: list[Message]) -> list[dict[str, Any]]:
        """
        将统一 Message 列表转换为 OpenAI 格式

        OpenAI 的消息格式与统一格式高度一致，主要需要处理：
        1. 多模态 content 的格式映射
        2. tool_calls 字段的传递

        Args:
            messages: 统一消息列表

        Returns:
            OpenAI 格式的消息字典列表
        """
        openai_messages: list[dict[str, Any]] = []

        for msg in messages:
            converted: dict[str, Any] = {"role": msg.role}

            # 处理 content：纯文本直接传递，多模态内容需转换格式
            if isinstance(msg.content, str):
                converted["content"] = msg.content
            elif isinstance(msg.content, list):
                # 多模态内容：统一格式与 OpenAI 格式基本一致
                converted["content"] = msg.content
            else:
                raise TypeError(
                    f"不支持的 content 类型: {type(msg.content).__name__}"
                )

            # 传递工具相关字段
            if msg.name is not None:
                converted["name"] = msg.name
            if msg.tool_call_id is not None:
                converted["tool_call_id"] = msg.tool_call_id
            if msg.tool_calls is not None:
                converted["tool_calls"] = msg.tool_calls

            openai_messages.append(converted)

        return openai_messages

    def _convert_tools(
        self, tools: list[ToolDefinition] | None
    ) -> list[dict[str, Any]] | None:
        """
        将统一 ToolDefinition 列表转换为 OpenAI 的 tools 格式

        OpenAI 使用 function 类型的 tool 定义，需要包装一层。

        Args:
            tools: 统一工具定义列表

        Returns:
            OpenAI 格式的 tools 列表，输入为 None 时返回 None
        """
        if tools is None:
            return None

        openai_tools: list[dict[str, Any]] = []
        for tool in tools:
            openai_tools.append({
                "type": "function",
                "function": {
                    "name": tool.name,
                    "description": tool.description,
                    "parameters": tool.parameters,
                },
            })

        return openai_tools

    def _parse_response(self, response: Any) -> LLMResponse:
        """
        将 OpenAI 响应对象转换为统一的 LLMResponse

        Args:
            response: OpenAI SDK 返回的 ChatCompletion 对象

        Returns:
            统一的 LLMResponse 对象
        """
        choice = response.choices[0]
        message = choice.message

        # 提取文本内容
        content: str = message.content or ""

        # 解析工具调用
        tool_calls: list[ToolCall] | None = None
        if message.tool_calls:
            tool_calls = [
                ToolCall(
                    id=tc.id,
                    name=tc.function.name,
                    arguments=tc.function.arguments,
                )
                for tc in message.tool_calls
            ]

        # 解析 token 用量
        usage: TokenUsage | None = None
        if response.usage:
            usage = TokenUsage(
                input_tokens=response.usage.prompt_tokens,
                output_tokens=response.usage.completion_tokens,
                total_tokens=response.usage.total_tokens,
            )

        return LLMResponse(
            content=content,
            tool_calls=tool_calls,
            usage=usage,
            raw_response=response,  # 保留原始响应
            model=response.model,
            finish_reason=choice.finish_reason,
        )

    def complete(
        self,
        messages: list[Message],
        tools: list[ToolDefinition] | None = None,
        **kwargs: Any,
    ) -> LLMResponse:
        """
        同步调用 OpenAI Chat Completions API

        Args:
            messages: 统一消息列表
            tools: 可选的工具定义列表
            **kwargs: 额外参数（temperature, max_tokens, top_p 等）

        Returns:
            LLMResponse: 统一的 LLM 响应对象
        """
        # 使用实例默认模型，除非调用时显式指定
        model = kwargs.pop("model", self.model)

        # 转换消息和工具格式
        openai_messages = self._convert_messages(messages)
        openai_tools = self._convert_tools(tools)

        # 构建请求参数
        request_params: dict[str, Any] = {
            "model": model,
            "messages": openai_messages,
        }
        if openai_tools:
            request_params["tools"] = openai_tools
        request_params.update(kwargs)

        # 发起 API 调用
        response = self.client.chat.completions.create(**request_params)

        return self._parse_response(response)

    async def acomplete(
        self,
        messages: list[Message],
        tools: list[ToolDefinition] | None = None,
        **kwargs: Any,
    ) -> LLMResponse:
        """
        异步调用 OpenAI Chat Completions API

        Args:
            messages: 统一消息列表
            tools: 可选的工具定义列表
            **kwargs: 额外参数

        Returns:
            LLMResponse: 统一的 LLM 响应对象
        """
        model = kwargs.pop("model", self.model)

        openai_messages = self._convert_messages(messages)
        openai_tools = self._convert_tools(tools)

        request_params: dict[str, Any] = {
            "model": model,
            "messages": openai_messages,
        }
        if openai_tools:
            request_params["tools"] = openai_tools
        request_params.update(kwargs)

        response = await self.async_client.chat.completions.create(**request_params)

        return self._parse_response(response)
```

## Anthropic 适配器实现

```python
"""
Anthropic Provider 适配器
将统一消息格式转换为 Anthropic API 格式，并将 Anthropic 响应转换回统一格式。

关键差异处理：
1. Anthropic 将 system 消息作为顶层参数，不放在 messages 列表中
2. Anthropic 的 tool_result 使用 user 角色而非 tool 角色
3. Anthropic 的工具调用格式（input_block）与 OpenAI 不同
"""

from typing import Any
from anthropic import Anthropic, AsyncAnthropic


class AnthropicProvider(LLMProvider):
    """
    Anthropic Provider 适配器

    处理 Anthropic Messages API 与统一格式之间的转换，
    特别关注 system 消息提取和 tool_result 格式差异。

    Attributes:
        client: Anthropic 同步客户端
        async_client: Anthropic 异步客户端
        model: 默认使用的模型名称
    """

    def __init__(
        self,
        api_key: str | None = None,
        model: str = "claude-sonnet-4-20250514",
        **kwargs: Any,
    ) -> None:
        """
        初始化 Anthropic Provider

        Args:
            api_key: Anthropic API Key，为 None 时从环境变量读取
            model: 默认模型名称
            **kwargs: 传递给 Anthropic 客户端的其他参数
        """
        client_kwargs: dict[str, Any] = {"api_key": api_key}
        client_kwargs.update(kwargs)

        self.client = Anthropic(**client_kwargs)
        self.async_client = AsyncAnthropic(**client_kwargs)
        self.model = model

    @property
    def provider_name(self) -> str:
        """返回 Provider 名称标识"""
        return "anthropic"

    def _convert_messages(
        self, messages: list[Message]
    ) -> tuple[str | None, list[dict[str, Any]]]:
        """
        将统一 Message 列表转换为 Anthropic 格式

        关键转换逻辑：
        1. 提取所有 system 消息，合并为单个 system 字符串
        2. 将 role="tool" 的消息转换为 Anthropic 的 tool_result 格式
        3. 将 assistant 消息中的 tool_calls 转换为 Anthropic 的 tool_use content block

        Args:
            messages: 统一消息列表

        Returns:
            元组：(system_prompt, anthropic_messages)
            - system_prompt: 合并后的系统提示词，无 system 消息时为 None
            - anthropic_messages: Anthropic 格式的消息列表
        """
        system_parts: list[str] = []
        anthropic_messages: list[dict[str, Any]] = []

        for msg in messages:
            # 处理 system 消息：提取并合并
            if msg.role == "system":
                if isinstance(msg.content, str):
                    system_parts.append(msg.content)
                else:
                    raise ValueError(
                        "Anthropic 的 system 消息仅支持纯文本内容"
                    )
                continue

            # 处理 tool 消息：转换为 Anthropic 的 tool_result 格式
            if msg.role == "tool":
                # Anthropic 使用 user 角色 + tool_result content block
                tool_result_block: dict[str, Any] = {
                    "type": "tool_result",
                    "tool_use_id": msg.tool_call_id,
                    "content": msg.content,
                }
                anthropic_messages.append({
                    "role": "user",
                    "content": [tool_result_block],
                })
                continue

            # 处理 assistant 消息：转换 tool_calls 为 tool_use content block
            if msg.role == "assistant" and msg.tool_calls:
                content_blocks: list[dict[str, Any]] = []

                # 先添加文本内容（如果有）
                if msg.content:
                    content_blocks.append({
                        "type": "text",
                        "text": msg.content,
                    })

                # 将每个 tool_call 转换为 tool_use block
                for tc in msg.tool_calls:
                    import json
                    content_blocks.append({
                        "type": "tool_use",
                        "id": tc["id"],
                        "name": tc["name"],
                        # Anthropic 的 input 是解析后的 dict，不是 JSON 字符串
                        "input": json.loads(tc["arguments"]),
                    })

                anthropic_messages.append({
                    "role": "assistant",
                    "content": content_blocks,
                })
                continue

            # 处理普通 user/assistant 消息（无工具调用）
            converted: dict[str, Any] = {"role": msg.role}
            if isinstance(msg.content, str):
                converted["content"] = msg.content
            elif isinstance(msg.content, list):
                converted["content"] = msg.content
            else:
                raise TypeError(
                    f"不支持的 content 类型: {type(msg.content).__name__}"
                )

            anthropic_messages.append(converted)

        # 合并所有 system 消息
        system_prompt: str | None = None
        if system_parts:
            system_prompt = "\n\n".join(system_parts)

        return system_prompt, anthropic_messages

    def _convert_tools(
        self, tools: list[ToolDefinition] | None
    ) -> list[dict[str, Any]] | None:
        """
        将统一 ToolDefinition 列表转换为 Anthropic 的 tools 格式

        Anthropic 的工具定义直接使用 name/description/input_schema，
        不需要像 OpenAI 那样包装一层 function。

        Args:
            tools: 统一工具定义列表

        Returns:
            Anthropic 格式的 tools 列表
        """
        if tools is None:
            return None

        anthropic_tools: list[dict[str, Any]] = []
        for tool in tools:
            anthropic_tools.append({
                "name": tool.name,
                "description": tool.description,
                "input_schema": tool.parameters,  # 注意：Anthropic 用 input_schema 而非 parameters
            })

        return anthropic_tools

    def _parse_response(self, response: Any) -> LLMResponse:
        """
        将 Anthropic 响应对象转换为统一的 LLMResponse

        Args:
            response: Anthropic SDK 返回的 Message 对象

        Returns:
            统一的 LLMResponse 对象
        """
        # 提取文本内容：遍历 content blocks，收集 text 类型
        text_parts: list[str] = []
        tool_calls: list[ToolCall] = []

        import json

        for block in response.content:
            if block.type == "text":
                text_parts.append(block.text)
            elif block.type == "tool_use":
                # 将 Anthropic 的 tool_use 转换为统一的 ToolCall
                tool_calls.append(
                    ToolCall(
                        id=block.id,
                        name=block.name,
                        # 统一格式使用 JSON 字符串，与 OpenAI 保持一致
                        arguments=json.dumps(block.input),
                    )
                )

        content = "\n".join(text_parts)

        # 解析 token 用量
        usage: TokenUsage | None = None
        if response.usage:
            usage = TokenUsage(
                input_tokens=response.usage.input_tokens,
                output_tokens=response.usage.output_tokens,
            )

        return LLMResponse(
            content=content,
            tool_calls=tool_calls if tool_calls else None,
            usage=usage,
            raw_response=response,
            model=response.model,
            finish_reason=response.stop_reason,
        )

    def complete(
        self,
        messages: list[Message],
        tools: list[ToolDefinition] | None = None,
        **kwargs: Any,
    ) -> LLMResponse:
        """
        同步调用 Anthropic Messages API

        Args:
            messages: 统一消息列表
            tools: 可选的工具定义列表
            **kwargs: 额外参数（temperature, max_tokens, top_p 等）

        Returns:
            LLMResponse: 统一的 LLM 响应对象
        """
        model = kwargs.pop("model", self.model)

        # 转换消息格式，分离 system prompt
        system_prompt, anthropic_messages = self._convert_messages(messages)
        anthropic_tools = self._convert_tools(tools)

        # 构建请求参数
        request_params: dict[str, Any] = {
            "model": model,
            "messages": anthropic_messages,
            "max_tokens": kwargs.pop("max_tokens", 4096),  # Anthropic 必须指定 max_tokens
        }

        # 添加 system 参数（如果有）
        if system_prompt:
            request_params["system"] = system_prompt

        # 添加工具定义（如果有）
        if anthropic_tools:
            request_params["tools"] = anthropic_tools

        request_params.update(kwargs)

        # 发起 API 调用
        response = self.client.messages.create(**request_params)

        return self._parse_response(response)

    async def acomplete(
        self,
        messages: list[Message],
        tools: list[ToolDefinition] | None = None,
        **kwargs: Any,
    ) -> LLMResponse:
        """
        异步调用 Anthropic Messages API

        Args:
            messages: 统一消息列表
            tools: 可选的工具定义列表
            **kwargs: 额外参数

        Returns:
            LLMResponse: 统一的 LLM 响应对象
        """
        model = kwargs.pop("model", self.model)

        system_prompt, anthropic_messages = self._convert_messages(messages)
        anthropic_tools = self._convert_tools(tools)

        request_params: dict[str, Any] = {
            "model": model,
            "messages": anthropic_messages,
            "max_tokens": kwargs.pop("max_tokens", 4096),
        }

        if system_prompt:
            request_params["system"] = system_prompt
        if anthropic_tools:
            request_params["tools"] = anthropic_tools

        request_params.update(kwargs)

        response = await self.async_client.messages.create(**request_params)

        return self._parse_response(response)
```

```python
def create_provider(

    elif provider == "anthropic":

    else:

            f"当前支持的 Provider: openai, anthropic"
        )


def create_provider_from_env(
    provider: str,
    **kwargs: Any,
) -> LLMProvider:
    """
    从环境变量自动读取配置创建 Provider

    自动从以下环境变量读取配置：
    - OPENAI_API_KEY / ANTHROPIC_API_KEY
    - OPENAI_BASE_URL / ANTHROPIC_BASE_URL（可选）

    Args:
        provider: Provider 名称标识符
        **kwargs: 额外参数，会覆盖环境变量中的同名配置

    if provider == "openai":
        # 从环境变量读取 OpenAI 配置
        config: dict[str, Any] = {}
        if "api_key" not in kwargs:
            config["api_key"] = os.environ.get("OPENAI_API_KEY")
        if "base_url" not in kwargs:
            base_url = os.environ.get("OPENAI_BASE_URL")
            if base_url:
                config["base_url"] = base_url
        config.update(kwargs)
        return OpenAIProvider(**config)

    elif provider == "anthropic":
        config = {}
        if "api_key" not in kwargs:
            config["api_key"] = os.environ.get("ANTHROPIC_API_KEY")
        config.update(kwargs)
        return AnthropicProvider(**config)

    else:
        raise ValueError(f"不支持的 Provider: {provider}")
```

### Provider 切换与 Fallback

```python
"""
Provider 切换与 Fallback 示例
"""


class FallbackProvider(LLMProvider):
    """
    带 Fallback 能力的 Provider 包装器

    当主 Provider 调用失败时，自动切换到备用 Provider。

    Attributes:
        primary: 主 Provider
        fallbacks: 备用 Provider 列表，按优先级排序
    """

    def __init__(
        self,
        primary: LLMProvider,
        fallbacks: list[LLMProvider],
    ) -> None:
        """
        初始化 Fallback Provider

        Args:
            primary: 主 Provider，优先使用
            fallbacks: 备用 Provider 列表，主 Provider 失败时按顺序尝试
        """
        self.primary = primary
        self.fallbacks = fallbacks

    @property
    def provider_name(self) -> str:
        """返回 Provider 名称，包含 fallback 信息"""
        fallback_names = [f.provider_name for f in self.fallbacks]
        return f"fallback({self.primary.provider_name} -> {', '.join(fallback_names)})"

    def complete(
        self,
        messages: list[Message],
        tools: list[ToolDefinition] | None = None,
        **kwargs: Any,
    ) -> LLMResponse:
        """
        带自动 Fallback 的同步调用

        依次尝试主 Provider 和备用 Provider，直到成功或全部失败。

        Args:
            messages: 统一消息列表
            tools: 可选的工具定义列表
            **kwargs: 额外参数

        Returns:
            LLMResponse: 第一个成功返回的 Provider 的响应

        Raises:
            Exception: 所有 Provider 都失败时，抛出最后一个异常
        """
        providers = [self.primary] + self.fallbacks
        last_exception: Exception | None = None

        for provider in providers:
            try:
                return provider.complete(messages, tools=tools, **kwargs)
            except Exception as e:
                last_exception = e
                print(
                    f"[Fallback] {provider.provider_name} 调用失败: {e}，"
                    f"尝试下一个 Provider..."
                )
                continue

        # 所有 Provider 都失败
        raise RuntimeError(
            f"所有 Provider 均调用失败: "
            f"{[p.provider_name for p in providers]}"
        ) from last_exception

    async def acomplete(
        self,
        messages: list[Message],
        tools: list[ToolDefinition] | None = None,
        **kwargs: Any,
    ) -> LLMResponse:
        """带自动 Fallback 的异步调用"""
        providers = [self.primary] + self.fallbacks
        last_exception: Exception | None = None

        for provider in providers:
            try:
                return await provider.acomplete(messages, tools=tools, **kwargs)
            except Exception as e:
                last_exception = e
                print(
                    f"[Fallback] {provider.provider_name} 调用失败: {e}，"
                    f"尝试下一个 Provider..."
                )
                continue

        raise RuntimeError(
            f"所有 Provider 均调用失败: "
            f"{[p.provider_name for p in providers]}"
        ) from last_exception


def fallback_example() -> None:
    """演示 Fallback Provider 的使用"""
    # 创建主 Provider（OpenAI）和备用 Provider（Anthropic）
    primary = create_provider("openai", model="gpt-4o")
    fallback = create_provider("anthropic", model="claude-sonnet-4-20250514")

    # 用 Fallback 包装器组合
    provider = FallbackProvider(primary=primary, fallbacks=[fallback])

    messages = [
        Message(role="user", content="解释什么是量子计算。"),
    ]

    # 正常情况下使用 OpenAI，失败时自动切换到 Anthropic
    response = provider.complete(messages)
    print(f"响应来自: {response.model}")
    print(f"回答: {response.content[:200]}...")
```

### 基于任务的路由策略

```python
"""
智能路由示例：根据任务特征自动选择最合适的 Provider
"""

from enum import Enum


class TaskType(Enum):
    """任务类型枚举"""
    SIMPLE_QA = "simple_qa"       # 简单问答
    CODE_GENERATION = "code"       # 代码生成
    LONG_CONTEXT = "long_context"  # 长上下文任务
    REASONING = "reasoning"        # 复杂推理


class SmartRouter:
    """
    智能路由器

    根据任务类型和消息特征，自动选择最合适的 LLM Provider。

    Attributes:
        providers: 已注册的 Provider 字典
        routing_rules: 任务类型到 Provider 的映射规则
    """

    def __init__(self) -> None:
        """初始化路由器"""
        self.providers: dict[str, LLMProvider] = {}
        self.routing_rules: dict[TaskType, str] = {}

    def register(
        self,
        name: str,
        provider: LLMProvider,
        task_types: list[TaskType] | None = None,
    ) -> None:
        """
        注册 Provider

        Args:
            name: Provider 的注册名称
            provider: Provider 实例
            task_types: 该 Provider 擅长的任务类型列表
        """
        self.providers[name] = provider
        if task_types:
            for task_type in task_types:
                self.routing_rules[task_type] = name

    def route(
        self,
        messages: list[Message],
        tools: list[ToolDefinition] | None = None,
        **kwargs: Any,
    ) -> LLMResponse:
        """
        智能路由调用

        根据消息特征判断任务类型，选择最合适的 Provider 进行调用。

        Args:
            messages: 统一消息列表
            tools: 可选的工具定义列表
            **kwargs: 额外参数

        Returns:
            LLMResponse: 被选中 Provider 的响应
        """
        # 简单的任务类型判断逻辑
        task_type = self._classify_task(messages)
        provider_name = self.routing_rules.get(task_type, "default")

        if provider_name not in self.providers:
            raise ValueError(f"未注册的 Provider: {provider_name}")

        print(f"[Router] 任务类型: {task_type.value}, 路由到: {provider_name}")
        return self.providers[provider_name].complete(messages, tools=tools, **kwargs)

    def _classify_task(self, messages: list[Message]) -> TaskType:
        """
        根据消息内容判断任务类型

        这里使用简单的启发式规则，实际项目中可以使用分类模型。

        Args:
            messages: 消息列表

        Returns:
            TaskType: 推断出的任务类型
        """
        # 合并所有消息内容用于判断
        total_length = sum(
            len(msg.content) if isinstance(msg.content, str) else 0
            for msg in messages
        )

        # 获取最后一条用户消息
        last_user_msg = ""
        for msg in reversed(messages):
            if msg.role == "user" and isinstance(msg.content, str):
                last_user_msg = msg.content
                break

        # 长上下文判断：总消息长度超过阈值
        if total_length > 50000:
            return TaskType.LONG_CONTEXT

        # 代码生成判断：包含代码相关关键词
        code_keywords = ["代码", "函数", "实现", "编程", "code", "function", "implement"]
        if any(kw in last_user_msg.lower() for kw in code_keywords):
            return TaskType.CODE_GENERATION

        # 复杂推理判断：包含推理相关关键词
        reasoning_keywords = ["分析", "推理", "论证", "证明", "analyze", "reason"]
        if any(kw in last_user_msg.lower() for kw in reasoning_keywords):
            return TaskType.REASONING

        return TaskType.SIMPLE_QA


def smart_router_example() -> None:
    """演示智能路由器的使用"""
    router = SmartRouter()

    # 注册 Provider 并指定各自擅长的任务类型
    router.register(
        "gpt-4o",
        create_provider("openai", model="gpt-4o"),
        task_types=[TaskType.CODE_GENERATION, TaskType.SIMPLE_QA],
    )
    router.register(
        "claude-sonnet",
        create_provider("anthropic", model="claude-sonnet-4-20250514"),
        task_types=[TaskType.LONG_CONTEXT, TaskType.REASONING],
    )

    # 简单问答 -> 自动路由到 GPT-4o
    response = router.route([
        Message(role="user", content="什么是机器学习？"),
    ])
    print(f"回答: {response.content[:100]}...")

    # 长上下文任务 -> 自动路由到 Claude
    response = router.route([
        Message(role="user", content="请分析这篇长文..." + "x" * 60000),
    ])
    print(f"回答: {response.content[:100]}...")
```

## 设计权衡讨论

### 统一抽象会丢失哪些特色能力？

| 能力 | OpenAI | Anthropic | 统一抽象中的处理 |
|------|--------|-----------|-----------------|
| **Prompt Caching** | 支持（部分模型） | 原生支持，可显著降低成本 | 无法直接使用，需通过 `raw_response` 访问底层 API |
| **Responses API** | OpenAI 独有的结构化输出 API | 不适用 | 完全无法通过统一抽象使用 |
| **Extended Thinking** | 不适用 | Claude 原生支持深度思考模式 | 无法直接启用，需直接调用 Anthropic SDK |
| **Vision 细节控制** | `detail` 参数控制图片精度 | 无此参数 | 需通过 `kwargs` 透传或 `raw_response` |
| **Structured Output** | `response_format` 原生支持 | 需通过 Prompt 约束 | 仅能通过 `kwargs` 透传 |

**核心原则**：统一抽象层的设计目标是覆盖 80% 的通用场景。对于剩余 20% 的 Provider 特色能力，通过以下两种方式保留访问途径：

1. **`**kwargs` 透传**：将 Provider 特有的参数通过 `**kwargs` 传递到底层 SDK。
2. **`raw_response` 保留**：在 `LLMResponse` 中保留原始响应对象，允许高级用户直接访问 Provider 特有字段。

### 何时该用统一抽象，何时该直接用原生 SDK？

**推荐使用统一抽象的场景：**

- 应用需要支持多个 LLM Provider，且核心逻辑相同
- 需要实现 Provider 之间的 Fallback 或负载均衡
- 需要做 A/B 测试，对比不同模型在同一任务上的表现
- 团队中不同成员负责不同 Provider 的接入，需要统一接口规范
- 快速原型开发，后续可能切换底层模型

**推荐直接使用原生 SDK 的场景：**

- 深度依赖某个 Provider 的独有功能（如 Anthropic 的 Prompt Caching、Extended Thinking）
- 对性能有极致要求，不希望引入额外的抽象层开销
- 只使用单一 Provider，没有切换需求
- 需要使用 Provider 最新的、尚未被抽象层支持的特性

### 与 LangChain/LiteLLM 等现有方案的对比

| 维度 | 本方案（自定义抽象） | LangChain | LiteLLM |
|------|---------------------|-----------|---------|
| **复杂度** | 低，约 300 行核心代码 | 高，庞大的框架依赖 | 中等，专注 API 统一 |
| **学习曲线** | 平缓，纯 Python 标准库 | 陡峭，概念众多 | 较平缓 |
| **灵活性** | 高，完全可控 | 中等，受框架约束 | 中等 |
| **Provider 覆盖** | 需自行扩展 | 丰富 | 丰富（100+ Provider） |
| **抽象层次** | 仅 API 调用层 | 覆盖 Agent/RAG/Chain 等 | 仅 API 调用层 |
| **依赖大小** | 极小（仅 SDK） | 大量依赖 | 适中 |
| **适用场景** | 需要精确控制的 Agent 开发 | 快速构建复杂 AI 应用 | 需要多 Provider 统一 + 成本追踪 |

**建议**：

- 如果你的项目是一个**轻量级的 Agent 应用**，只需要统一 API 调用，自定义抽象层是最佳选择——代码量少、完全可控、无隐藏行为。
- 如果你需要**快速构建包含 RAG、Chain、Memory 等完整功能的 AI 应用**，LangChain 提供了开箱即用的组件。
- 如果你需要**支持大量 Provider 并追踪成本**，LiteLLM 是更成熟的选择。
- 在实际项目中，这些方案并不互斥。你可以在自定义抽象层的基础上，针对特定场景引入 LangChain 的组件或 LiteLLM 的路由能力。

## 要点总结

1. **统一抽象层的核心价值**在于解耦业务逻辑与 Provider 实现，使多模型支持、容灾、成本优化成为可能。

2. **消息格式统一**是抽象层最关键的设计。OpenAI 和 Anthropic 的消息格式存在结构性差异（system 消息位置、tool_result 格式），适配器需要仔细处理这些差异。

3. **工具调用统一**需要关注参数格式差异：OpenAI 的 `function.arguments` 是 JSON 字符串，Anthropic 的 `tool_use.input` 是解析后的字典。

4. **`raw_response` 是安全网**——它确保统一抽象不会成为能力的天花板，高级用户始终可以访问 Provider 的原生能力。

5. **Fallback 和路由**是统一抽象最直接的价值体现。`FallbackProvider` 和 `SmartRouter` 展示了如何基于统一接口构建高可用和智能调度的 LLM 调用层。

6. **没有银弹**——统一抽象适合通用场景，但当你深度依赖某个 Provider 的独有功能时，直接使用原生 SDK 仍然是更好的选择。关键是在**标准化**和**灵活性**之间找到适合项目需求的平衡点。

7. **与现有方案的关系**：自定义抽象层、LangChain、LiteLLM 各有适用场景，可以根据项目复杂度和需求灵活选择或组合使用。
