# 第2章 认证与安全

## 概述

认证（Authentication）是调用任何 LLM API 的第一步，也是最容易被忽视的安全环节。OpenAI 和 Anthropic 均以 **API Key** 作为主要认证手段，但在云身份集成、企业级安全能力等方面存在明显差异。本章将从认证方式、代码示例、差异对比、安全最佳实践以及代理与自定义端点五个维度，对两家平台的认证与安全机制进行全面对比。

---

## OpenAI 认证方式

### 1. API Key（环境变量 `OPENAI_API_KEY`）

OpenAI 最常用的认证方式是通过 **Project API Key**。你可以在 [OpenAI Platform > API Keys](https://platform.openai.com/api-keys) 页面创建密钥，并将其设置为环境变量。

```python
import os
from openai import OpenAI

# 方式一：自动读取环境变量 OPENAI_API_KEY（推荐）
client = OpenAI()  # SDK 会自动从 os.environ 中获取密钥

response = client.chat.completions.create(
    model="gpt-4o",
    messages=[{"role": "user", "content": "Hello!"}],
)
print(response.choices[0].message.content)
```

```python
from openai import OpenAI

# 方式二：直接传入 API Key（不推荐，仅用于调试）
client = OpenAI(api_key="sk-xxxxxxxxxxxxxxxxxxxxxxxx")

response = client.chat.completions.create(
    model="gpt-4o",
    messages=[{"role": "user", "content": "Hello!"}],
)
```

### 2. Workload Identity（K8s / Azure / GCP / 自定义 Token Provider）

OpenAI SDK 支持通过 `default_headers` 注入自定义认证头，或借助云平台的 Workload Identity 实现免密钥调用。

#### Azure OpenAI Service

```python
from openai import AzureOpenAI

# Azure OpenAI 使用 API Key 认证
client = AzureOpenAI(
    api_key=os.environ["AZURE_OPENAI_API_KEY"],
    api_version="2024-06-01",
    azure_endpoint="https://<your-resource>.openai.azure.com",
)

response = client.chat.completions.create(
    model="gpt-4o",  # 部署名称，非模型名称
    messages=[{"role": "user", "content": "Hello from Azure!"}],
)
```

#### Azure AD / Entra ID 令牌认证

```python
from azure.identity import DefaultAzureCredential, get_bearer_token_provider
from openai import AzureOpenAI

# 使用 DefaultAzureCredential 获取令牌（适用于 K8s / Managed Identity 等）
credential = DefaultAzureCredential()
token_provider = get_bearer_token_provider(
    credential, "https://cognitiveservices.azure.com/.default"
)

client = AzureOpenAI(
    api_version="2024-06-01",
    azure_endpoint="https://<your-resource>.openai.azure.com",
    azure_ad_token_provider=token_provider,  # 使用 AD 令牌替代 API Key
)

response = client.chat.completions.create(
    model="gpt-4o",
    messages=[{"role": "user", "content": "Hello via Azure AD!"}],
)
```

#### GCP Vertex AI

```python
from google.auth.transport.requests import Request
from google.oauth2 import service_account
from openai import OpenAI

# 加载 GCP 服务账号凭据，获取访问令牌
credentials = service_account.Credentials.from_service_account_file(
    "path/to/service-account.json",
    scopes=["https://www.googleapis.com/auth/cloud-platform"],
)
credentials.refresh(Request())

# 通过自定义 Header 注入 GCP 令牌
client = OpenAI(
    base_url="https://<region>-aiplatform.googleapis.com/v1/projects/<project>/locations/<region>/publishers/google/models",
    default_headers={
        "Authorization": f"Bearer {credentials.token}",
    },
)

response = client.chat.completions.create(
    model="gemini-1.5-pro",
    messages=[{"role": "user", "content": "Hello from Vertex AI!"}],
)
```

#### 自定义 Token Provider（通用模式）

```python
import requests
from openai import OpenAI


def get_custom_token() -> str:
    """从内部认证服务获取访问令牌"""
    resp = requests.post(
        "https://your-internal-auth.example.com/token",
        json={"client_id": "my-app", "client_secret": os.environ["APP_SECRET"]},
        timeout=10,
    )
    resp.raise_for_status()
    return resp.json()["access_token"]


client = OpenAI(
    base_url="https://your-gateway.example.com/v1",
    default_headers={"Authorization": f"Bearer {get_custom_token()}"},
)

response = client.chat.completions.create(
    model="gpt-4o",
    messages=[{"role": "user", "content": "Hello via custom auth!"}],
)
```

## Anthropic 认证方式

### 1. API Key（环境变量 `ANTHROPIC_API_KEY`）

Anthropic 的认证方式与 OpenAI 类似，通过 API Key 进行认证。密钥可在 [Anthropic Console > API Keys](https://console.anthropic.com/settings/keys) 中创建。

```python
import os
from anthropic import Anthropic

# 方式一：自动读取环境变量 ANTHROPIC_API_KEY（推荐）
client = Anthropic()  # SDK 会自动从 os.environ 中获取密钥

message = client.messages.create(
    model="claude-sonnet-4-20250514",
    max_tokens=1024,
    messages=[{"role": "user", "content": "Hello!"}],
)
print(message.content[0].text)
```

```python
from anthropic import Anthropic

# 方式二：直接传入 API Key（不推荐，仅用于调试）
client = Anthropic(api_key="sk-ant-xxxxxxxxxxxxxxxxxxxxxxxx")

message = client.messages.create(
    model="claude-sonnet-4-20250514",
    max_tokens=1024,
    messages=[{"role": "user", "content": "Hello!"}],
)
```

### 2. AWS Bedrock 认证

Anthropic 模型可通过 AWS Bedrock 调用，此时使用 AWS 签名认证（Signature V4），无需 Anthropic API Key。

```python
import os
from anthropic import AnthropicBedrock

# 自动使用 AWS SDK 的凭据链（环境变量 / IAM Role / Profile 等）
client = AnthropicBedrock()  # 自动读取 AWS_DEFAULT_REGION, AWS_ACCESS_KEY_ID 等

message = client.messages.create(
    model="anthropic.claude-sonnet-4-20250514-v1:0",
    max_tokens=1024,
    messages=[{"role": "user", "content": "Hello from Bedrock!"}],
)
print(message.content[0].text)
```

### 3. Google Cloud Vertex AI 认证

```python
from anthropic import AnthropicVertex

# 自动使用 Google Cloud Application Default Credentials (ADC)
client = AnthropicVertex(
    project_id="your-gcp-project",
    region="us-east5",
)

message = client.messages.create(
    model="claude-sonnet-4@20250514",
    max_tokens=1024,
    messages=[{"role": "user", "content": "Hello from Vertex AI!"}],
)
print(message.content[0].text)
```

## 认证差异对比表

| 对比维度 | OpenAI | Anthropic |
|---------|--------|-----------|
| **主要认证方式** | Project API Key | API Key |
| **环境变量名** | `OPENAI_API_KEY` | `ANTHROPIC_API_KEY` |
| **默认密钥获取** | SDK 自动从 `os.environ` 读取同名变量 | SDK 自动从 `os.environ` 读取同名变量 |
| **Azure 集成** | 原生支持 `AzureOpenAI` 客户端，支持 Azure AD / Entra ID 令牌 | 不支持 Azure（需通过 AWS Bedrock 或 GCP Vertex） |
| **AWS 集成** | 无原生支持（需自定义 Header 注入） | 原生支持 `AnthropicBedrock` 客户端，使用 AWS 签名认证 |
| **GCP 集成** | 无原生支持（需自定义 Header 注入） | 原生支持 `AnthropicVertex` 客户端，使用 ADC 认证 |
| **OAuth 支持** | 仅 Azure OpenAI 支持 Azure AD OAuth | 不支持 OAuth |
| **自定义 Token Provider** | 通过 `default_headers` 或 `azure_ad_token_provider` 注入 | 通过 `AnthropicBedrock` / `AnthropicVertex` 子类实现 |
| **多密钥 / 项目级隔离** | 支持 Project 级别密钥，可绑定权限范围 | 仅支持账户级密钥，无项目隔离 |
| **密钥前缀** | `sk-` | `sk-ant-` |

## 安全最佳实践（通用）

以下安全实践适用于 OpenAI 和 Anthropic（以及任何第三方 API 调用场景）。

### 1. 使用环境变量 / `.env` 文件

```bash
# .env 文件（切勿提交到版本控制）
OPENAI_API_KEY=sk-xxxxxxxxxxxxxxxxxxxxxxxx
ANTHROPIC_API_KEY=sk-ant-xxxxxxxxxxxxxxxxxxxxxxxx
```

```python
# Python 中使用 python-dotenv 加载
from dotenv import load_dotenv

load_dotenv()  # 从 .env 文件加载环境变量到 os.environ
```

确保 `.env` 文件已添加到 `.gitignore`：

```gitignore
# .gitignore
.env
.env.local
.env.*.local
```

### 2. 不要硬编码密钥

```python
# 错误示范 —— 密钥硬编码在源码中
client = OpenAI(api_key="sk-xxxxxxxx")  # 绝对不要这样做！

# 正确示范 —— 从环境变量读取
import os
from openai import OpenAI

client = OpenAI(api_key=os.environ["OPENAI_API_KEY"])
```

### 3. 密钥轮换

- **定期轮换**：建议每 90 天轮换一次 API Key
- **泄露后立即撤销**：一旦发现密钥泄露，立即在控制台撤销并生成新密钥
- **多环境隔离**：开发、测试、生产环境使用不同的 API Key
- **OpenAI**：支持在 Project 级别创建多个密钥，便于轮换
- **Anthropic**：支持创建多个密钥，但无项目级别隔离

```python
# 密钥轮换示例：使用密钥别名，便于切换
import os

# 通过环境变量切换密钥，无需修改代码
API_KEY = os.environ["LLM_API_KEY"]  # 运维层面切换即可
```

### 4. 权限最小化

- **OpenAI**：创建 Project 级别密钥时，仅授予必要的权限（如仅允许特定模型访问）
- **Anthropic**：目前不支持细粒度权限控制，建议通过网关层做限流和访问控制
- **云平台**：使用 IAM Role / Managed Identity 时，仅授予最小必要权限

```python
# 使用 OpenAI Project 级别密钥（限定模型和权限范围）
# 在 OpenAI Platform > Projects > API Keys 中创建限定权限的密钥
from openai import OpenAI

client = OpenAI(
    api_key=os.environ["OPENAI_PROJECT_KEY"],
    project="proj_abc123",  # 限定到特定项目
)
```

### 5. 请求日志脱敏

在记录 API 请求日志时，务必对密钥和敏感信息进行脱敏处理。

```python
import logging
from openai import OpenAI

logger = logging.getLogger(__name__)


def mask_api_key(key: str) -> str:
    """
    对 API Key 进行脱敏处理，仅保留前缀和后4位。

    Args:
        key: 原始 API Key

    Returns:
        脱敏后的 API Key，格式如 sk-...xxxx
    """
    if not key or len(key) < 8:
        return "***"
    return f"{key[:7]}...{key[-4:]}"


# 在日志中记录脱敏后的密钥
api_key = os.environ.get("OPENAI_API_KEY", "")
logger.info("Using API Key: %s", mask_api_key(api_key))

client = OpenAI(api_key=api_key)
```

## 代理与自定义端点

两家 SDK 均支持通过 `base_url` 参数自定义请求端点，这在企业内网代理、API 网关、自托管等场景中非常实用。

### OpenAI 自定义端点

```python
from openai import OpenAI

# 通过代理或网关访问 OpenAI API
client = OpenAI(
    base_url="https://your-proxy.example.com/v1",
    api_key=os.environ["OPENAI_API_KEY"],
    default_headers={
        # 可添加自定义 Header（如认证令牌、追踪 ID 等）
        "X-Request-Id": "req-12345",
    },
)

response = client.chat.completions.create(
    model="gpt-4o",
    messages=[{"role": "user", "content": "Hello via proxy!"}],
)
```

### Anthropic 自定义端点

```python
from anthropic import Anthropic

# 通过代理或网关访问 Anthropic API
client = Anthropic(
    base_url="https://your-proxy.example.com",
    api_key=os.environ["ANTHROPIC_API_KEY"],
    default_headers={
        # 可添加自定义 Header
        "X-Request-Id": "req-12345",
    },
)

message = client.messages.create(
    model="claude-sonnet-4-20250514",
    max_tokens=1024,
    messages=[{"role": "user", "content": "Hello via proxy!"}],
)
```

### 对比说明

| 对比维度 | OpenAI SDK | Anthropic SDK |
|---------|-----------|---------------|
| **自定义端点参数** | `base_url` | `base_url` |
| **默认值** | `https://api.openai.com/v1` | `https://api.anthropic.com` |
| **自定义 Header** | `default_headers` 参数 | `default_headers` 参数 |
| **HTTP 代理** | 支持 `http_client` 传入自定义 `httpx.Client` | 支持 `http_client` 传入自定义 `httpx.Client` |
| **重试配置** | `max_retries` 参数（默认 2） | `max_retries` 参数（默认 2） |
| **超时配置** | `timeout` 参数（默认 10 分钟） | `timeout` 参数（默认 10 分钟） |

### 使用 HTTP 代理示例

```python
import httpx
from openai import OpenAI
from anthropic import Anthropic

# OpenAI：通过 HTTP 代理访问
openai_client = OpenAI(
    base_url="https://api.openai.com/v1",
    api_key=os.environ["OPENAI_API_KEY"],
    http_client=httpx.Client(proxy="http://your-proxy:8080"),
)

# Anthropic：通过 HTTP 代理访问
anthropic_client = Anthropic(
    base_url="https://api.anthropic.com",
    api_key=os.environ["ANTHROPIC_API_KEY"],
    http_client=httpx.Client(proxy="http://your-proxy:8080"),
)
```

## 要点总结

1. **认证方式相似**：OpenAI 和 Anthropic 均以 API Key 为主要认证手段，SDK 均支持自动从环境变量读取密钥。
2. **云集成差异显著**：OpenAI 原生支持 Azure（含 Azure AD），Anthropic 原生支持 AWS Bedrock 和 GCP Vertex AI。选择哪家平台应考虑你现有的云基础设施。
3. **权限粒度不同**：OpenAI 提供 Project 级别的密钥隔离和权限控制，Anthropic 目前仅支持账户级密钥。
4. **安全实践通用**：无论使用哪家平台，都应遵循"环境变量管理、不硬编码、定期轮换、最小权限、日志脱敏"五大原则。
5. **自定义端点灵活**：两家 SDK 均支持 `base_url` 自定义，便于接入企业代理、API 网关或自托管服务。
6. **密钥前缀可辨识**：`sk-` 前缀为 OpenAI，`sk-ant-` 前缀为 Anthropic，在日志和配置中可快速区分。
