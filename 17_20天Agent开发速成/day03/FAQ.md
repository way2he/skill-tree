# 📚 Day03 常见问题深度解答

## ❓ 问题1：流式（stream）和非流式到底是同一个接口吗？只是参数不同？

### 📝 详细解答

**是同一个 HTTP 接口，但底层走的协议形态不一样**。

| 维度 | 非流式 | 流式 |
|------|--------|------|
| 请求参数 | `stream=False` | `stream=True` |
| HTTP 协议 | 普通 POST，一次性返回 JSON | POST 后保持长连接，逐块返回 |
| 返回格式 | 一个完整 JSON 对象 | 多行 `data: {...}\n\n`（SSE 格式） |
| SDK 抽象 | 同步函数返回字符串 | 返回生成器（Generator） |
| 取消方式 | 直接 close 连接 | 也是 close 连接，但已生成的 token 不退费 |

#### 关键点：**已生成的 token 全部计费**

很多人以为「我中途取消了就不算钱」，错。OpenAI/DeepSeek 的计费是按「实际生成」算的，你 close 连接也只能阻止后续生成，前面 push 出来的 token 全要付钱。

#### 一句话总结
- 流式 = 模型「边想边说」
- 非流式 = 模型「想好再说」

## ❓ 问题2：tiktoken 算出来的 Token 数，和官方账单为什么有时候对不上？

### 📝 详细解答

**3 个最常见的原因**：

#### 原因 1：模型 tokenizer 不同
- GPT-3.5 / GPT-4 用 `cl100k_base`（旧）
- GPT-4o / GPT-5.x 用 `o200k_base`（2024 起新主流）
- DeepSeek、Qwen、文心一言用各自的 tokenizer，**tiktoken 算不准**！

国产模型推荐方法：
- 直接调一次 API 看返回的 `usage.prompt_tokens` / `usage.completion_tokens`
- 或用模型厂商提供的官方 SDK 自带的计数函数

#### 原因 2：消息有「固定开销」
每条消息除了 content，还会被 tokenize 出额外的特殊 token：
- 每条消息开销：3 个 token
- 整个对话末尾再加：3 个 token
- 如果有 `name` 字段，再加 1 个 token

100 轮对话光开销就 300+ token，长对话会差距明显。

#### 原因 3：System Prompt 累计
每次调用都把 system prompt 带过去，会重复计费。1K System Prompt × 100 次调用 = 100K Token。

#### 实战建议
- 预估时用 tiktoken 算个大概
- 算成本以 API 返回的 `usage` 字段为准
- 建立日志监控，实时累计每天的 Token 用量

## ❓ 问题3：DeepSeek 用 `base_url` 直接复用 OpenAI SDK，是巧合还是有标准？

### 📝 详细解答

**是有事实标准的**：OpenAI Chat Completions API 已经成了大模型 API 的「事实标准」，绝大多数后起厂商都选择兼容这个接口。

### 兼容 OpenAI SDK 的国产/开源模型

| 模型 | base_url |
|------|----------|
| DeepSeek | `https://api.deepseek.com/v1` |
| 阶跃星辰 | `https://api.stepfun.com/v1` |
| 智谱 GLM | `https://open.bigmodel.cn/api/paas/v4` |
| Moonshot Kimi | `https://api.moonshot.cn/v1` |
| 通义千问（DashScope 兼容模式） | `https://dashscope.aliyuncs.com/compatible-mode/v1` |
| 本地 Ollama | `http://localhost:11434/v1` |
| 本地 vLLM | `http://localhost:8000/v1` |

### 为什么这样设计这么爽？

1. **生态复用**：所有围绕 OpenAI SDK 写的工具（LangChain、LlamaIndex 等）开箱即用
2. **迁移成本低**：换模型只改两行（base_url + api_key），业务代码 0 改动
3. **统一接口层更简单**：本质上 OpenAILLM 这个类就能覆盖大半厂商

### ⚠️ 但要注意

不是所有功能都完全兼容：
- Function Calling 的 JSON Schema 兼容性参差不齐
- 流式输出的边缘 case（如错误恢复）不一定一致
- 部分模型不支持 `logprobs`、`response_format` 等高级参数

> 💡 **面试加分**：能说出「OpenAI API 是大模型领域的 POSIX」会让面试官眼前一亮。

## ❓ 问题4：降级和负载均衡有什么区别？

### 📝 详细解答

很多同学把这两个混着用，其实是**两件事**：

| 维度 | 负载均衡（Load Balance） | 降级（Fallback） |
|------|------------------------|------------------|
| **目的** | 把流量均匀分摊到多个节点，提高吞吐 | 主链路挂了切到备用链路，保住可用性 |
| **触发条件** | 一直在做（每次请求） | 只在主节点失败时触发 |
| **节点关系** | 平等的多个节点 | 有优先级的主备节点 |
| **典型策略** | 轮询、加权、最少连接、一致性哈希 | 主→备→兜底，按优先级试 |
| **本质** | 性能问题 | 可用性问题 |

### 实战中常常组合使用

```
用户请求
  ↓
负载均衡器（10 台 DeepSeek 实例间分流）
  ↓ （全部挂了）
降级到 GPT-5.4
  ↓ （也挂了）
降级到本地 Ollama / 缓存的常见回答
  ↓ （还挂）
返回友好提示「服务繁忙，请稍后再试」
```

### 面试加分话术

> 一句话区分：负载均衡是「**分摊**」，降级是「**兜底**」。一个解决「忙不过来」，一个解决「跑不起来」。生产系统两者都要有，缺一不可。

## ❓ 问题5：F1 Score 为什么用调和平均，不用算术平均？

### 📝 详细解答

**核心原因**：调和平均会被「最小值」拉低，避免「单边倒」骗指标。

#### 反例：用算术平均会怎样

假设一个模型：精确率 P=0.1，召回率 R=1.0（疯狂报警，全报成正例）

| 平均方式 | 公式 | 结果 |
|---------|------|------|
| 算术平均 | (0.1+1.0)/2 | **0.55**（看起来还行） |
| 几何平均 | √(0.1×1.0) | 0.316 |
| 调和平均（F1） | 2×0.1×1.0/(0.1+1.0) | **0.18**（暴露问题） |

#### 数学意义

调和平均的特点是：**对小值非常敏感**，只要 P 或 R 有一个很小，F1 就会被拉到接近那个小值。

这正好符合我们要的「精确率和召回率必须都不能太低」的需求——光会报告全不准（高 R 低 P）或者只敢说几个但都对（高 P 低 R）的模型都不算好模型。

#### 一句话记忆

> F1 = 短板效应的数学化表达。**P 和 R 必须都不掉链子，F1 才会高**。

#### 拓展：Fβ Score

通用形式：`Fβ = (1+β²) × P × R / (β²×P + R)`

- β = 1：F1，P 和 R 同等重要
- β = 2：F2，召回率权重更大（医疗筛查偏好）
- β = 0.5：F0.5，精确率权重更大（垃圾邮件偏好）

## ❓ 问题6：Agent 评估为什么不能只用「用户满意度」一个指标？

### 📝 详细解答

**核心问题**：用户满意度高度主观，且无法定位问题原因。

#### 单看用户满意度的 4 个坑

1. **滞后性**：得收集足够多反馈才能形成判断，迭代慢
2. **不可定位**：用户给 3 分，是回答错了？太慢了？还是工具用错了？说不清
3. **采样偏差**：愿意给反馈的往往是极端用户（特别满意 / 特别不满意）
4. **不可复现**：换批用户结果就变，难以做版本对比

#### 多维度指标的真正价值

```
用户给了 2 分（不满意）
  ↓ 单看满意度：只能干瞪眼
  ↓ 多维度交叉分析：
    - 正确性：90%（没问题）
    - 工具使用：60%（这里出问题了！）
    - 效率：5 秒（没问题）
    - 用户体验：流畅
  ↓ 结论：工具调用错了，针对性优化工具选择
```

#### 面试加分话术

> 用户满意度是「**结果指标**」，多维度指标是「**过程指标**」。结果指标告诉你「好不好」，过程指标告诉你「为什么好/不好」。优化必须靠过程指标，验收才看结果指标。

| 易混点 | 一句话区分 |

| 流式 vs 非流式 | 边想边说 vs 想好再说，**已生成的 token 都要付钱** |
