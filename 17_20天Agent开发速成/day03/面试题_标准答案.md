# ✅ Day03 面试题标准答案

## 基础题

### 第 1 题：流式输出的原理是什么？有什么好处？

#### 标准答案（2 分钟）：

流式输出基于 SSE（Server-Sent Events）技术，服务器和客户端建立长连接，模型生成一个 token 就返回一个 token，不需要等完整生成完才返回。

主要有三个好处：
1. **用户体验好**：用户不需要等几秒钟才看到回复，而是逐字显示，感觉像真人在打字
2. **首字延迟低（TTFT）**：第一个字通常几百毫秒就能出来，用户感知速度快
3. **内存占用小**：不需要在服务器端缓存完整的响应，特别是长文本，能节省很多内存

什么时候用流式什么时候不用？
- 用户界面交互用流式，体验好
- Agent 内部思考用非流式，需要等完整结果才继续下一步

### 第 2 题：1K Token 大概是多少字？中英文有区别吗？Token 是怎么计费的？

#### 标准答案（2 分钟）：

1K Token 对应的字符数中英文不一样：
- 英文：约 750 个单词 = 1K Token
- 中文：约 500 个汉字 = 1K Token，因为中文每个字大概占 2 个 Token

所以同样长度的中文内容，Token 消耗是英文的 1.5 倍左右。

计费方式：
1. 输入 Token 和输出 Token 分开计费
2. 输出 Token 通常比输入贵 2-3 倍
3. 长上下文模型的 Token 单价更贵，比如 GPT-5.5 1M 上下文超过 272K 时按 2x 输入/1.5x 输出倍率计费

举个例子：
- GPT-5.5：输入 $5 / 1M，输出 $30 / 1M
- GPT-5.4：输入 $2.5 / 1M，输出 $10 / 1M
- Claude Sonnet 4.6：输入 $3 / 1M，输出 $15 / 1M（性价比首选）
- DeepSeek V4 Flash：输入 ¥1 / 1M，输出 ¥2 / 1M
- DeepSeek V4 Pro：输入 ¥3 / 1M，输出 ¥6 / 1M（2.5折优惠期）

### 第 3 题：精确率和召回率的区别是什么？分别在什么场景下使用？

#### 标准答案（3 分钟）：

先搞清楚四个基础概念：TP（真正例）、FP（假正例）、FN（假负例）、TN（真负例）。

精确率 Precision：
```
精确率 = TP / (TP + FP)
```
意思是「你说它是正例的，里面真正是正例的比例」，核心是「别误报」，思想是「宁可放过，不可杀错」。

召回率 Recall：
```
召回率 = TP / (TP + FN)
```
意思是「所有真正是正例的，有多少被你找到了」，核心是「别漏报」，思想是「宁可杀错，不可放过」。

什么时候用哪个？看 FP 和 FN 哪个代价更大：

看重精确率的场景（FP 代价大）：
- 垃圾邮件过滤：把正常邮件判成垃圾代价很大 → 宁放过不杀错
- 司法判决：冤枉好人代价很大 → 宁放过不杀错
- 医疗手术：没病绝不能开刀 → 宁放过不杀错

看重召回率的场景（FN 代价大）：
- 新冠检测：漏掉感染者代价很大 → 宁杀错不放过
- 安全漏洞扫描：漏掉高危漏洞代价很大 → 宁杀错不放过
- 癌症筛查：漏掉癌症患者代价很大 → 宁杀错不放过

如果两者都重要，就用 F1 Score，它是精确率和召回率的调和平均数。

## 进阶题

### 第 4 题：指数退避重试为什么要加 jitter（随机抖动）？不加会有什么问题？

#### 标准答案（3 分钟）：

指数退避加 jitter 是为了避免「惊群效应」（Thundering Herd），这是分布式系统中非常经典的问题。

举个例子：假设 100 个请求同时打到服务器，同时遇到 429 限流错误。
- 如果不加 jitter，所有请求都会在 1 秒后同时重试，然后 2 秒后又同时重试，4 秒后又同时重试...
- 这样会形成请求风暴，一波接一波地打向服务器，本来服务器只是暂时过载，结果被重试请求彻底打垮。

加上随机抖动后，每个请求的重试时间会分散开。比如第一次重试不是精确的 1 秒后，而是 0.5 到 1.5 秒之间的一个随机时间。这样请求就不会同时打过去，服务器能更平滑地恢复。

这个设计思想在整个分布式系统领域都很常见，不止是大模型 API 调用，数据库连接重试、微服务调用重试，都应该加 jitter。

### 第 5 题：为什么要设计统一接口层？直接调用各家 SDK 不行吗？用什么设计模式？

#### 标准答案（3 分钟）：

直接调用各家 SDK 当然可以，但项目大了会有很多问题：
1. 换模型需要改所有业务代码，侵入性太强
2. 每个地方都要自己写错误处理、重试、日志，重复代码多
3. 不方便加监控、统计、限流这些横切关注点
4. 违反开闭原则：每次加新模型都要改业务代码

设计统一接口层的好处：
1. 换模型只需要改一行配置，业务代码完全不用动
2. 统一的错误处理、重试、日志、监控
3. 方便做降级策略，一个模型挂了自动切另一个
4. 符合开闭原则：加新模型只需要加一个新类，不用改现有代码

用什么设计模式？
1. **策略模式**：每个模型一个策略类，实现统一的接口
2. **工厂模式**：根据配置创建对应的模型实例
3. **适配器模式**：把各家不同的 SDK 接口适配成统一的接口

一句话总结：统一接口层让模型的选择变成了配置问题，而不是代码问题。

### 第 6 题：准确率有什么陷阱？举一个准确率很高但模型完全没用的例子。

#### 标准答案（3 分钟）：

准确率最大的陷阱就是**样本不均衡**。

准确率 = 预测对的 / 全部，当样本极度不均衡时，准确率会非常有误导性。

举个经典的例子：
- 1000 个邮件，其中 999 个是正常邮件，1 个是垃圾邮件
- 模型 A：什么都不学，全部预测成正常邮件
- 准确率 = 999 / 1000 = **99.9%**
- 但这个模型**完全没用**，因为它一个垃圾邮件都没抓到

再举个更极端的风控例子：
- 10000 个用户，其中 10 个是欺诈用户
- 模型全部预测成正常用户，准确率 99.9%
- 但欺诈用户一个都没抓到，损失可能几百万

这就是为什么样本不均衡时绝对不能只看准确率，一定要看精确率和召回率，特别是少数类的召回率。

面试官常追问：那样本不均衡怎么办？
1. 重采样：过采样少数类，或欠采样多数类
2. 调整样本权重：少数类的错误惩罚更大
3. 换评估指标：用 F1、AUC 而不是准确率

## 开放题

### 第 7 题：怎么设计一个完整的 Agent 效果评估方案？从哪些维度评估？

#### 标准答案（5 分钟）：

评估 Agent 效果比评估普通分类器复杂得多，我一般会从五个方面设计完整的评估方案：

**第一步：设计测试集**
- 覆盖所有典型场景：正常场景、边界场景、异常场景、极端场景
- 每个场景至少 20-30 个测试用例
- 每个测试用例有明确的评估标准或标准答案
- 测试集一旦确定就不要随便改，方便版本对比

**第二步：定义评估指标体系，从四个维度评估**
1. **正确性维度**：任务完成度、回答准确率、事实错误率
   - 用精确率、召回率、F1 这些指标
   - 大模型自动打分 + 人工抽样校验
2. **工具使用维度**：工具调用成功率、参数正确率、工具选择合理性
   - 统计有多少次工具调用是没必要的，有多少次参数传错了
3. **效率维度**：平均响应时间、Token 消耗量、平均思考步数
   - 效果好但太慢太贵也不行
4. **用户体验维度**：对话流畅度、逻辑清晰度、自然程度
   - 人工评分 1-5 分

**第三步：自动化评估**
- 写脚本批量跑测试集
- 输出标准化的评估报告
- 每次版本迭代都跑同样的测试集，确保优化是真的有效果，不是拆东墙补西墙

**第四步：Bad Case 复盘**
- 把所有失败的 case 整理出来
- 分类分析原因：是 prompt 问题？模型能力问题？工具问题？
- 针对性优化，然后回归测试

**第五步：持续迭代**
- 评估不是一次性的，是持续的过程
- 每次发版都要有评估报告
- 建立版本基线，每个版本都要比上一个版本好

最后补充一点：Agent 评估没有绝对的标准，关键是找到和业务目标对齐的指标。比如客服 Agent 重点看问题解决率，代码 Agent 重点看 bug 发现率和误报率。

## 📝 答题技巧总结

1. **先总后分**：先一句话总结，再分点展开
2. **举例子**：每个概念都配一个真实场景的例子，面试官印象深
3. **结构化**：用第一、第二、第三，逻辑清晰
4. **不贪多**：每个问题说 3-5 个点就够了，说多了反而记不住
5. **实事求是**：不会就说这个我没深入研究过，但我知道大概是这样
