## 🔌 二、大模型 API 核心参数详解

### 2.1 temperature 温度参数（面试 100% 考）

**本质是什么？**
- 控制下一个词的概率分布的「平扁程度」
- temperature = 0 → 概率分布变成尖峰，永远选概率最大的那个词
- temperature = 2 → 概率分布拉平，所有词概率差不多

**怎么选？**

| 场景 | 推荐温度值 | 为什么 |
|------|-----------|--------|
| 代码生成、SQL 生成 | 0.0 - 0.3 | 追求准确，不能胡来 |
| 事实问答、客服回复 | 0.3 - 0.5 | 基本准确，略有变化 |
| 翻译、摘要 | 0.5 - 0.7 | 平衡准确和流畅 |
| 创意写作、头脑风暴 | 0.7 - 1.0 | 追求多样性，越天马行空越好 |

**💡 经验之谈**：
- 大部分场景用 0.7 就挺好，OpenAI 默认值
- 不要设 0！设 0.1 效果更好，完全一样的输出体验不好
- temperature > 1 之后，输出会很乱，容易胡说八道

---

### 2.2 top_p 核采样参数

**本质是什么？**
- 把所有候选词按概率从高到低排，累加概率，直到累加和达到 top_p
- 只在这些词里面选，后面的词直接扔掉

**temperature vs top_p 怎么选？**

官方建议：**两个参数只改一个，不要同时改！**

我的经验：
- 想要「确定性」：调 temperature 到 0.1-0.3
- 想要「不要太离谱，但也不要完全一样」：调 top_p 到 0.8-0.9
- 两个一起改，效果叠加，很难调，不建议

---

### 2.3 其他重要参数

| 参数 | 作用 | 建议 |
|------|------|------|
| `max_tokens` | 最大输出长度 | 一定要设！防止模型无限输出，跑掉你的钱 |
| `stream` | 流式输出 | 用户端一定要开！体验提升 10 倍 |
| `presence_penalty` | 存在惩罚 | -2 到 2，正值鼓励说新话题 |
| `frequency_penalty` | 频率惩罚 | -2 到 2，正值减少重复 |
| `seed` | 随机种子 | 想要可复现的结果时用 |

## ⚡ 三、生产级调用增强（写代码必看）

### 3.1 错误类型与处理策略（血泪教训）

| 错误码 | 类型 | 处理策略 |
|--------|------|---------|
| **429** | 限流 | 指数退避重试，等久一点，别猛打 |
| **5xx** | 服务端错误 | 重试 2-3 次，不行就降级 |
| **401** | Key 错误 | 不要重试！立刻告警，人工处理 |
| **403** | 权限不够 | 不要重试！检查账号权限 |
| **400** | 参数错误 | 不要重试！检查你的代码 |

**不要什么错误都重试！** 4xx 错误重试 100 次也没用，只会浪费资源，还可能被封 IP。

---

### 3.2 指数退避重试最佳实践

**为什么是指数退避，不是固定间隔？**
- 服务端 5xx 通常是负载高，你越频繁打，服务端越忙，越容易挂
- 第一次失败等 1s，第二次等 2s，第三次等 4s
- 给服务端恢复的时间，也给你自己省资源

**完整 Python 代码模板（直接复制用）**：

```python
import openai
import time
from tenacity import (
    retry,
    stop_after_attempt,
    wait_exponential,
    retry_if_exception_type,
)

# 只对这些错误重试
RETRY_EXCEPTIONS = (
    openai.RateLimitError,        # 429 限流
    openai.APIError,              # 5xx 服务端错误
    openai.APITimeoutError,       # 超时
    openai.APIConnectionError,    # 连接错误
)

class LLMClient:
    def __init__(self, api_key: str):
        self.client = openai.OpenAI(api_key=api_key)
    
    @retry(
        stop=stop_after_attempt(3),               # 最多重试 3 次
        wait=wait_exponential(multiplier=1, min=1, max=10),  # 1s → 2s → 4s
        retry=retry_if_exception_type(RETRY_EXCEPTIONS),  # 只重试指定错误
        reraise=True,  # 最后一次还是失败，把异常抛出去
    )
    def chat(
        self,
        messages: list,
        model: str = "gpt-3.5-turbo",
        temperature: float = 0.7,
        stream: bool = False,
        max_tokens: int = 1000,
    ):
        """
        生产级大模型调用函数
        """
        try:
            response = self.client.chat.completions.create(
                model=model,
                messages=messages,
                temperature=temperature,
                stream=stream,
                max_tokens=max_tokens,
            )
            
            if stream:
                return self._handle_stream(response)
            else:
                return self._handle_normal(response)
                
        except openai.AuthenticationError:
            # 401 Key 错误，不要重试
            print("❌ API Key 错误，请检查")
            raise
        except openai.BadRequestError as e:
            # 400 参数错误，不要重试
            print(f"❌ 参数错误: {e}")
            raise
        except Exception as e:
            print(f"❌ 调用失败: {e}")
            raise
    
    def _handle_normal(self, response):
        """处理非流式响应，统计 Token 和成本"""
        input_tokens = response.usage.prompt_tokens
        output_tokens = response.usage.completion_tokens
        total_tokens = response.usage.total_tokens
        
        # 计算成本（这里可以扩展支持多模型）
        cost = self._calculate_cost(input_tokens, output_tokens, response.model)
        
        print(f"✅ 调用成功 | Token: {input_tokens} + {output_tokens} = {total_tokens} | 成本: ${cost:.6f}")
        
        return {
            "content": response.choices[0].message.content,
            "input_tokens": input_tokens,
            "output_tokens": output_tokens,
            "cost": cost,
            "model": response.model,
        }
    
    def _handle_stream(self, response):
        """处理流式响应"""
        for chunk in response:
            if chunk.choices[0].delta.content:
                yield chunk.choices[0].delta.content
    
    def _calculate_cost(self, input_tokens: int, output_tokens: int, model: str) -> float:
        """计算调用成本，单位：美元"""
        # 价格表，定期更新
        PRICES = {
            "gpt-3.5-turbo": {"input": 0.0005 / 1000, "output": 0.0015 / 1000},
            "gpt-4o": {"input": 5.0 / 1_000_000, "output": 15.0 / 1_000_000},
        }
        
        price = PRICES.get(model, PRICES["gpt-3.5-turbo"])
        return input_tokens * price["input"] + output_tokens * price["output"]
```

**💡 工程亮点**：
- ✅ 只重试可重试的错误，不瞎重试
- ✅ 自动 Token 统计和成本计算
- ✅ 流式非流式统一接口
- ✅ tenacity 库，比自己写的重试靠谱 100 倍

### 3.3 多模型统一适配层设计

**为什么需要？**
1. 不要把业务代码和某个厂商绑定，防止被锁死
2. 一家挂了可以自动切另一家，高可用
3. 不同任务用不同模型，省钱

**优雅的设计：面向接口编程**

```python
from abc import ABC, abstractmethod
from dataclasses import dataclass

@dataclass
class LLMResponse:
    """统一的响应格式，不管哪家模型返回的都是这个结构"""
    content: str
    input_tokens: int
    output_tokens: int
    cost: float
    model: str

class BaseLLM(ABC):
    """所有模型都实现这个接口"""
    
    @abstractmethod
    def chat(self, messages: list, **kwargs) -> LLMResponse:
        pass

# ---------- OpenAI 实现 ----------
class OpenAILLM(BaseLLM):
    def __init__(self, api_key: str):
        self.client = openai.OpenAI(api_key=api_key)
    
    def chat(self, messages: list, **kwargs) -> LLMResponse:
        # OpenAI 具体实现
        pass

# ---------- 通义千问实现 ----------
class QwenLLM(BaseLLM):
    def __init__(self, api_key: str):
        # 初始化通义千问客户端
        pass
    
    def chat(self, messages: list, **kwargs) -> LLMResponse:
        # 1. 把通用参数转成千问的参数格式
        # 2. 调用千问 API
        # 3. 把千问的响应转成统一的 LLMResponse
        pass

# ---------- 带自动降级的多模型 ----------
class FailoverLLM(BaseLLM):
    def __init__(self, llms: list[BaseLLM]):
        self.llms = llms  # 按优先级排列
    
    def chat(self, messages: list, **kwargs) -> LLMResponse:
        for i, llm in enumerate(self.llms):
            try:
                return llm.chat(messages, **kwargs)
            except Exception as e:
                print(f"⚠️ 模型 {i} 失败: {e}, 尝试下一个")
                continue
        
        raise RuntimeError("所有模型都失败了！")

# ---------- 使用方式 ----------
llm = FailoverLLM([
    OpenAILLM(api_key="xxx"),    # 主模型
    QwenLLM(api_key="yyy"),      # 备用1
    DeepSeekLLM(api_key="zzz"),  # 备用2
])

# 业务代码完全不用改，自动降级
response = llm.chat([{"role": "user", "content": "你好"}])
```

**这个设计的好处**：
- 业务代码和具体厂商解耦
- 加新模型只需要加一个类，不用改业务代码
- 自动降级，高可用

## 💰 四、成本优化（省下来的都是利润）

### 4.1 Token 是什么？到底怎么计费？

**一句话理解**：
> Token 是大模型的「字」，但不是我们理解的字。1 Token ≈ 0.7 个中文汉字 ≈ 0.75 个英文单词。

**计费公式（必须背下来）**：
```
总成本 = 输入 Token × 输入单价 + 输出 Token × 输出单价
```

⚠️ **重要提醒**：输入和输出是分开计费的！而且输出通常比输入贵 2-3 倍！

**例子**：GPT-3.5-turbo
- 输入：$0.0005 / 1k Token
- 输出：$0.0015 / 1k Token
- 一次对话：2000 输入 + 1000 输出 = 2×0.0005 + 1×0.0015 = $0.0025 ≈ 1 分 8 厘人民币

---

### 4.2 成本优化四板斧（至少省 50%）

#### 🔪 第一板斧：模型分级，能省则省

不要什么都用 GPT-4！

| 任务难度 | 推荐模型 | 成本对比 |
|---------|---------|---------|
| 简单任务：分类、翻译、摘要 | GPT-3.5 / 国产模型 | 1x |
| 中等任务：写代码、推理 | GPT-3.5 16k / Claude 3 | 2-5x |
| 复杂任务：复杂推理、创意 | GPT-4o | 10-20x |

**经验**：80% 的任务用 3.5 就能搞定，只有 20% 真的需要 4。一分级，成本直接降 80%！

---

#### 🔪 第二板斧：控制历史，不要什么都带

很多人写代码：把整个对话历史全塞进去，越长越贵。

**优化方案**：
1. **滑动窗口**：只带最近的 N 轮对话
2. **摘要压缩**：旧的对话让大模型总结成摘要，不用带全文
3. **重要性评分**：给每轮对话打重要性分，不重要的直接删掉

---

#### 🔪 第三板斧：限制输出，不要让模型长篇大论

- `max_tokens` 一定要设！
- 能 100 Token 说明白的，不要让模型输出 1000 Token
- 输出是输入的 3 倍价格！

---

#### 🔪 第四板斧：缓存问答，相同问题不要重复算

- 相同的问题，直接返回缓存的答案
- 简单任务缓存命中率可以到 30%-50%
- 既能省钱，又能提升速度！

## 🚀 五、进阶概念

### 5.1 MCP 模型控制协议

**什么是 MCP？**
Model Control Protocol，模型控制协议。

**为什么需要 MCP？**
现在每家 API 都不一样：
- OpenAI 一套
-  Anthropic 一套
- 百度阿里腾讯每家一套

每家 SDK 都不一样，写业务代码写死你。

**MCP 的目标**：
- 统一的调用接口，写一次代码，所有模型都能用
- 统一的能力发现，自动检测模型支持什么功能
- 统一的中间件：日志、监控、限流、缓存，只写一次

**一句话总结**：MCP 就是大模型界的 JDBC，一套标准，接所有数据库。

---

### 5.2 模型微调：黄金决策树（面试必考）

**灵魂拷问：我需要微调吗？99% 的情况答案是：不需要！**

```
我需要微调吗？
    ↓
能靠提示词 + Few-shot 解决吗？
    ├─ 能 → ❌ 不需要微调
    └─ 不能 → 能靠 RAG 解决吗？
              ├─ 能 → ❌ 不需要微调
              └─ 不能 → 满足以下任意条件吗？
                        ├─ ✅ 成本要求极高，要把 GPT-4 的效果搬到小模型
                        ├─ ✅ 延迟要求极低，<100ms 响应
                        ├─ ✅ 准确率要求 99%+，提示词已经到极限
                        ├─ ✅ 超大量私有格式输出，RAG 搞不定
                        │
                        ├─ 满足任意一个 → ✅ 可以考虑微调
                        └─ 都不满足 → ❌ 还是不需要！
```

**黄金原则：能靠提示词解决的，绝不微调！**

为什么？因为微调：
1. **贵**：标注数据贵，训练也贵
2. **慢**：准备数据 + 训练 + 测试，几周就过去了
3. **效果不一定好**：很多时候优化提示词效果更好

---

#### 微调的前提条件（缺一个都别做）

1. **数据量足够**：至少 100 条高质量标注数据，最好 500-1000 条
2. **数据质量极高**：格式 100% 统一，没有错误，没有噪音
3. **提示词已经到极限**：把提示词优化到极致了，还是达不到要求
4. **有评估标准**：有客观的评估指标，知道怎么算「更好」

---

#### 微调五步法

1. **数据准备**：整理成对话格式 `[{"messages": [...]}]`
2. **数据清洗**：去重、去低质量、格式统一、train/val 拆分
3. **发起微调**：用 OpenAI 或其他平台的微调接口，选基础模型
4. **测试评估**：微调后的模型和原模型做对比测试，看有没有提升
5. **上线部署**：效果好就上线，监控一段时间，不行就回滚

## ⚠️ 六、高频踩坑总结

### ❌ 踩坑 1：什么错误都重试

```python
# ❌ 错误：什么异常都重试，401 错误重试 100 次也没用
try:
    call_api()
except:
    time.sleep(1)
    call_api()  # 没用！
```

✅ **正确做法**：只重试可重试的错误（429、5xx、超时、连接错误），4xx 错误绝对不要重试！

---

### ❌ 踩坑 2：不设 max_tokens

```python
# ❌ 错误：不设最大输出长度，模型可能无限输出，一次调用花掉你几十块
response = client.chat.completions.create(
    model="gpt-4o",
    messages=messages
)
```

✅ **正确做法**：永远设 max_tokens！根据场景设合理值，既防跑飞，又省钱

---

### ❌ 踩坑 3：所有任务都用 GPT-4

很多人图省事，什么任务都用 GPT-4，结果月底一看账单傻眼了。

✅ **正确做法**：模型分级！
- 分类、翻译这种简单活：3.5 搞定
- 写代码、推理这种中等活：3.5 16k 或 Claude
- 特别复杂的才用 GPT-4

成本直接降 80%！

### ❌ 踩坑 4：整个对话历史全塞进去

用户聊了 100 轮，你把 100 轮历史全塞进去，Token 能不贵吗？

✅ **正确做法**：
