## 1️⃣ 协程和线程的区别？分别适用什么场景？

### 📌 核心区别

| 对比维度 | 协程 | 线程 |
|---------|------|------|
| **调度方式** | 用户态调度，程序自己控制 | 内核态调度，操作系统控制 |
| **切换开销** | 极小，相当于函数调用（几十纳秒） | 较大，需要上下文切换（几微秒） |
| **并发数量** | 轻松支持10万+ | 一般几百个就到瓶颈 |

### 🎯 适用场景

**协程适合 IO密集型场景：**
- 调用大模型API
- 批量HTTP请求
- 数据库查询
- 网络IO操作

**线程适合 CPU密集型场景：**
- 大量计算任务
- 模型推理
- 数据处理

## 2️⃣ async/await 到底在做什么？await 到底在等什么？

### 📝 核心概念

**async 关键字：**
- 定义协程函数
- 调用async函数不会立即执行，返回一个**协程对象**

**await 关键字：**
- **让出CPU的信号**：遇到await时，当前协程暂停执行
- 将CPU让给事件循环，去执行其他就绪的协程
- 等待**可等待对象**完成后，再切回继续执行

**await等待的对象：**
- 协程（Coroutine）
- Task对象
- Future对象

本质是等待某个IO操作完成（HTTP响应、sleep时间等）

## 3️⃣ 什么是事件循环？它的工作原理是什么？

### 🧠 核心定义

事件循环是异步编程的**核心调度器**，单线程运行，负责调度所有协程。

### 🔄 工作原理

```
┌─────────────────────────────────────┐
│          事件循环 (Event Loop)       │
├─────────────────────────────────────┤
│  1. 检查 → 2. 执行 → 3. 切换 → 4. 循环│
└─────────────────────────────────────┘
```

**详细流程：**
1. **检查就绪协程**：查看有哪些协程可以执行
2. **执行协程**：运行就绪的协程
3. **遇到await切换**：暂停当前协程，执行下一个就绪协程
4. **循环往复**：持续上述过程

### 💡 核心价值

用单线程模拟并发，利用IO等待时间做其他事情，**避免线程切换开销**。

## 4️⃣ 为什么要用 aiohttp 不用 requests？aiohttp 快在哪里？

### ⚡ 核心差异

| 特性 | requests (同步) | aiohttp (异步) |
|-----|----------------|----------------|
| 执行方式 | 阻塞等待，串行执行 | 非阻塞，并发执行 |
| 资源利用 | 等待时线程闲置 | 等待时执行其他任务 |

### 🚀 性能对比

假设发送100个请求，每个响应1秒：

**同步方式（requests）：**
```
请求1 → 等待1秒 → 请求2 → 等待1秒 → ... → 请求100
总耗时：100秒
```

**异步方式（aiohttp）：**
```
请求1、请求2、...、请求100 同时发送
同时等待响应
总耗时：1-2秒
```

**性能提升：50-100倍**

## 5️⃣ ClientSession 为什么要复用？每个请求新建一个会怎么样？

### 🎯 复用的本质

**TCP连接复用**，避免每次请求都进行三次握手。

### ⏱️ 开销对比

| 操作 | 耗时 |
|-----|------|
| TCP三次握手 | 10-30ms |
| 100个请求新建连接 | 1-3秒额外开销 |
| 复用连接 | 几乎零开销 |

### 💥 不复用的后果

1. **性能下降**：每次都要三次握手
2. **端口耗尽**：大量TIME_WAIT状态占用本地端口
3. **资源浪费**：重复建立和销毁连接

### ✅ 正确做法

```python
async with aiohttp.ClientSession() as session:
    # 所有请求复用同一个session
    await session.get(url1)
    await session.get(url2)
    # session内部自动维护连接池
```

## 6️⃣ 信号量 Semaphore 的工作原理？为什么需要控制并发？

### 🔐 工作原理

信号量本质是一个**许可证计数器**：

```
┌────────────────────────────┐
│      Semaphore (3个许可证)   │
├────────────────────────────┤
│  [可用] [可用] [可用]       │
│    ↓     ↓     ↓          │
│  协程1   协程2   协程3     │
│  执行中  执行中  执行中    │
└────────────────────────────┘
         ↓ 协程4等待
    [占用] [占用] [占用]
         ↓ 协程1完成归还
    [可用] [占用] [占用]
         ↓ 协程4获取执行
```

### 🛡️ 控制并发的原因

1. **服务器限流**：避免触发QPS限制，防止429错误或IP被封
2. **端口限制**：TCP连接数有上限，并发太高会耗尽端口
3. **性能优化**：过高并发会导致每个请求变慢，吞吐量反而下降

## 7️⃣ 指数退避重试为什么是 2 的幂次？固定间隔重试有什么问题？

### 📈 指数退避策略

```
重试间隔：1秒 → 2秒 → 4秒 → 8秒 → ...
```

### 🎯 为什么是2的幂次

| 增长速度 | 效果 |
|---------|------|
| 太慢 | 用户体验差，等待时间过长 |
| 太快 | 相当于没退避，服务器压力大 |
| **2的幂次** | 增长适中，工业界标准（TCP重传也用此策略） |

### 💥 固定间隔的问题

**重试风暴（Retry Storm）：**
- 100个请求同时失败
- 1秒后100个同时重试
- 2秒后又100个同时重试
- 相当于DDoS攻击，导致雪崩

### ✅ 正确的重试策略

```python
# 4xx客户端错误：不重试
if 400 <= status < 500:
    return None

# 5xx服务器错误：指数退避重试
for attempt in range(max_retries):
    delay = 1 * (2 ** attempt)  # 1, 2, 4, 8...
    await asyncio.sleep(delay)
```

## 8️⃣ 不加超时会怎么样？所有外部 IO 都要加超时吗？

### ⚠️ 不加超时的风险

```
请求发送 → 网络卡住 → 永远等待 → 协程挂住 → 系统卡死
```

**真实案例：**
- 大模型API调用卡住几分钟不返回
- 数据库连接超时无响应
- 网络分区导致请求悬停

### ✅ 必须加超时的场景

| 场景 | 推荐超时时间 |
|-----|-------------|
| HTTP请求 | 10-30秒 |
| 大模型API | 30-60秒 |
| 数据库查询 | 5-10秒 |
| Redis调用 | 1-5秒 |

**工程化基本原则：90%的线上故障源于未设置超时**

## 9️⃣ asyncio.create_task() 和直接 await 的区别？

### 🚦 执行方式对比

**直接 await（串行执行）：**
```python
await a()  # 等a完成
await b()  # 再执行b
# 总耗时 = a耗时 + b耗时
```

**create_task（并发执行）：**
```python
task1 = asyncio.create_task(a())  # 立即调度，不等待
task2 = asyncio.create_task(b())  # a和b同时执行
await task1
await task2
# 总耗时 = max(a耗时, b耗时)
```

### 📊 性能对比

| 方式 | 10个大模型调用耗时 |
|-----|-------------------|
| 串行 await | ~100秒 |
| create_task并发 | ~10秒 |

## 🔟 手写：带并发控制、超时、指数退避重试的异步请求函数

### 📝 完整代码

```python
import asyncio
import aiohttp
from aiohttp import ClientTimeout
import logging

logging.basicConfig(level=logging.INFO)
logger = logging.getLogger(__name__)

MAX_CONCURRENT = 3
MAX_RETRIES = 3
INITIAL_DELAY = 1
TIMEOUT = 10

async def fetch_with_retry(session: aiohttp.ClientSession, 
                          url: str, 
                          max_retries: int = MAX_RETRIES) -- dict | None:
    """带指数退避重试的异步请求"""
    for attempt in range(max_retries):
        try:
            async with session.get(url) as response:
                if response.status == 200:
                    logger.info(f"✅ 成功: {url}, 第 {attempt+1} 次尝试")
                    return await response.json()
                else:
                    logger.warning(f"⚠️ 状态码 {response.status}: {url}")
                    if 400 <= response.status < 500:
                        return None

        except Exception as e:
            logger.warning(f"❌ 失败: {url}, 第 {attempt+1} 次尝试, 错误: {str(e)}")

        if attempt < max_retries - 1:
            delay = INITIAL_DELAY * (2 ** attempt)
            logger.info(f"⏳ {delay} 秒后重试...")
            await asyncio.sleep(delay)

    logger.error(f"❌ 最终失败: {url}, 重试了 {max_retries} 次")
    return None

async def main(urls: list[str]) -- list[dict | None]:
    semaphore = asyncio.BoundedSemaphore(MAX_CONCURRENT)
    timeout = ClientTimeout(total=TIMEOUT)

    async def bounded_fetch(url: str) -- dict | None:
        async with semaphore:
            return await fetch_with_retry(session, url)

    async with aiohttp.ClientSession(timeout=timeout) as session:
        tasks = [bounded_fetch(url) for url in urls]
        results = await asyncio.gather(*tasks)
        return results

if __name__ == "__main__":
    import sys
    if sys.platform.startswith('win'):
        asyncio.set_event_loop_policy(asyncio.WindowsSelectorEventLoopPolicy())

    urls = [
        "https://httpbin.org/delay/1",
        "https://httpbin.org/status/500",
        "https://httpbin.org/status/404",
    ]
    asyncio.run(main(urls))
```

### ⭐ 代码亮点

| 特性 | 说明 |
|-----|------|
| ✅ **信号量控制** | BoundedSemaphore限制并发数 |
| ✅ **全局超时** | ClientTimeout防止永久等待 |
| ✅ **指数退避** | 1→2→4秒间隔重试 |
| ✅ **智能重试** | 4xx不重试，5xx和网络错误重试 |
| ✅ **日志完善** | 详细记录每步操作 |
| ✅ **跨平台兼容** | Windows平台特殊处理 |
