# RAG 完整流程详解 🔄

## 什么是 RAG？

### 定义
**RAG = Retrieval-Augmented Generation**（检索增强生成）

用检索到的外部知识来增强大模型的生成能力，让大模型回答更准确、更有依据。

### 为什么需要 RAG？

| 问题 | RAG 的解决方案 |
|------|---------------|
| 大模型知识截止日期 | 检索最新文档，补充最新知识 |
| 幻觉（Hallucination） | 用检索到的知识做依据，减少胡说八道 |
| 领域/私有知识 | 把企业/个人知识库里的内容检索出来 |
| 可追溯性 | 回答可以追溯到原始文档 |

## RAG 完整流程（7 步）

```mermaid
graph TD
    A[原始文档] --&gt; B[文档分块]
    B --&gt; C[文本向量化]
    C --&gt; D[(向量数据库)]
    E[用户查询] --&gt; F[查询向量化]
    F --&gt; G[相似度检索]
    D --&gt; G
    G --&gt; H[重排序]
    H --&gt; I[上下文增强]
    I --&gt; J[大模型生成]
    J --&gt; K[最终回答]
```

## 第 1 步：原始文档准备 📄

### 常见文档类型
| 类型 | 示例 |
|------|------|
| **纯文本** | .txt, .md |
| **结构化文档** | .pdf, .docx, .pptx |
| **网页** | HTML, URL |
| **数据库** | SQL, NoSQL |
| **音视频转文字** | 会议记录、课程视频 |

### 文档预处理要点
- ✅ 格式统一：都转成纯文本
- ✅ 去除噪声：页眉页脚、广告、重复内容
- ✅ 元数据保留：标题、作者、日期、来源
- ✅ 编码统一：UTF-8，防止乱码

---

## 第 2 步：文档分块 ✂️

### 为什么要分块？
- 大模型上下文窗口有限（例如 GPT-4o 是 128K）
- 太长的文档一次塞不下
- 检索太粗粒度，答案找不到

### 常见分块策略（Day06 详细讲）

| 策略 | 说明 | 优点 | 缺点 |
|------|------|------|------|
| **固定大小分块** | 每 512/1024 个 Token 一块 | 简单易实现 | 可能切断语义 |
| **语义分块** | 根据段落、标题分块 | 语义完整 | 实现复杂 |
| **父子分块** | 大父块做检索，小子块做回答 | 兼顾召回和精准 | 复杂度高 |

### 分块大小选择建议
| 场景 | 建议大小 |
|------|---------|
| **问答** | 256-512 tokens |
| **摘要** | 512-1024 tokens |
| **长文档** | 1024-2048 tokens |

## 第 3 步：文本向量化 🔢

### 什么是向量化？
把人类可读的文本 → 计算机可读的高维向量（一组数字）

```
"今天天气真好" 
    ↓ Embedding 模型
[0.123, -0.456, 0.789, ..., 0.321]  (1024 维向量)
```

### 向量的意义
- **语义相似** → 向量距离近（夹角小）
- **语义不同** → 向量距离远（夹角大）

```
"猫" ←(很近)→ "猫咪" ←(中等)→ "狗" ←(很远)→ "计算机"
```

### Embedding 模型（下一讲详细对比）
- OpenAI text-embedding-3-small/large
- BGE (BAAI General Embedding)
- M3E (M3E 多语言向量模型)
- GTE (General Text Embedding)

---

## 第 4 步：向量存储 💾

### 向量数据库 vs 传统数据库

| 对比项 | 传统数据库（MySQL/PostgreSQL） | 向量数据库（Chroma/Qdrant） |
|--------|------------------------------|---------------------------|
| **存储内容** | 结构化数据 | 高维向量 |
| **查询方式** | SQL 精确匹配 | 相似度检索 |
| **索引类型** | B-tree, Hash | HNSW, IVF |
| **适用场景** | 精确查询 | 语义搜索 |

### 常见向量数据库（下一节详细讲）
- **Chroma**：轻量级，本地运行，开发首选
- **Qdrant**：生产级，高性能，分布式
- **Pinecone**：云服务，完全托管，贵
- **Weaviate**：图 + 向量，混合检索
- **Milvus**：LF AI 基金会，功能全

## 第 5 步：查询向量化 + 相似度检索 🔍

### 完整检索流程
```
用户问题 → 向量化 → 向量数据库 → Top-K 最相似的块 → 返回
```

### 相似度算法（下一节详细讲）
| 算法 | 说明 | 适用场景 |
|------|------|---------|
| **余弦相似度** | 看向量夹角 | 通用，最常用 |
| **欧氏距离** | 看直线距离 | 低维空间 |
| **内积** | 向量点积 | 归一化后 = 余弦 |

### Top-K 选择建议
- K=3-5：通用场景
- K=5-10：需要更多上下文
- K=10+：可能引入噪声，需要重排序

---

## 第 6 步：重排序 Rerank 📋

### 为什么需要重排序？
向量检索是**粗筛**，很快但不够精准；重排序是**精排**，慢但更准确。

```
向量检索返回 Top 20 → Reranker 重排 → 返回 Top 3
     (快，召回高)         (准，精准高)
```

### 常见 Reranker
- **BGE-Reranker**：BGE 系列的重排序模型
- **ColBERT**：Contextualized Late Interaction
- **Cross-Encoder**：直接给 pair 打分

### 重排序效果
| 阶段 | 准确率 | 速度 |
|------|--------|------|
| 向量检索 | 70% | ⚡ 极快 |
| + 重排序 | 85% | 🐢 慢 5-10 倍 |

prompt = """

{query}

【回答要求】

### 生成过程
```

```

## RAG 完整代码示例（最简版）

```python
# -*- coding: utf-8 -*-
"""
最简 RAG 系统示例
"""

import numpy as np
from sentence_transformers import SentenceTransformer, util

# 1. 准备文档
documents = [
    "Python 是一种解释型、高级、通用的编程语言，由 Guido van Rossum 于 1991 年首次发布。",
    "Python 的设计哲学强调代码的可读性，使用显著的缩进。",
    "Python 支持多种编程范式，包括面向对象、命令式、函数式和过程式编程。",
    "Python 的标准库提供了丰富的功能，被称为'电池包括'哲学。",
    "Python 广泛应用于 Web 开发、数据分析、人工智能、科学计算等领域。"
]

# 2. 加载 Embedding 模型
model = SentenceTransformer('BAAI/bge-small-zh-v1.5')

# 3. 文档向量化
doc_embeddings = model.encode(documents, convert_to_tensor=True)

# 4. 用户查询
query = "Python 是谁创建的？"
query_embedding = model.encode(query, convert_to_tensor=True)

# 5. 相似度检索
hits = util.semantic_search(query_embedding, doc_embeddings, top_k=3)

# 6. 输出检索结果
print("=" * 50)
print(f"用户查询: {query}")
print("=" * 50)
print("\n最相关的文档:")
for i, hit in enumerate(hits[0], 1):
    print(f"\n--- Top {i} (相似度: {hit['score']:.3f}) ---")
    print(documents[hit['corpus_id']])

# 7. 构建增强提示词
context = "\n\n".join([
    f"[资料{i+1}] {documents[hit['corpus_id']]}"
    for i, hit in enumerate(hits[0])
])

prompt = f"""
你是一个问答助手，只基于下面提供的参考资料回答用户问题。
如果参考资料里没有答案，请说"抱歉，我在提供的资料里找不到答案"。

【参考资料】
{context}

【用户问题】
{query}

【回答要求】
1. 只使用参考资料里的信息
2. 回答要简洁准确
3. 请标注答案来源

【回答】
"""

print("\n" + "=" * 50)
print("生成的提示词:")
print("=" * 50)
print(prompt)
```

## RAG 常见问题 QA

### Q1: RAG 和 Fine-tuning 怎么选？

| 维度 | RAG | Fine-tuning |
|------|-----|-------------|
| **知识更新** | 快（加文档就行） | 慢（重新训） |
| **可追溯性** | ✅ 可以溯源 | ❌ 不能溯源 |
| **幻觉** | 较少 | 可能有 |
| **成本** | 低 | 高（数据+算力） |
| **适用场景** | 知识变化快、需要溯源 | 风格对齐、特定任务 |

### Q2: RAG 效果不好怎么办？
检查清单：
1. 文档分块合理吗？
2. Embedding 模型选对了吗？
3. Top-K 大小合适吗？
4. 有没有加 Rerank？
5. 提示词写得好吗？
