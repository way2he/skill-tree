# 文档分块策略详解 ✂️

&gt; 📖 **本讲学习目标**：掌握常见文档分块策略，理解各自优缺点  
&gt; ⏰ **预计学习时间**：1 小时

---

## 为什么要分块？

| 原因 | 说明 |
|------|------|
| **大模型上下文窗口有限** | GPT-4o 是 128K，太长的文档塞不下 |
| **检索精准度** | 太长的块可能同时包含多个话题，检索到的噪声多 |
| **平衡** | 块太小：信息不完整 / 块太大：不精准 |

---

## 常见分块策略

### 策略 1：固定大小分块（最简单）

#### 原理
按固定 Token/字符数切分，可选重叠（overlap）。

#### 例子
```
块大小 = 512 tokens
重叠 = 50 tokens

块 1: Token 0-511
块 2: Token 462-973 (重叠 50)
块 3: Token 924-1435
...
```

#### 优点
- ✅ 实现简单
- ✅ 计算快
- ✅ 可控性强

#### 缺点
- ❌ 可能切断语义（比如一句话切到两个块里）
- ❌ 不够灵活

#### 适用场景
- 快速原型
- 简单文档

#### 代码演示
```python
# 简单固定大小分块
def simple_chunking(text, chunk_size=512, overlap=50):
    chunks = []
    i = 0
    while i &lt; len(text):
        chunks.append(text[i:i+chunk_size])
        i += (chunk_size - overlap)
    return chunks
```

---

### 策略 2：语义分块（推荐）

#### 原理
按语义边界切分：段落、标题、句子等。

#### 常见切分边界
| 边界类型 | 说明 |
|---------|------|
| **段落** | `\n\n` 分割，保持段落完整 |
| **标题** | Markdown 标题 `#`、`##` |
| **句子** | 按句号、问号、感叹号切分 |
| **递归** | 先按大边界切，太大了再按小边界切 |

#### 优点
- ✅ 保持语义完整
- ✅ 检索结果更准确
- ✅ 减少切断语义的情况

#### 缺点
- ❌ 实现复杂
- ❌ 块大小不均匀

#### 适用场景
- 生产环境
- 对精准度要求高

---

### 策略 3：父子分块（Parent-Child）

#### 原理
- **父块**：大的块（如 2048 tokens），语义完整，用于检索
- **子块**：小的块（如 256 tokens），精准，用于生成答案

#### 流程
1. 用户查询 → 检索父块（召回）
2. 找到父块对应的子块 → 填到提示词里（精排/生成）

#### 优点
- ✅ 兼顾召回率（父块）和精准度（子块）
- ✅ 效果最好

#### 缺点
- ❌ 实现最复杂
- ❌ 存储开销大（存两套块）

#### 适用场景
- 高质量 RAG
- 对效果要求极高

---

## 三种策略对比

| 策略 | 实现难度 | 效果 | 速度 | 推荐场景 |
|------|---------|------|------|---------|
| **固定大小** | ⭐ 简单 | ⭐⭐ | ⭐⭐⭐⭐⭐ | 快速原型 |
| **语义分块** | ⭐⭐⭐ 中等 | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ | 生产环境推荐 |
| **父子分块** | ⭐⭐⭐⭐⭐ 复杂 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | 高质量 RAG |

---

## 分块大小选择建议

| 场景 | 推荐块大小 |
|------|-----------|
| **问答** | 256-512 tokens |
| **摘要** | 512-1024 tokens |
| **长文档** | 1024-2048 tokens |
| **父子分块** | 父: 1024-2048 / 子: 128-256 |

---

## 分块最佳实践

### 建议清单
1. ✅ **优先语义分块**，再考虑其他
2. ✅ **加重叠**，避免关键信息切断
3. ✅ **保留元数据**（标题、章节、作者等）
4. ✅ **测试评估**，看哪种策略在你的数据上效果最好

---

## 本讲小结 ✅

### 核心要点
1. **固定大小**：简单但可能切断语义
2. **语义分块**：按段落/标题切，推荐
3. **父子分块**：兼顾召回和精准，效果最好但复杂
