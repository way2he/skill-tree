## 🎯 一、核心概念总结

### 1. 协程 vs 线程 - 四个核心维度

| 维度　　　　 | 协程（Coroutine）　　　　　　　| 线程（Thread）　　　　　　 |
| --------------| --------------------------------| ----------------------------|
| **调度方式** | 用户态调度，由事件循环控制　　 | 内核态调度，由操作系统控制 |
| **切换开销** | 极小（函数调用级别，几十纳秒） | 大（上下文切换，几微秒）　 |
| **并发数量** | 轻松 10000+，受内存限制　　　　| 几百个就到瓶颈，受内核限制 |
| **数据安全** | 单线程内不需要锁　　　　　　　 | 需要锁机制保证线程安全　　 |

**💡 理解口诀**：协程是协作式，你让我我才让你；线程是抢占式，我想抢就抢。

---

### 2. `await` 的本质 - 让出-调度-唤醒三部曲

```
遇到 await
    ↓
当前协程暂停，交出控制权  →  事件循环接管
    ↓
事件循环调度其他可运行的协程  →  并发执行
    ↓
等待条件满足（如 sleep 结束、IO 返回）
    ↓
事件循环把协程标记为「可运行」
    ↓
下次调度到时，从暂停处继续执行
```

**关键理解**：`await` 不是「等」，而是「让」——让出 CPU 给别人用。

---

### 3. 并发 vs 并行 - 一字之差，天壤之别

| 概念 | 本质 | 硬件要求 | 适用场景 |
|------|------|----------|----------|
| **并发** Concurrency | 交替执行，看起来同时 | 单核也能并发 | IO 密集型（网络请求、文件读写） |
| **并行** Parallelism | 真的同时在跑 | 必须多核 CPU | CPU 密集型（科学计算、数据处理） |

**💡 生动类比**：
- 并发 = 你一个人边吃饭边看手机边回消息（快速切换）
- 并行 = 两个人同时吃饭，一个人吃饭，一个人看手机（真的同时）

## ⚠️ 二、高频易错点总结

### ❌ 易错点 1：每个请求新建 ClientSession

```python
# ❌ 错误写法 - 性能灾难！
async def fetch(url):
    async with aiohttp.ClientSession() as session:  # 每次新建
        async with session.get(url) as resp:
            return await resp.text()

# ✅ 正确写法 - 全局复用一个 Session
async def main():
    async with aiohttp.ClientSession() as session:  # 只建一次
        tasks = [fetch(session, url) for url in urls]
        await asyncio.gather(*tasks)
```

**为什么这么重要？**
- TCP 连接复用，减少三次握手开销
- HTTP keep-alive，性能提升 10 倍以上
- 避免耗尽客户端端口，避免触发服务端限流

---

### ❌ 易错点 2：for 循环 + await = 串行执行

```python
urls = ["url1", "url2", "url3"]

# ❌ 错误写法 - 串行执行，3个请求要 3 秒
for url in urls:
    await fetch(url)

# ✅ 正确写法 - 并发执行，3个请求只需要 1 秒
tasks = [fetch(url) for url in urls]
await asyncio.gather(*tasks)
```

**💡 面试高频**：一眼看出代码是串行还是并发——看到 `for + await` 就是串行！

---

### ❌ 易错点 3：重试不用指数退避

```python
# ❌ 错误写法 - 固定间隔 1 秒，容易造成重试风暴
for _ in range(3):
    try:
        return await request()
    except:
        await asyncio.sleep(1)  # 固定 1 秒

# ✅ 正确写法 - 指数退避，越错等越久
for i in range(3):
    try:
        return await request()
    except:
        await asyncio.sleep(2 ** i)  # 1s → 2s → 4s
```

**为什么指数退避？**
1. 保护服务器：越忙越不要打
2. 提高成功率：500 通常是临时的，等久一点恢复概率大
3. 节省客户端资源：避免无效重试

## 💻 三、代码模板库

### 模板 1：带信号量的并发请求（面试必背）

```python
import asyncio
import aiohttp
from aiohttp import ClientTimeout

async def fetch_with_semaphore(
    session: aiohttp.ClientSession,
    url: str,
    semaphore: asyncio.Semaphore
):
    """带并发控制的请求函数"""
    async with semaphore:  # 自动获取和释放信号量
        try:
            async with session.get(url) as response:
                return await response.text()
        except Exception as e:
            return f"Error: {str(e)}"

async def main(urls: list, max_concurrent: int = 3, timeout: int = 5):
    """
    并发请求主函数
    - max_concurrent: 最大并发数
    - timeout: 每个请求超时时间（秒）
    """
    semaphore = asyncio.Semaphore(max_concurrent)
    client_timeout = ClientTimeout(total=timeout)
    
    async with aiohttp.ClientSession(timeout=client_timeout) as session:
        tasks = [fetch_with_semaphore(session, url, semaphore) for url in urls]
        return await asyncio.gather(*tasks)

# 使用
if __name__ == "__main__":
    urls = ["url1", "url2", "url3", ...]
    results = asyncio.run(main(urls, max_concurrent=10))
```

**✨ 模板亮点**：
- ✅ 信号量控制并发数
- ✅ 请求超时控制
- ✅ 全局复用 ClientSession
- ✅ 异常捕获，不影响其他请求
- ✅ 类型注解，代码规范

### 模板 2：指数退避重试装饰器

```python
import asyncio
from functools import wraps

def retry(max_retries: int = 3, base_delay: float = 1.0):
    """指数退避重试装饰器"""
    def decorator(func):
        @wraps(func)
        async def wrapper(*args, **kwargs):
            for attempt in range(max_retries):
                try:
                    return await func(*args, **kwargs)
                except Exception as e:
                    if attempt == max_retries - 1:
                        raise  # 最后一次重试失败，抛出异常
                    delay = base_delay * (2 ** attempt)
                    print(f"第 {attempt + 1} 次失败，{delay} 秒后重试: {e}")
                    await asyncio.sleep(delay)
        return wrapper
    return decorator

# 使用
@retry(max_retries=3, base_delay=1.0)
async def fetch_data(url):
    async with session.get(url) as resp:
        return await resp.json()
```

## 🏗️ 四、高并发爬虫系统设计（场景题标准答案）

### 整体架构：生产者消费者模式

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│  URL 生产者 │ →   │  任务队列   │ →   │  消费者池  │
│ （产生任务）│     │（缓冲+削峰）│     │（并发爬取）│
└─────────────┘     └─────────────┘     └─────────────┘
                                                   ↓
                                            ┌─────────────┐
                                            │  结果持久化 │
                                            │  去重/存储  │
                                            └─────────────┘
```

### 1. 并发控制
- **信号量**：控制最大并发数（根据目标网站承受能力调整，通常 10-50）
- **令牌桶**：更精细的 QPS 控制
- **动态调整**：根据响应时间、错误率自动调整并发数

### 2. 失败重试策略
- **重试条件**：5xx、超时、连接错误才重试；4xx 不重试
- **重试次数**：3-5 次足够
- **退避策略**：指数退避 + 随机抖动（避免雪崩）
- **死信队列**：多次失败的任务进入死信队列，人工处理

### 3. 反封禁策略（5板斧）
| 策略 | 说明 |
|------|------|
| **User-Agent 轮换** | 伪装成不同浏览器，请求头随机化 |
| **IP 代理池** | 大量代理 IP，轮换使用 |
| **请求间隔随机** | 固定间隔容易被检测，随机 1-3 秒 |
| **Cookie 池** | 模拟真实用户登录态 |
| **请求头完整** | Referer、Accept-Language 等都带上 |

### 4. 架构设计要点
- **去重机制**：URL 去重（Bloom Filter、Redis Set）
- **断点续爬**：任务状态持久化，失败可以接着爬
- **监控告警**：成功率、响应时间、QPS、错误率实时监控
- **分布式扩展**：多机部署，任务队列共享（Redis/RabbitMQ）

## 🧠 五、面试答题技巧

### 概念题答题公式
**定义 + 对比 + 场景 + 举例** = 满分答案

**示例**：协程和线程的区别？
> 协程是用户态调度的轻量级执行单元，线程是内核态调度的执行单元。（定义）
> 协程切换开销极小，线程切换开销大；协程能开几万个，线程只能开几百个。（对比）
> 协程适合 IO 密集型场景，线程适合 CPU 密集型场景。（场景）
> 比如 Node.js 就是单线程 + 协程模型，高效处理大量并发请求。（举例）

---

### 代码题答题技巧
1. **先找问题，再改代码** - 不要上来就写，先把问题列出来
2. **注释说明** - 改了什么，为什么这么改
3. **边界情况** - 异常处理、超时、空值等
4. **代码规范** - 命名清晰，结构合理

---

### 设计题答题技巧
1. **先讲整体架构** - 让面试官知道你有全局视野
2. **再分模块细说** - 每个模块怎么实现，为什么这么设计
3. **讲权衡 Trade-off** - 没有完美的方案，讲清楚你做了什么取舍
4. **讲扩展性** - 未来怎么升级，怎么应对更大的规模
