## 1. 协程和线程的区别？分别适用什么场景？

**标准答案（直接背！）：

> **区别主要有三点：
> 
> 第一，**调度方式不同**：协程是用户态调度，由程序自己控制切换；线程是内核态调度，由操作系统控制切换。
> 
> 第二，**切换开销不同**：协程切换开销极小，相当于函数调用，只有几十纳秒；线程切换开销大，需要几微秒，差100倍以上。
> 
> 第三，**并发数量不同**：协程可以轻松开10万+个；线程一般开几百个就到瓶颈了，内存和调度都扛不住。
> 
> **适用场景**：
> - 协程适合**IO密集型**场景：比如调用大模型API、批量HTTP请求、数据库查询，这些场景大部分时间在等IO，用协程可以充分利用等待时间。
> - 线程适合**CPU密集型**场景：比如大量计算、模型推理，需要多核CPU并行。

## 2. async/await 到底在做什么？await 到底在等什么？

**标准答案（直接背！）：

> `async` 关键字用来定义协程函数，调用async函数不会立即执行，只会返回一个协程对象。
> 
> `await` 是**让出CPU的信号**：遇到await的时候，当前协程暂停执行，把CPU让给事件循环，事件循环去跑其他就绪的协程。等await等待的IO操作完成了，再切回来继续执行后面的代码。
> 
> await等待的是**可等待对象**，可以是协程、Task、Future，本质上是在等某个IO操作完成，比如等HTTP响应回来、等sleep时间到了。

## 3. 什么是事件循环？它的工作原理是什么？

**标准答案（直接背！）：

> 事件循环是异步编程的核心调度器，单线程运行，负责调度所有协程。
> 
> 工作原理很简单，就是一个无限循环：
> 1. **检查**：有哪些协程就绪了可以跑
> 2. **执行**：执行就绪的协程
> 3. **切换**：遇到await就暂停，切去跑下一个就绪的协程
> 4. **循环**：不断重复上面三步
> 
> 本质就是**用单线程模拟并发**，利用IO等待的时间去做别的事，避免线程切换的开销。在Agent开发中，事件循环让我们可以同时调用几十个甚至上百个大模型API，而不需要开几十个线程。

## 4. 为什么要用 aiohttp 不用 requests？aiohttp 快在哪里？

**标准答案（直接背！）：

> requests是同步的，发请求的时候整个线程卡住，什么也干不了，只能等响应回来才能发下一个；aiohttp是异步的，等响应的时候可以去发别的请求。
> 
> 快在哪里？核心就是**并发**：比如发100个请求，每个请求响应时间1秒：
> - 用requests同步发：要100秒，一个一个等
> - 用aiohttp异步发：只要1-2秒，同时发，同时等
> 
> 性能提升了50-100倍，这就是为什么Agent开发中一定要用异步HTTP客户端，批量调用大模型API的时候差距特别明显。

## 5. ClientSession 为什么要复用？每个请求新建一个会怎么样？

**标准答案（直接背！）：

> ClientSession复用本质是**TCP连接复用**，避免每次都三次握手。
> 
> TCP建立连接需要三次握手，大概需要10-30ms，如果每个请求都新建一个ClientSession，每次都要三次握手，100个请求就多了1-3秒的额外开销，而且会占用大量本地端口，甚至导致端口耗尽。
> 
> 复用同一个ClientSession，会自动维护一个连接池，同一个域名的请求复用同一个TCP连接，省去了三次握手的开销，性能提升2-3倍。

## 6. 信号量 Semaphore 的工作原理？为什么需要控制并发？

**标准答案（直接背！）：

> 信号量本质是一个**许可证计数器**，工作原理：
> 1. 初始化的时候设定有几个许可证，比如3个
> 2. 每个协程进来先拿许可证，拿到了才能执行
> 3. 执行完归还许可证，下一个等待的协程才能拿到
> 4. 许可证没了就排队等
> 
> 为什么需要控制并发？三个原因：
> 1. 服务器端有QPS限制，并发太高会被封IP或者返回429限流
> 2. 本地端口有限，TCP连接数有上限，并发太高会耗尽端口
> 3. 太多并发会导致每个请求都很慢，反而整体吞吐量下降，甚至大量超时

## 7. 指数退避重试为什么是 2 的幂次？固定间隔重试有什么问题？

**标准答案（直接背！）：

> 指数退避是1秒、2秒、4秒、8秒...间隔越来越长，核心目的是**给服务器恢复的时间，避免重试风暴**。
> 
> 为什么是2的幂次？因为2的指数增长速度适中：
> - 太慢：重试间隔太长，用户体验不好
> - 太快：相当于没退避，服务器压力还是大
> - 2的幂次是工业界标准做法，TCP重传也是用的指数退避
> 
> 固定间隔重试的问题：如果100个请求同时失败，1秒后100个同时重试，2秒后又100个同时重试，相当于给服务器发起DDoS攻击，导致雪崩，这就是**重试风暴**。指数退避让重试越来越稀疏，避免了这个问题。
> 
> 另外还要注意：不是所有错误都要重试，4xx是客户端错误，重试也没用，直接放弃；5xx是服务器错误，可以重试；网络超时可以重试。

## 8. 不加超时会怎么样？所有外部 IO 都要加超时吗？

**标准答案（直接背！）：

> 不加超时的话，网络是不可靠的，请求可能永远回不来，协程会永远挂住，整个系统卡住。
> 
> **是的！所有外部IO必须加超时！**
> 
> 包括：HTTP请求、数据库查询、Redis调用、大模型API调用，凡是需要跟外部系统交互的，都必须加超时。这是工程化的基本要求，90%的线上故障都是因为某个地方忘了加超时，导致整个系统挂住。
> 
> 在Agent开发中，大模型API调用尤其要加超时，OpenAI有时候会卡住几分钟不返回，不加超时整个Agent就死了。

## 9. asyncio.create_task() 和直接 await 的区别？

**标准答案（直接背！）：

> 直接await是**串行执行**：await一个协程的时候，会等它执行完了才会执行下一行，一个一个来。
> 
> create_task()是**并发执行**：把协程包装成Task，交给事件循环去调度，马上返回，不会等它执行完，后面再await结果。
> 
> 简单说：
> ```python
> await a()  # 等a做完了才做b，串行
> await b()
> 
> task1 = create_task(a())  # 不用等a做完，马上开始b
> task2 = create_task(b())  # a和b并发跑
> await task1
> await task2
> ```
> 
> Agent开发中批量调用10个大模型，肯定要用create_task并发跑，而不是一个一个await。

## 10. 手写：带并发控制、超时、指数退避重试的异步请求函数

**标准答案（直接背，10分钟内默写出！）：

```python
import asyncio
import aiohttp
from aiohttp import ClientTimeout
import logging

logging.basicConfig(level=logging.INFO)
logger = logging.getLogger(__name__)

MAX_CONCURRENT = 3
MAX_RETRIES = 3
INITIAL_DELAY = 1
TIMEOUT = 10

async def fetch_with_retry(session, url, max_retries=MAX_RETRIES):
    """带指数退避重试的异步请求"""
    for attempt in range(max_retries):
        try:
            async with session.get(url) as response:
                if response.status == 200:
                    logger.info(f"✅ 成功: {url}, 第 {attempt+1} 次尝试")
                    return await response.json()
                else:
                    logger.warning(f"⚠️ 状态码 {response.status}: {url}")
                    if 400 <= response.status < 500:
                        return None  # 4xx客户端错误，不重试
                    
        except Exception as e:
            logger.warning(f"❌ 失败: {url}, 第 {attempt+1} 次尝试, 错误: {str(e)}")
        
        # 指数退避等待
        if attempt < max_retries - 1:
            delay = INITIAL_DELAY * (2 ** attempt)  # 1s → 2s → 4s
            logger.info(f"⏳ {delay} 秒后重试...")
            await asyncio.sleep(delay)
    
    logger.error(f"❌ 最终失败: {url}, 重试了 {max_retries} 次")
    return None

async def main(urls):
    semaphore = asyncio.BoundedSemaphore(MAX_CONCURRENT)
    timeout = ClientTimeout(total=TIMEOUT)
    
    async def bounded_fetch(url):
        async with semaphore:  # 信号量控制并发
            return await fetch_with_retry(session, url)
    
    async with aiohttp.ClientSession(timeout=timeout) as session:
        tasks = [bounded_fetch(url) for url in urls]
        results = await asyncio.gather(*tasks)
        return results

if __name__ == "__main__":
    import sys
    if sys.platform.startswith('win'):
        asyncio.set_event_loop_policy(asyncio.WindowsSelectorEventLoopPolicy())
    
    urls = [
        "https://httpbin.org/delay/1",
        "https://httpbin.org/status/500",
        "https://httpbin.org/status/404",
    ]
    asyncio.run(main(urls))
```

**代码亮点（面试要说出来！）：
1. ✅ BoundedSemaphore 信号量控制并发，避免无限并发
2. ✅ ClientTimeout 全局超时，防止永远挂住
3. ✅ 指数退避重试，间隔 1→2→4 秒
4. ✅ 4xx 错误不重试，5xx 和网络错误重试
5. ✅ 完整的日志记录，方便排查问题
6. ✅ Windows 平台特殊处理，避免 Event Loop 报错

当面试官问你对异步编程的理解，直接用这个 3 分钟模板：

> "我对 Python 异步编程有比较深入的理解和实战经验。简单说，异步编程就是在单线程里利用 IO 等待的时间来同时处理多个任务，特别适合 IO 密集型场景，比如调用大模型 API、批量 HTTP 请求这种场景。
> 
> 核心概念有三个：协程、事件循环、可等待对象。协程可以暂停和恢复，开销很小；事件循环是调度器，单线程跑所有协程；遇到 await 的时候当前协程暂停，让事件循环去跑别的就绪的协程，等 IO 完成了再切回来。
> 
> 我实战中用 aiohttp 写过异步批量请求，做了三层优化：第一是 ClientSession 复用，避免每次新建 TCP 连接的三次握手开销；第二是用 BoundedSemaphore 信号量控制并发数，避免并发太高被服务器封或者本地端口耗尽；第三是加了超时控制和指数退避重试，1 秒、2 秒、4 秒间隔重试，4xx 客户端错误不重试，5xx 和网络错误才重试。
> 
> 最后效果是原来同步请求 100 次要 100 秒，用异步优化后只需要 5-10 秒，性能提升了 10-20 倍。"
