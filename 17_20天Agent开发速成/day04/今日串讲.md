## 🎯 09:00-10:00　Function Calling 基础原理

### ① 一句话本质
**FC = 模型决策 + 代码执行 + 结果回灌**，模型本身从不执行任何函数。

### ② 标准三步舞
1. **第一轮**：用户问题 + 工具清单 → 模型返回 `tool_calls`（函数名 + JSON 参数）
2. **本地执行**：你的代码解析 JSON，调用函数，拿结果
3. **第二轮**：把结果作为 `role=tool` 消息回传 → 模型生成自然语言回复

### ③ tools 三要素
- **name**：英文蛇形（`get_weather`）
- **description**：模型理解"该不该调"的唯一依据
- **parameters**：JSON Schema 描述参数结构

### ④ vs Prompt 让模型输 JSON
1. FC 是 SFT 训练过的能力，准确率 99%+
2. Prompt 输 JSON 准确率 70-90%，经常引号/逗号错
3. FC 协议层支持并行/流式/结果回灌

### 💡 一句话杀手锏
**「FC 和 Prompt 输 JSON 看着像，本质完全不同：FC 是模型在 SFT 阶段专门训练的能力，有特殊 token 控制，准确率 99%+；Prompt 是骗模型按格式输出，没强约束。」**

## 🎯 10:00-11:00　JSON Schema 设计

### ① 7 大字段速查
1. **type** — string/number/integer/boolean/array/object
2. **description** — 参数含义说明（最关键！）
3. **enum** — 枚举值约束
4. **default** — 默认值
5. **required** — 必填字段数组（properties 外层）
6. **items** — 数组元素 schema
7. **properties** — 对象嵌套字段

### ② 5 大原则
1. **描述精准胜过简短**：5 字 desc 不如 50 字
2. **能枚举不用 string**：消除"上海/上海市/SH"歧义
3. **数字加范围**：minimum/maximum 防离谱值
4. **能省的字段就别要**：必填 ≤3 个
5. **反例写进 desc**：「不要写成 2026/05/22」

### ③ desc 黄金法则
- 函数级：「做什么 + 不做什么（边界）」，30-80 字
- 参数级：「含义 + 格式 + 反例」，20-60 字

### 💡 一句话杀手锏
**「把 desc 从 5 字优化到 50 字，参数准确率从 60% 提升到 95%+ 是常态。某客服 Agent 订单号格式错误率从 40% 降到 3%，就是改了 desc。」**

## 🎯 11:00-12:00　错误处理与幂等设计

### ① 4 类工具错误
1. **参数错误** — 格式/类型/缺字段 → 校验回灌让模型自纠正
2. **业务错误** — 城市不存在/余额不足 → 业务报错原文回灌
3. **网络错误** — 超时/不通 → 指数退避重试
4. **模型幻觉** — 调不存在的工具 → 严格校验函数名，直接报错

### ② 幂等性 = 同参数 N 次 = 1 次
- GET 天然幂等
- POST/PUT/DELETE **必须显式设计**
- 标准实现：**request_id 去重**（业界标杆）

### ③ 死循环 3 道防线
1. **MAX_TOOL_ITERATIONS = 5** 硬上限
2. **(name, args_hash) 重复检测**
3. **降级回退**：超限时切到"直接回答用户"模式

### 💡 一句话杀手锏
**「MAX_TOOL_ITERATIONS=5 硬上限 + 重复参数去重，能解决 99% 死循环。AutoGPT 早期最大的坑就是没这个上限，烧光 API 额度的人比比皆是。」**

## 🎯 14:00-15:00　多工具智能路由

### ① 三难题
1. 工具多了模型选不准（>20 个准确率显著下降）
2. 相似工具容易混淆
3. 全量塞入 token 爆炸

### ② 4 大路由策略
1. **关键词路由** — "天气" → 只挂天气工具
2. **语义向量路由** — 工具 desc 向量化，余弦相似度 Top-K 召回
3. **分级 Agent** — 路由 Agent 决定子领域 → 子 Agent 拿专属工具集
4. **成本路由** — 简单决策用便宜模型，复杂决策升级旗舰

### ③ 工具命名分组规范
1. 前缀分组：`db_query`、`db_insert`、`api_get`
2. 同类合并：`crud(action: enum)` 比 4 个独立工具准
3. 冷门下沉到二级 Agent

### 💡 一句话杀手锏
**「某客服 Agent 从 200 工具全量塞入降为 Top-5 召回后，单次 token 从 12K 降到 1.5K，月成本下降 87%，准确率反而从 78% 提升到 92%。」**

## 🎯 15:00-16:00　OpenAI vs Anthropic 协议对比

### ① 协议层差异速查
| 维度 | OpenAI | Anthropic |
|------|--------|-----------|
| 工具定义 | 	ools=[{type:"function",function:{...}}] | 	ools=[{name,description,input_schema}] |
| 模型返回 | message.tool_calls[] | content[] 含 	ype=tool_use block |
| 参数字段 | rguments（JSON 字符串）需 parse | input（已是 dict） |
| 结果回灌 | ole=tool, tool_call_id, content | ole=user, content=[tool_result block] |
| 强制调用 | 	ool_choice={type:"function",function:{name}} | 	ool_choice={type:"tool",name} |

### ② 3 个核心坑
1. **arguments 类型不同**：OpenAI 字符串、Anthropic dict
2. **回灌角色不同**：tool vs user+tool_result
3. **流式协议不同**：增量拼接 vs content_block_delta

### ③ 国产模型生态
- DeepSeek V4 / Qwen3-Max / GLM-5.1 都兼容 OpenAI 协议
- Claude 是"少数派"但工具调用最稳，SWE-bench 第一

### 💡 一句话杀手锏
**「OpenAI 是 role 级别分类，Anthropic 是 content block 级别分类。国产模型都向 OpenAI 协议靠拢，Claude 是少数派但工具调用最稳。」**

## 🎯 16:00-17:00　梯度下降与优化器

### ① 梯度下降三变体
1. **BGD** — 全样本，慢，不适合大数据
2. **SGD** — 单样本，快但震荡大
3. **Mini-batch SGD** — batch=32/64/128，**工程事实标准**

### ② 5 大优化器演进
1. **SGD** — 朴素梯度下降
2. **Momentum** — 加动量项，像小球滚下山
3. **RMSProp** — 自适应学习率，频繁更新的参数 lr 小
4. **Adam = Momentum + RMSProp** — 最常用
5. **AdamW** — Adam + 解耦权重衰减，**大模型标配**

### ③ Adam vs AdamW 本质区别

#### Adam + L2 正则（错误做法）
1. 把 λ*θ 加到梯度里：g_t = g_t + λ*θ
2. 这个项被自适应学习率 √v_t 缩放
3. 频繁更新的参数衰减强度被稀释
4. 结果：不同参数衰减强度不一致，正则化失真

#### AdamW（正确做法）
1. 把 weight decay 从梯度计算中解耦
2. 直接乘到参数更新上：θ_t = θ_{t-1} - lr*(m_t/√v_t + λ*θ_{t-1})
3. 所有参数衰减强度一致
4. BERT/GPT/LLaMA 全系列都用 AdamW

### ④ 超参经验值
- **lr**：Adam 默认 1e-3，Transformer 微调 1e-5 ~ 5e-5
- **weight_decay**：AdamW 默认 0.01，大模型 0.1
- **warmup**：前 5-10% step 从 0 线性升到目标值

### 💡 一句话杀手锏
**「Adam 的 weight decay 是假的，AdamW 才是真的。所以 BERT 之后所有主流大模型都用 AdamW。」**
