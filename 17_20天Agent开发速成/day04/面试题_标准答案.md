## 第 1 题：Function Calling 的本质是什么？模型真的会执行函数吗？

### 一句话回答
**模型只输出"我想调用 xxx 函数、参数是 yyy"的结构化 JSON，真正执行函数的是你的代码，再把结果回灌让模型继续生成最终回复。**

### 详细回答（2 分钟）

Function Calling 的标准流程是"三步舞"：

1. **第一轮：模型决策**
   - 你把工具列表（tools）+ 用户问题一起发给模型
   - 模型分析用户意图，决定要不要调用工具
   - 如果要调用，返回 `tool_calls` 字段（含函数名 + 参数 JSON 字符串）

2. **本地执行**
   - 你的代码解析 JSON，找到对应函数，传参执行
   - 拿到真实执行结果（可能是 API 返回、数据库查询、计算结果）

3. **第二轮：结果回灌**
   - 把执行结果作为 `role=tool` 消息加进对话历史
   - 再次调用模型，模型基于结果生成自然语言回复

### 加分回答

**关键认知**：模型本身从不执行任何函数。这一点很多新人会搞错。模型的"工具使用能力"本质上是"生成符合 schema 的 JSON 的能力"。模型在 SFT 阶段专门训练过这个能力，所以稳定性比让模型自己写 JSON 强很多。

### 易错点

- ❌ 错答：「模型直接调用 API 拿到天气」
- ✅ 正解：「模型输出'我要调 get_weather('上海')'，你的代码去调 API，结果再回灌」

## 第 2 题：JSON Schema 中 description 字段的作用是什么？怎么写好？

### 一句话回答
**description 是模型理解"该不该调用此工具、参数怎么填"的唯一依据，写好它能把工具调用准确率从 60% 提升到 95%+。**

### 详细回答（2 分钟）

description 有两层作用：

1. **函数级 description**：决定模型在多个工具里"选哪个"
   - 写法：「做什么 + 不做什么（边界）」，约 30-80 字
   - 例：「预订国内航班机票。仅支持中国大陆机场，不支持港澳台和国际航班。」

2. **参数级 description**：决定模型"怎么填这个字段"
   - 写法：「含义 + 格式 + 反例」，约 20-60 字
   - 例：「出发日期，格式必须为 YYYY-MM-DD，例如 2026-05-22。不要使用 2026/05/22 或 5月22日。」

### 5 大原则

1. **描述精准胜过简短**：5 字的 desc 不如 50 字的 desc
2. **能枚举不用 string**：用 enum 避免歧义
3. **数字加范围约束**：minimum/maximum 避免离谱值
4. **能省的字段就别要**：参数越多模型越容易乱填
5. **反例写进 description**：「不要写成 xxx」效果立竿见影

### 真实案例
某客服 Agent 的 `query_order` 工具，原 desc 是 "query order"，订单号格式错误率 40%。改成「查询订单详情。订单号格式为 ORD- 开头 + 10 位数字，如 ORD-2026052201。不要使用 #、空格或省略前缀。」后错误率降到 3%。

## 第 3 题：工具的幂等性是什么？为什么重要？怎么实现？

### 一句话回答
**幂等 = 同样参数调 N 次和调 1 次效果完全一样，能避免超时重试导致的重复扣款、重复下单等灾难性后果。**

### 详细回答（3 分钟）

**定义**：
- GET 类操作天然幂等（查询不改变状态）
- POST/PUT/DELETE 默认不幂等，必须显式设计

**为什么 Agent 场景特别重要**：
1. 模型可能因为超时重试同一个工具调用
2. 用户可能误以为没成功，让 Agent 再试一次
3. 工具调用链路长，中间某一步失败时整个链路可能被重放

**3 种实现方式**：

1. **业务层 request_id 去重（推荐）**
   ```python
   def transfer(request_id, from_acc, to_acc, amount):
       if cache.exists(f"req:{request_id}"):
           return cache.get(f"req:{request_id}")  # 直接返回上次结果
       result = do_transfer(from_acc, to_acc, amount)
       cache.set(f"req:{request_id}", result, ttl=86400)
       return result
   ```

2. **数据库唯一索引**：在订单号、流水号上加 UNIQUE 约束，重复插入失败

3. **状态机校验**：「订单已支付」状态下再支付直接拒绝

### 加分回答
真正生产级的幂等还要考虑：
- request_id 的生成策略（UUID / snowflake）
- 缓存的过期时间（通常 24h ~ 7 天）
- 缓存击穿、雪崩的防护
- 跨服务调用的幂等传递（HTTP header 传 `X-Request-Id`）

## 第 4 题：模型反复调错工具陷入死循环怎么办？怎么防护？

### 一句话回答
**三道防线：MAX_TOOL_ITERATIONS 硬上限、(name, args_hash) 重复检测、降级到直接回答用户，能解决 99% 死循环。**

### 详细回答（3 分钟）

**典型死循环场景**：
1. 模型调 search_web 返回空 → 又调 search_web → 又空 → ...
2. 模型调 get_user(id=1) 报错 → 改成 get_user(id=2) → 报错 → 改成 id=3 → ...
3. 模型选错工具，连续 5 次都选同一个错的

**3 道防线**：

1. **最大迭代次数硬上限**
   ```python
   MAX_TOOL_ITERATIONS = 5  # 经验值
   for i in range(MAX_TOOL_ITERATIONS):
       resp = chat(...)
       if not resp.tool_calls:
           return resp.content
   raise ToolLoopError("达到最大调用次数")
   ```

2. **重复参数检测**
   ```python
   seen = set()
   for tc in tool_calls:
       key = (tc.function.name, tc.function.arguments)
       if key in seen:
           result = "ERROR: 你已经用相同参数调用过这个工具，请换个思路"
       else:
           seen.add(key); result = execute(tc)
   ```

3. **降级回退**：超过迭代上限时切换 prompt：「请基于已有信息直接回答用户，不要再调用工具」

### 加分回答
工业级方案还会做：
- **调用图谱可视化**：把 tool_calls 链路画出来便于 debug
- **熔断器**：单个工具失败率超过阈值时禁用
- **白名单回退**：死循环时自动切换到一个保守的"兜底回答模板"

LangChain、LangGraph 都内置了 `recursion_limit` 参数，AutoGPT 早期最大的坑就是没这个上限，烧光 API 额度的人比比皆是。

## 第 5 题：多工具场景（100+工具）下 token 成本爆炸，怎么解？

### 一句话回答
**四招组合拳：向量召回 Top-K、description 精简、分级缓存、小模型路由，能把 token 降 90%+。**

### 详细回答（3 分钟）

**问题来源**：
- 100 个工具全塞进 tools 参数 ≈ 10K token，每轮对话都重复消耗
- 模型在多工具中选错的概率也上升

**4 招优化**：

1. **向量召回 Top-K**（最有效）
   - 离线：把所有工具 description 向量化存 Milvus/Qdrant
   - 运行时：用户 query 向量化，余弦相似度召回 Top-5 工具
   - 效果：100 工具 → 5 工具，token 降 95%

2. **description 精简**
   - 把示例和反例放外部文档，工具内描述压到 50 字以内
   - 用 markdown 链接指向详细文档（虽然模型不会去访问，但 desc 简洁了）

3. **分级缓存**
   - 相同语义的 query 召回的工具集大概率一样
   - 用 query 的 embedding 做 KNN 匹配，命中则复用工具集

4. **小模型分流**
   - 用便宜模型（gpt-5.4-mini / DeepSeek V4 Flash）做路由决策
   - 旗舰模型只做最终生成
   - 单次成本可降 5-10 倍

### 加分回答
**真实案例**：某客服 Agent 从 200 工具全量塞入降为 Top-5 召回后：
- 单次对话 token：12K → 1.5K
- 单次成本：.12 → .015
- 工具选择准确率反而从 78% 提升到 92%（少了干扰项）
- 月成本下降 87%

**进阶方案**：分级 Agent 架构 —— 路由 Agent 决定子领域（订单/物流/售后）→ 子 Agent 拿专属工具集，每个子 Agent 只挂 5-10 个工具。

## 第 6 题：OpenAI 和 Anthropic 的工具调用协议有什么本质区别？

### 一句话回答
**OpenAI 是 role 级别分类（role=tool），Anthropic 是 content block 级别分类（type=tool_use/tool_result）。**

### 详细回答（3 分钟）

**3 大核心差异**：

| 维度 | OpenAI | Anthropic |
|------|--------|-----------|
| arguments 类型 | JSON 字符串需 json.loads | 已解析的 dict |
| 结果回灌角色 | role=tool, tool_call_id | role=user, content=tool_result block |
| 流式协议 | tool_calls 增量拼接 | content_block_delta 事件 |

**举例对比**：

OpenAI 风格：
`python
# 模型返回
msg.tool_calls[0].function.arguments  # "{\"city\":\"上海\"}" 字符串
# 结果回灌
{"role": "tool", "tool_call_id": "call_abc", "content": "22°C 多云"}
`

Anthropic 风格：
`python
# 模型返回
for block in response.content:
    if block.type == "tool_use":
        block.input  # {"city": "上海"} 直接是 dict
# 结果回灌
{"role": "user", "content": [{
    "type": "tool_result", "tool_use_id": "toolu_xxx", "content": "22°C 多云"
}]}
`

### 加分回答

**设计哲学差异**：
1. OpenAI 早期是 unction_call 单工具，后来才升级 	ool_calls 多工具，有历史包袱
2. Anthropic 从 Claude 2.1 起步就是 content block 数组设计，所有内容（文本/图片/工具）平等对待
3. **国产模型趋同 OpenAI**：DeepSeek V4、Qwen3-Max、GLM-5.1 都兼容 OpenAI 协议，Claude 是"少数派"

**工程上怎么解决差异？**
统一抽象适配器层（见 Day03 + Day04 的 ToolAdapter 模式），业务代码完全不感知厂商。

## 第 7 题：Adam 和 AdamW 的本质区别是什么？为什么大模型预训练都用 AdamW？

### 一句话回答
**区别在 weight decay 的实现方式：Adam 的 weight decay 是假的（被自适应学习率稀释的 L2 正则），AdamW 把它解耦出来做真正的权重衰减。**

### 详细回答（5 分钟）

**Adam 的优化更新公式**：
`
m_t = β1*m_{t-1} + (1-β1)*g_t        # 一阶动量
v_t = β2*v_{t-1} + (1-β2)*g_t²       # 二阶动量
θ_t = θ_{t-1} - lr * m_t / (√v_t + ε)
`

**Adam + L2 正则（错误做法）**：
- 把 λ*θ 加到梯度里：g_t = g_t + λ*θ
- 这个项也会被除以 √v_t 缩放
- **后果**：频繁更新的参数 v_t 大 → 衰减强度被稀释
- **再后果**：不同参数的衰减强度不一致，正则化效果失真

**AdamW 的修正**：
`
m_t, v_t 计算同 Adam
θ_t = θ_{t-1} - lr * (m_t / (√v_t + ε) + λ*θ_{t-1})
                                          ↑ 把 weight decay 从梯度里挪出来
`
- 衰减项不再被 √v_t 缩放
- **所有参数的衰减强度一致**，正则化才真的起作用

### 为什么大模型必须用 AdamW

1. **参数量巨大**：Transformer 动辄几亿到几千亿参数，weight decay 是关键正则手段
2. **泛化敏感**：Adam 的稀释问题在大模型上会显著影响验证集表现
3. **实证证据**：
   - 论文 *Decoupled Weight Decay Regularization* (Loshchilov & Hutter, 2019) 系统性证明 AdamW 更优
   - BERT、GPT-2/3/4、LLaMA、PaLM 全系列都是 AdamW
   - Vision Transformer、Swin、ConvNeXt 也是 AdamW

### 加分回答

**典型超参（大模型场景）**：
- lr：1e-4（预训练）、1e-5 ~ 5e-5（微调）
- weight_decay：0.1（预训练）、0.01（微调）
- β1, β2, ε：0.9, 0.95（大模型常用，比默认 0.999 略低）, 1e-8
- warmup：前 1000-2000 step 从 0 线性升到目标 lr

**面试杀手锏一句话**：「Adam 的 weight decay 是假的，AdamW 才是真的。所以 BERT 之后所有主流大模型都用 AdamW。」
