## ✅ 核心学习方法

### ❌ 错误的重学方法（之前可能这么学的）
> 花 1 个月把 py-ml 全部学完，再花 1 个月学 py-dl，再花 1 个月学 py-nlp，最后学 Agent...
> **结果：学了就忘，全是纸上谈兵，不知道用在哪**

### ✅ 正确的重学方法（现在这么学）
> **每天 30 分钟学基础，当天就用在 Agent 上！**
>
> 例如：
> - 早上学「词向量和相似度计算」
> - 下午写 RAG 代码时，就用这个知识来调试检索结果的好坏
> - 晚上复盘：为什么有的文档检索准，有的不准？和余弦相似度有什么关系？

> **结果：学的东西马上用，印象深刻，20 天后既学会了 ML/DL/NLP，又写出了完整的 Agent！** 🚀

### 📌 第一类：基础概念题（1-5题）

#### 1. AI Agent 和传统 Chatbot 的核心区别？
**参考答案：**
- 传统 Chatbot：仅对话，无自主执行
- AI Agent：自主规划 + 长短期记忆 + 工具调用 + 目标导向执行

#### 2. Agent 和 RAG 的区别？
**参考答案：**
- RAG = "给 AI 一本书让它查"
- Agent = "给 AI 一个任务让它自己规划、调用工具、执行、反馈"
- Agent = LLM + 规划 + 工具调用 + 记忆

#### 3. Agent 的"自主性"体现在哪五个方面？
**参考答案：**
1. 无需逐步指导，独立运作
2. 感知环境变化并调整策略
3. 目标导向主动执行
4. 自然交互适配人类
5. 从历史任务中学习优化

#### 4. Agent 的核心组件有哪些？
**参考答案：**
- **Planning（规划）**：任务拆解、反思优化
- **Memory（记忆）**：短期对话记忆、长期知识库记忆
- **Tools（工具）**：检索、代码执行、API调用等
- **Action（行动）**：执行并获取反馈

#### 5. ReAct 模式的核心思想是什么？
**参考答案：**
ReAct = Reasoning（推理）+ Acting（行动）
让大模型在思考和行动之间循环交替：思考下一步做什么 → 调用工具执行 → 观察结果 → 继续思考，直到完成任务

---

### 📌 第二类：Planning 模块深度题（6-10题）

#### 6. Agent Planning 的主流方法有哪些？
**参考答案：**
- CoT（链式推理）：线性一步步想
- ToT（树状思维）：多路径探索，择优
- Reflexion：自我反思 + 迭代改进
- Plan-and-Solve：先做整体规划，再分步执行

#### 7. 任务执行失败时 Planning 如何处理？
**参考答案：**
- 回退到上一步，更换分解策略
- 反思失败原因，调整 Prompt
- 降低任务复杂度，拆成更小子任务
- 向用户求助（Human-in-the-loop）

#### 8. 什么是 Reflexion？它和普通 CoT 有什么区别？
**参考答案：**
- CoT：单次推理链，完成就结束
- Reflexion：在 CoT 基础上增加了「反思阶段」，产出结果后会自我评估，如果不满意就修正重来
- 核心价值：能从失败中学习，显著提升复杂任务的成功率

#### 9. 长程任务中，如何避免计划漂移？
**参考答案：**
- 明确的任务目标和验收标准
- 定期检查点（Checkpoint）回查目标对齐度
- 短期子任务里程碑设置
- 任务完成度量化评估

#### 10. 如何让 Agent 的规划更符合人类习惯？
**参考答案：**
- Few-shot 示例：给几个人类规划的样本
- 角色设定：明确 Agent 是「有经验的项目经理」等
- 人类反馈数据微调：用真实人类规划数据微调
- 约束输出格式：要求按人类习惯的步骤输出

### 📌 第三类：RAG 系统设计题（大厂必问，11-20题）

#### 11. 如何设计一个企业级 RAG 系统？
**参考答案：**
```
输入层 → 预处理（清洗/去重） → 文档切分（优化粒度） → 向量化（Embedding）
                                                            ↓
用户查询 → 查询改写 → 向量检索 + 关键词检索（混合） → 精排（Rerank）
                                                            ↓
                                                    上下文拼接 → Prompt 约束
                                                            ↓
                                                    LLM 生成 → 事实校验 → 输出
```

#### 12. 文档切分粒度如何选择？太长/太短有什么副作用？
**参考答案：**
- 太短：丢失上下文语义，检索准确率下降
- 太长：噪声多，Token 浪费，关键信息被稀释
- 最佳实践：按语义切分（LangChain SemanticChunker），512-2048 Token，带 10-20% 重叠

#### 13. 如何系统性评估 RAG 系统？
**参考答案：**
- **检索指标**：Context Relevance（上下文相关性）、Precision/Recall
- **生成指标**：Answer Faithfulness（忠实度，无幻觉）、Answer Relevance
- **工具**：RAGAS、TruLens、LLM-as-Judge（GPT-4 自动打分）
- **在线评估**：用户点赞/点踩、采纳率、停留时间

#### 14. Self-RAG 和传统 RAG 的本质区别？
**参考答案：**
- 传统 RAG：单向流水线（检索 → 生成），一锤子买卖
- Self-RAG：闭环反馈，可"反思检索结果不够好 → 重写查询 → 再检索"
- LangGraph 的条件边 + 状态共享是实现 Self-RAG 的最佳方式

#### 15. 五层幻觉防御体系怎么设计？
**参考答案：**
1. **第1层：检索层** - Query Rewriting、HyDE、多路召回提升信噪比
2. **第2层：Prompt层** - 强制"只根据 Reference 回答"，要求 Citation 溯源
3. **第3层：生成层** - 结构化输出约束，Temperature 调低（<0.3）
4. **第4层：后处理层** - NLI（自然语言推断）校验答案与引用的一致性
5. **第5层：反馈层** - 用户负面反馈自动回流，优化检索和 Prompt

#### 16. RAG 中检索到的文档太多怎么办？
**参考答案：**
- 精排（Rerank）：用 CrossEncoder 给检索结果排序，只取 Top 3-5
- 文档摘要：先让 LLM 给每个文档生成 1 句话摘要，再拼接
- 相关性过滤：设置相似度阈值，低于阈值的直接丢弃
- Map-Reduce 策略：先让 LLM 分别处理每个文档，再汇总结果

#### 17. 如何提升 RAG 的召回率？
**参考答案：**
- 多路召回：向量 + 关键词 + 同义词
- 查询改写：让大模型把用户查询改写成 3 个不同版本一起检索
- 混合检索：BM25 + 向量检索加权融合
- HyDE：先用大模型生成一个假设的理想答案，再用这个答案去检索

#### 18. RAG 和 Fine-tuning 分别适用什么场景？
**参考答案：**
| 场景 | RAG | Fine-tuning |
|------|-----|-------------|
| **知识更新频率** | 高（每周/每天更新） | 低（季度/年度） |
| **知识来源** | 外部文档、数据库 | 特定风格、任务格式 |
| **可解释性** | 高（可溯源引用来源） | 低（黑盒） |
| **实现成本** | 低 | 高（数据、算力） |
| **幻觉控制** | 好 | 差 |

#### 19. 什么是 RAG 的「丢失在中间」问题？如何解决？
**参考答案：**
- 现象：大模型倾向于关注上下文开头和结尾的信息，中间的容易忽略
- 解决方案：
  1. Lost-in-the-middle 重排序：把最相关的文档放在开头和结尾
  2. 减少上下文长度，只保留最相关的 3-5 个文档
  3. Map-Reduce 策略，分段处理再汇总

#### 20. 如何优化 RAG 的延迟？
**参考答案：**
- 缓存：相同 Query 直接返回缓存结果
- 异步检索：多个检索源并行执行
- 向量数据库优化：索引优化、分片、预热
- 流式生成：边检索边生成，不用等全部检索完
- Embedding 模型量化：用 4-bit 量化加速向量化

### 📌 第四类：LangGraph 深度题（21-28题）

#### 21. 为什么选 LangGraph 而不是 LangChain SequentialChain/RouterChain？
**参考答案：**
- SequentialChain：严格线性，无法重试/回退
- RouterChain：能路由但无法处理状态依赖的循环
- LangGraph：状态机模型 + 显式条件跳转，完美支持非线性流程，状态在节点间自由传递

#### 22. GraphState 设计的核心考量？
**参考答案：**
- 哪些字段需要在节点间传递？（messages、relevance_scores、query_history）
- 哪些字段需要持久化？（用户会话、执行日志）
- 如何避免状态膨胀？（只保留关键信息，大结果存外部存储）

#### 23. LangGraph 中的 Human-in-the-loop 怎么实现？
**参考答案：**
- 用 `interrupt_before` 在某个节点执行前暂停
- 等待人类输入或确认后再继续
- 典型场景：高风险操作前人工确认、工具调用失败人工介入、需要人类提供额外信息

#### 24. 如何用 LangGraph 实现 Self-RAG？
**参考答案：**
1. Retrieve 节点：检索文档
2. Relevance Check 节点：判断文档是否相关
3. 条件边：不相关 → Rewrite Query 节点 → 回到 Retrieve；相关 → Generate 节点
4. Generate 节点：生成答案
5. Fact Check 节点：校验答案是否有事实依据
6. 条件边：不通过 → 修正；通过 → 结束

#### 25. LangGraph 的 checkpoint 机制有什么用？
**参考答案：**
- 断点续跑：任务中断后可以从断点继续，不用重来
- 时间旅行：可以回到过去任意节点，查看当时的状态
- 调试方便：可以重放任意步骤，排查问题
- 用户体验：用户可以中途修改输入或撤销操作

#### 26. 什么是 Agent 的「状态膨胀」问题？如何解决？
**参考答案：**
- 现象：随着对话进行，messages 越来越长，Token 消耗爆炸，速度变慢
- 解决方案：
  1. 滑动窗口：只保留最近 N 轮对话
  2. 对话摘要：定期让 LLM 把历史对话压缩成摘要
  3. 重要性评分：只保留高分重要记忆
  4. 长期记忆外置：不重要的记忆存向量数据库，需要时再检索

#### 27. LangGraph 中如何处理并行节点？
**参考答案：**
- 使用 `Send` API 向多个节点并行发送消息
- 等待所有并行节点完成后再进入下一节点
- 场景：多个工具并行调用、多个专家 Agent 并行分析

#### 28. LangGraph 和 AutoGen/CrewAI 的区别？
**参考答案：**
| 维度 | LangGraph | AutoGen/CrewAI |
|------|-----------|----------------|
| **定位** | 底层编排框架 | 高层多 Agent 框架 |
| **灵活度** | 极高，完全可控 | 开箱即用，但定制化难 |
| **状态管理** | 一等公民，强大 | 内置，较难扩展 |
| **适用场景** | 复杂生产系统 | 快速原型、演示 |
| **学习曲线** | 较陡 | 平缓 |

#### 29. 如何设计一个多 Agent 客服系统？

用户 → 路由 Agent → 判断意图

#### 30. 多 Agent 协作的常见模式有哪些？
**参考答案：**
- 层级式：Manager + Worker（最常用）
- 并行式：多个专家同时干活，结果汇总

#### 31. 多 Agent 系统中如何避免死循环？
**参考答案：**
- 最大步数限制：超过 N 步强制结束
- 重复检测：检测是否在说同样的话，循环了
- 超时机制：单次任务超时终止
- 仲裁者 Agent：专门角色负责检测和打破死循环

#### 32. Agent 之间的消息协议怎么设计？

#### 33. 多 Agent 和单 Agent 的适用场景对比？

|------|---------|---------|

#### 35. 什么是「代理模式」和「中介模式」？各有什么优缺点？

  - 优点：解耦，扩展性好
  - 缺点：复杂度高，调试困难

### 📌 第六类：工具调用（Tool Calling）（36-42题）

#### 36. 什么是工具调用幻觉？如何防范？
**参考答案：**
- 幻觉表现：调用不存在的工具、传错误参数、无视工具返回结果
- 防范手段：
  1. 严格 Schema 校验（JSON Schema）
  2. 工具名白名单机制
  3. 参数强校验（类型、范围、枚举值）
  4. 调用频率和并发限制
  5. 执行结果二次确认（高风险操作）

#### 37. Tool Calling 和 RAG 的关系？
**参考答案：**
- RAG 是特殊的 Tool Calling——"检索知识"就是一个专属工具
- Agent 通过工具调用触发 RAG 检索，获取精准知识再生成答案
- 区别：RAG 的"工具"是只读的知识库，普通 Tool 可以有副作用（发邮件、删数据）

#### 38. 如何让大模型学会调用工具的底层原理是什么？
**参考答案：**
- 微调阶段：用大量 Tool Calling 样本让模型学会输出特定格式
- 推理阶段：服务端检测到工具调用格式，拦截执行，再把结果拼回去继续生成
- 本质：模型学会了识别"要调用工具时，输出特定格式的 JSON

#### 39. 工具调用失败了怎么办？
**参考答案：**
1. 自动重试：指数退避重试 2-3 次
2. 参数修正：让大模型分析错误，自动修正参数再调用
3. 工具降级：这个工具用不了，换一个类似工具
4. 告知用户：实在不行，告诉用户当前功能暂时不可用

#### 40. 如何设计一个可扩展的工具注册中心？
**参考答案：**
```python
class ToolRegistry:
    def register(self, name, func, schema, description)
    def list_tools(self) -> list[Tool]
    def get_tool(self, name) -> Optional[Tool]
    def execute(self, name, params) -> Result
```
核心要点：统一注册、统一发现、统一执行、统一错误处理

#### 41. 什么是 Function Calling 和 Structured Outputs 的区别？
**参考答案：**
- Function Calling：模型输出工具调用格式，服务端拦截执行
- Structured Outputs：模型直接输出指定格式的 JSON，不执行
- 关系：Structured Outputs 是 Function Calling 的基础

#### 42. 安全敏感操作（如删数据、发邮件）的工具调用需要什么防护机制？
**参考答案：**
1. 白名单：只有授权用户能调用
2. 二次确认：执行前弹出确认框
3. 操作审计：所有操作记录日志
4. 权限最小化：每个工具只给必要权限
5. 沙箱隔离：代码执行必须在安全沙箱中

### 📌 第七类：工程化与性能优化（43-50题）

#### 43. 如何设计 Agent 系统的监控指标？
**参考答案：**
- **调用指标**：工具调用成功率、平均调用耗时、重试次数
- **效果指标**：答案准确率、幻觉率、用户满意度
- **成本指标**：单次对话 Token 消耗、API 成本
- **性能指标**：并发支持、响应延迟（P50/P95/P99）

#### 44. 如何降低大模型推理成本和延迟？
**参考答案：**
- 模型分层：简单问题用小模型，复杂问题用大模型
- 缓存机制：相同/相似 Query 直接返回缓存结果
- 异步处理：非实时操作后台异步执行
- 模型量化：4-bit/8-bit 量化，推理速度提升 2-4 倍
- vLLM/Text Generation Inference：PagedAttention 显存优化

#### 45. Agent 系统的高可用架构怎么设计？
**参考答案：**
```
用户 → 负载均衡 → Agent 服务集群
                    ↓
                模型网关（降级/重试/熔断）
                    ↓
        多个模型提供商（OpenAI + 开源模型）
```
核心：多模型提供商冗余，避免单点故障

#### 46. 如何做 Token 成本控制？
**参考答案：**
- 对话历史压缩：定期摘要历史对话
- 模型分层：简单问题用便宜的小模型
- 最大步数限制：防止死循环烧 Token
- 工具输出截断：工具返回结果太长时截断
- 缓存：相同问题直接返回缓存答案

#### 47. 如何处理大模型的速率限制（Rate Limit）？
**参考答案：**
1. 客户端限流：用令牌桶算法控制请求速率
2. 指数退避重试：遇到 429 错误时自动重试
3. 队列机制：超过限额的请求排队等待
4. 多 Key 轮询：多个 API Key 轮询使用
5. 降级策略：主模型超限，自动切备用模型

#### 48. 如何让 Agent 支持并发处理多个用户请求？
**参考答案：**
- 每个用户会话独立：State 按用户隔离
- 异步请求处理：FastAPI + Uvicorn 异步框架
- 连接池管理：数据库、Redis 连接池
- 资源隔离：不同用户的 Agent 执行环境隔离

#### 49. 什么是「冷启动」问题？如何优化？
**参考答案：**
- 冷启动：新用户第一次使用时，没有历史上下文，表现不好
- 优化：
  1. 欢迎语引导用户提供更多信息
  2. 预设通用角色和能力
  3. 从用户画像预加载相关知识
  4. 前几次交互多问问题，快速建立画像

#### 50. 如何做 A/B 测试来优化 Agent？
**参考答案：**
1. 定义清晰的评估指标（准确率、完成率、用户满意度）
2. 流量灰度：小流量切新版本
3. 对照组实验：A/B 两组同时运行
4. 统计显著性：结果是否有显著差异
5. 全量发布：效果好就全量上线
