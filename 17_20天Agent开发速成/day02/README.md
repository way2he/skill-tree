## 🎯 09:00-10:00：Prompt 工程核心：角色设定方法论

### 📝 核心知识点 + 对应面试考点

| 知识点 | 面试考点 | 回答要点 |
|--------|---------|---------|
| **黄金公式** | 一个好的系统提示词应该包含哪些部分？ | 1. 清晰的角色定位<br>2. 具体的背景描述<br>3. 明确的任务要求<br>4. 严格的输出格式<br>5. 边界条件与约束 |
| **角色人设** | 为什么要给大模型设定角色？直接说任务不行吗？ | 1. 激活大模型对应领域的知识参数<br>2. 设定专业语境，减少幻觉<br>3. 让输出风格更符合预期<br>4. 比如设定成"高级架构师"输出就比"普通程序员"质量高 |
| **约束条件** | 约束条件写得越多越好吗？ | 1. 不是！约束太多 = 过拟合，稍微变一点就不行<br>2. 约束太少 = 欠拟合，输出太发散<br>3. 3-5 条核心约束最佳 |

## 🎯 10:00-11:00：思维链 CoT 进阶 + Few-shot 最佳实践

### 📝 核心知识点 + 对应面试考点

| 知识点 | 面试考点 | 回答要点 |
|--------|---------|---------|
| **CoT 原理** | 思维链 CoT 为什么有效？ | 1. 大模型是按 token 逐步推理的<br>2. 让它把思考过程说出来 = 强迫它一步步来<br>3. 减少跳步导致的逻辑错误<br>4. 数学题、逻辑题效果尤其明显 |
| **零样本 CoT** | 什么是零样本 CoT？怎么用？ | 1. 不需要给示例，只加一句话<br>2. "请一步步思考，先分析问题，再给出答案"<br>3. "Let's think step by step" 就行 |
| **少样本 CoT** | Few-shot 示例怎么设计？多少个合适？ | 1. 3-5 个示例最佳，太多反而过拟合<br>2. 示例要多样化，覆盖不同边界情况<br>3. 每个示例都要包含完整的思考过程 |
| **自一致性** | 什么是 Self-Consistency？ | 1. 生成多个答案，投票选最优<br>2. 类似集成学习，减少单次输出的随机性<br>3. 代价是 Token 消耗增加 N 倍 |

## 🎯 11:00-12:00：JSON 结构化输出 100% 稳定方案

### 📝 核心知识点 + 对应面试考点

| 知识点 | 面试考点 | 回答要点 |
|--------|---------|---------|
| **为什么要结构化输出？** | 为什么要让大模型输出 JSON？直接输出不行吗？ | 1. 方便后续程序处理，不需要解析自然语言<br>2. 输出有固定结构，减少不确定性<br>3. 可以做参数校验，不符合格式自动重试 |
| **Pydantic 结构化输出** | 怎么保证大模型 100% 输出正确格式的 JSON？ | 1. 用 Pydantic 定义 Schema<br>2. OpenAI 原生支持 `response_format` 参数<br>3. 校验失败自动重试 2-3 次<br>4. 重试时把错误信息给大模型，让它修正 |
| **常见格式问题** | JSON 输出常见的坑有哪些？ | 1. Markdown 代码块包裹（```json）<br>2. 末尾多了逗号（JSON 不支持尾逗号）<br>3. 字符串里有未转义的引号<br>4. 输出了 JSON 以外的解释性文字 |

### 💻 必写代码 4：Pydantic + 自动重试完整实现

```python
# -*- coding: utf-8 -*-
from pydantic import BaseModel, Field
from typing import List
import openai
import json

class BugItem(BaseModel):
    line: int = Field(description="第几行发现的bug")
    problem: str = Field(description="问题描述")
    severity: str = Field(description="严重程度：高/中/低")
    suggestion: str = Field(description="具体改进建议")

class CodeReviewResult(BaseModel):
    bug_count: int = Field(description="发现的bug总数")
    bugs: List[BugItem] = Field(description="所有bug的列表")
    overall_score: int = Field(description="0-100分的整体质量评分")
    summary: str = Field(description="一句话总结")

def code_review(code: str, max_retries=3) -> CodeReviewResult:
    """带自动重试的代码审查，保证100%返回正确格式"""
    system_prompt = """你是高级Python代码审查专家，请严格按照JSON格式输出审查结果。
    不要输出任何JSON以外的内容，不要加```json标记，不要加解释。"""
    
    for attempt in range(max_retries):
        try:
            response = openai.beta.chat.completions.parse(
                model="gpt-4o-mini",
                messages=[
                    {"role": "system", "content": system_prompt},
                    {"role": "user", "content": code}
                ],
                response_format=CodeReviewResult,
                temperature=0
            )
            return response.choices[0].message.parsed
        except Exception as e:
            print(f"第 {attempt+1} 次尝试失败: {e}")
            if attempt == max_retries - 1:
                raise
    
    raise Exception(f"重试 {max_retries} 次全部失败")

# 使用
result = code_review("""
def add(a, b):
    return a + b
""")
print(result.json(indent=2))
```

## 🎯 15:00-16:00：ML基础：过拟合与欠拟合、偏差vs方差

### 📝 核心知识点 + 对应面试考点

| 知识点 | 面试考点 | 回答要点 |
|--------|---------|---------|
| **欠拟合 Underfitting** | 什么是欠拟合？有什么表现？ | 1. 模型太简单，连训练数据都学不好<br>2. 表现：训练集差，测试集也差<br>3. 类比 Prompt：写得太简单，大模型不知道你要什么 |
| **过拟合 Overfitting** | 什么是过拟合？有什么表现？ | 1. 模型太复杂，把训练数据的噪音也学进去了<br>2. 表现：训练集特别好，测试集特别差<br>3. **类比 Prompt：写得太死，大模型只会死记硬背你的示例，稍微变一点就不行** |
| **拟合刚好** | 什么是理想的拟合状态？ | 1. 训练集和测试集表现都好<br>2. 泛化能力强，没见过的数据也能处理<br>3. **类比 Prompt：恰到好处，既能理解意图，又能灵活应对** |
| **偏差 Bias** | 什么是偏差？偏差高说明什么？ | 1. 模型预测值与真实值的差距<br>2. 偏差高 = 模型太简单 = 欠拟合<br>3. 类比：Prompt 太模糊，大模型理解错了你的意图 |
| **方差 Variance** | 什么是方差？方差高说明什么？ | 1. 模型在不同训练集上的预测差异<br>2. 方差高 = 模型太复杂 = 过拟合<br>3. **类比：Prompt 太具体，输入稍微变一点输出就完全不一样** |
| **偏差-方差权衡** | 偏差和方差的关系？ | 1. 此消彼长，无法同时最小化<br>2. 太简单 = 偏差高，方差低<br>3. 太复杂 = 偏差低，方差高<br>4. **好的模型/Prompt 要找到最佳平衡点** |

### 📊 直观图解理解

```
训练误差
  ^
  │
  │ ●  过拟合区域（方差太高）
  │  ●
  │    ●
  │      ●
  │        ● ✅ 最佳平衡点
  │           ●
  │             ●
  │                ●  欠拟合区域（偏差太高）
  └───────────────────────────────> 模型复杂度 / Prompt约束程度
```

## 🎯 16:00-17:00：正则化 5 种方法在 Prompt 工程中的应用

### 📝 核心知识点 + 对应面试考点

| 正则化方法 | ML 中的作用 | Prompt 工程中的应用 | 类比解释 |
|----------|-----------|-------------------|---------|
| **L1 正则化** | 让不重要的权重变成0，特征选择 | 删掉 Prompt 里没用的废话，只留核心要求 | "你是一个很聪明的 AI" → 删掉 → 只留"你是高级Python代码审查专家" |
| **L2 正则化** | 惩罚大的权重，让所有权重变小 | 不要让某一个示例/约束太极端 | "必须严格按照格式，一个字都不能改" → 改成 → "尽量参考以下格式" |
| **Dropout** | 训练时随机让一部分神经元不工作 | 示例多样化，不要过度依赖某一个示例 | 3-5个不同情况的示例，而不是1个极端示例 |
| **早停 Early Stopping** | 验证集误差不再下降就停止训练 | Prompt约束够了就停，不要无限叠加约束 | 不要写了5条约束还不够，还要加第6、7、8条 |
| **数据增强** | 对训练数据做各种变换，增加多样性 | 示例覆盖各种边界情况，不要都是简单情况 | 代码审查示例要包含bug、性能问题、风格问题 |

### 💻 实战：用正则化思想优化 Prompt

```

#### ✅ 优化后（正则化恰到好处）
```markdown
你是一个拥有10年经验的高级Python代码审查专家。

你的任务是审查用户提供的Python代码，并给出具体的改进建议。

输出格式参考：
{
  "bug_count": 发现的bug数量,
  "bugs": [...],
  "overall_score": 0-100分,
  "summary": 一句话总结
}

约束：
1. 严重bug优先级最高
2. 建议要具体可执行
3. 只输出JSON
```

| 维度 | 优化前（过拟合） | 优化后（刚刚好） |
|------|----------------|----------------|
| **Prompt长度** | 300+ 字 | 150 字 |
| **约束数量** | 10+ 条 | 3 条核心 |
| **示例数量** | 1 个极端示例 | 0-3 个多样示例 |
| **泛化能力** | ❌ 输入稍微变一点就不行 | ✅ 各种代码都能处理 |
| **输出质量** | ❌ 容易死板照搬示例 | ✅ 灵活且高质量 |

> "我对 Prompt 工程有比较深入的理解和实战经验。我认为好的 Prompt 核心是找到偏差和方差的最佳平衡点——既要足够清晰，让大模型准确理解你的意图（降低偏差），又不能写得太死，让它只会死记硬背示例（降低方差）。
> 
> 我总结出了一套系统提示词的黄金公式：角色 + 背景 + 任务 + 格式 + 约束，五个部分。我会给大模型设定清晰的专业角色，激活对应领域的知识参数，然后明确任务要求和输出格式，最后加3-5条核心约束，太多约束会导致过拟合，太少会欠拟合。
> 
> 对于复杂的逻辑问题我会用思维链CoT，让大模型把思考过程说出来，减少跳步错误。对于需要结构化输出的场景，我用Pydantic定义Schema，加上2-3次自动重试，保证100%输出正确格式的JSON。
> 
> 我还会用机器学习里正则化的思想来优化Prompt：用L1正则化删掉没用的废话，用Dropout思想做示例多样化，用早停思想避免约束够了就不再加，不要无限叠加。
> 
> 另外安全方面我也有经验，我知道5种提示注入攻击手法和对应的4种防护方案，包括输入过滤、提示词加固、输出校验等等。"
