## 1. 简介
在Unix/Linux系统中，重定向是一种强大的机制，允许我们改变命令的输入来源和输出目的地。通过重定向，我们可以将命令的输出保存到文件中，将文件内容作为命令的输入，或者将一个命令的输出传递给另一个命令。

重定向是shell编程和系统管理中不可或缺的工具，掌握重定向技术可以极大地提高工作效率，简化复杂任务的实现。

## 2. 重定向的基本概念
重定向（Redirection）是指改变命令默认的输入、输出设备。在Linux系统中，每个进程默认打开三个文件描述符：

- 标准输入（stdin）：通常是键盘，文件描述符为0
- 标准输出（stdout）：通常是终端，文件描述符为1
- 标准错误（stderr）：通常是终端，文件描述符为2

重定向的作用就是改变这些默认的输入输出位置，使其指向文件或其他设备。

## 3. 标准文件描述符
文件描述符（File Descriptor，简称FD）是Linux系统中用于标识打开文件的非负整数。所有的I/O操作都是通过文件描述符完成的。

除了前面提到的三个标准文件描述符外，Linux系统还允许使用其他文件描述符（3及以上），但这些需要显式创建。

可以通过`lsof`命令查看进程打开的文件描述符：
```bash
lsof -p $$  # 查看当前shell进程打开的文件描述符
```

## 4. 输出重定向
输出重定向是最常用的重定向操作，它允许我们将命令的输出保存到文件中，而不是显示在终端上。

### 4.1 标准输出重定向
使用`>`操作符可以将标准输出重定向到文件。如果文件不存在，会创建该文件；如果文件已存在，会覆盖原有内容。

语法：
```bash
command > filename
```

示例：
```bash
# 将ls命令的输出保存到filelist.txt
ls > filelist.txt

# 将echo命令的输出保存到greeting.txt
echo "Hello, World!" > greeting.txt
```

### 4.2 标准错误重定向
标准错误重定向使用`2>`操作符（2是标准错误的文件描述符）。

语法：
```bash
command 2> errorfile
```

示例：
```bash
# 将错误信息保存到error.log
ls non_existent_file 2> error.log
```

### 4.3 同时重定向标准输出和错误
有多种方法可以同时重定向标准输出和错误：

1. 使用`&>`操作符（Bash特定语法）：
```bash
command &> output.log
```

2. 使用`>&`操作符：
```bash
command > output.log 2>&1
```

3. 使用POSIX兼容语法：
```bash
command > output.log 2> output.log
```

注意：第三种方法不推荐，因为可能会导致输出内容交错或丢失。

## 5. 输入重定向
输入重定向允许我们将文件内容作为命令的输入，而不是从键盘输入。使用`<`操作符。

语法：
```bash
command < filename
```

示例：
```bash
# 统计文件行数
wc -l < file.txt

# 将文件内容作为邮件发送
mail -s "Report" user@example.com < report.txt
```

## 6. 追加重定向
使用`>>`操作符可以将输出追加到文件末尾，而不是覆盖原有内容。

语法：
```bash
command >> filename
```

示例：
```bash
# 将当前日期追加到log.txt
date >> log.txt

# 将错误信息追加到error.log
command_that_may_fail 2>> error.log
```

## 7. 重定向操作符详解
Linux提供了多种重定向操作符，用于不同的场景：

| 操作符 | 描述 |
|--------|------|
| `>` | 将标准输出重定向到文件，覆盖原有内容 |
| `>>` | 将标准输出追加到文件末尾 |
| `<` | 从文件读取输入 |
| `2>` | 将标准错误重定向到文件，覆盖原有内容 |
| `2>>` | 将标准错误追加到文件末尾 |
| `&>` | 将标准输出和错误同时重定向到文件 |
| `>&` | 将一个文件描述符重定向到另一个 |
| `<<` | Here Document 语法开始 |
| `<<<` | Here String 语法 |

## 8. Here Document
Here Document（简称HereDoc）允许我们在命令行中指定多行输入，通常用于向交互式命令提供输入或创建包含多行内容的文件。

语法：
```bash
command << DELIMITER
line1
line2
...
DELIMITER
```

其中DELIMITER是一个自定义的分隔符，用于标识输入的开始和结束。

示例：
```bash
# 创建包含多行内容的文件
cat > example.txt << EOF
This is a Here Document example.
It allows multiple lines of input.
EOF

# 向ftp命令提供输入
ftp -n ftp.example.com << END_SCRIPT
user username password
binary
get file.zip
quit
END_SCRIPT
```

注意：如果DELIMITER前面加上`-`（如`<<-DELIMITER`），则输入中的前导制表符（Tab）会被忽略，这有助于保持脚本的可读性。

## 9. Here String
Here String是Here Document的简化形式，用于传递单行字符串作为输入。使用`<<<`操作符。

语法：
```bash
command <<< string
```

示例：
```bash
# 统计字符串长度
echo -n "Hello" | wc -c  # 传统方式
wc -c <<< "Hello"        # Here String方式

# 搜索字符串中的模式
grep "world" <<< "hello world example"
```

## 10. 实际应用示例

### 10.1 日志记录
将命令的输出和错误同时记录到日志文件，同时在终端显示：
```bash
command | tee output.log  # 只记录stdout
command 2>&1 | tee output.log  # 同时记录stdout和stderr
```

### 10.2 丢弃输出
有时我们需要执行命令但不关心其输出，可以将输出重定向到`/dev/null`：
```bash
command > /dev/null 2>&1  # 丢弃所有输出
```

### 10.3 文件描述符复制
使用`exec`命令可以永久重定向当前shell的文件描述符：
```bash
# 保存当前stdout到文件描述符3
exec 3>&1
# 将stdout重定向到log.txt
exec > log.txt
# 执行一些命令，输出会写入log.txt
command1
command2
# 恢复stdout
exec 1>&3
# 关闭文件描述符3
exec 3>&-
```

### 10.4 进程替换
使用`>(command)`语法可以将输出重定向到另一个命令（进程替换）：
```bash
# 同时将输出保存到文件并显示在终端
ls -l | tee >(grep "^d" > directories.txt) >(grep "^-" > files.txt)
```

## 11. 高级技巧

### 11.1 临时文件重定向
使用`mktemp`创建临时文件进行重定向：
```bash
tempfile=$(mktemp)
command > "$tempfile"
# 使用临时文件内容
echo "Command output: $(cat "$tempfile")"
# 删除临时文件
rm "$tempfile"
```

### 11.2 命名管道重定向
使用命名管道（FIFO）进行进程间通信：
```bash
mkfifo mypipe
# 在后台运行一个读取管道的进程
while read line; do echo "Received: $line"; done < mypipe &
# 向管道写入数据
echo "Hello from another process" > mypipe
# 清理
rm mypipe
```

### 11.3 重定向到多个文件
使用`tee`命令可以将输出同时发送到多个文件：
```bash
command | tee file1.txt file2.txt file3.txt
# 追加模式
command | tee -a file1.txt file2.txt
```

### 11.4 自定义文件描述符
除了标准文件描述符外，我们还可以使用自定义文件描述符：
```bash
# 创建新的文件描述符并指向文件
exec 4> custom_output.txt
# 使用自定义文件描述符
echo "This goes to custom file descriptor" >&4
# 关闭文件描述符
exec 4>&-
```

## 12. 常见问题与解决方案

### 12.1 权限问题
**问题**：重定向时出现"Permission denied"错误。
**原因**：没有目标文件或目录的写入权限。
**解决方案**：
- 检查文件权限
- 使用sudo（谨慎使用）
- 更改目标路径到有权限的位置

```bash
# 错误示例
echo "test" > /root/test.txt  # 普通用户无/root目录写入权限

# 正确示例
echo "test" > ~/test.txt  # 写入用户自己的家目录
```

### 12.2 背景进程输出
**问题**：后台进程的输出污染终端。
**原因**：后台进程的stdout/stderr仍连接到终端。
**解决方案**：将后台进程的输出重定向到文件：

```bash
long_running_command > output.log 2>&1 &
```

### 12.3 重定向顺序
**问题**：重定向顺序错误导致不符合预期的结果。
**原因**：重定向操作符的顺序很重要。
**解决方案**：记住正确的顺序：

```bash
# 正确：先重定向stdout，再将stderr重定向到stdout
command > output.log 2>&1

# 错误：顺序颠倒
command 2>&1 > output.log
```

### 12.4 重定向与管道混用
**问题**：尝试将重定向与管道结合使用时出现问题。
**原因**：管道和重定向的优先级和结合方式需要正确理解。
**解决方案**：使用子shell或进程替换：

```bash
# 将管道输出重定向到文件
(command1 | command2) > output.log

# 将多个命令的输出重定向到同一文件
{
  command1
  command2
  command3
} > combined_output.log
```

### 12.5 缓冲区问题
**问题**：重定向的输出没有立即出现在文件中。
**原因**：标准输出在重定向到文件时是块缓冲的，而不是行缓冲。
**解决方案**：使用`stdbuf`命令改变缓冲行为：

```bash
# 行缓冲模式
stdbuf -oL command > output.log

# 无缓冲模式
stdbuf -o0 command > output.log
```

## 13. 经验总结

### 13.1 最佳实践

1. **始终检查重定向目标**：在使用`>`操作符前，确认目标文件是否重要，避免意外覆盖。

2. **优先使用追加重定向**：不确定时，优先使用`>>`代替`>`以避免数据丢失。

3. **错误处理**：在编写脚本时，始终考虑错误处理和日志记录：
```bash
#!/bin/bash
LOG_FILE="script.log"
error() {
  echo "[$(date +'%Y-%m-%dT%H:%M:%S%z')] ERROR: $*" >&2
}

# 记录所有输出和错误
exec > >(tee -a "$LOG_FILE") 2>&1

echo "Script started"
# 执行命令
if ! command; then
  error "Command failed"
  exit 1
fi
```

4. **使用`set -euo pipefail`**：在脚本开头设置这些选项可以使脚本更健壮：
```bash
#!/bin/bash
set -euo pipefail
```
- `-e`: 命令失败时退出
- `-u`: 引用未定义变量时出错
- `-o pipefail`: 管道中任何命令失败则整个管道失败

5. **日志轮转**：对于长期运行的进程，考虑使用日志轮转工具如`logrotate`防止日志文件过大。

### 13.2 常见陷阱与规避

1. **覆盖重要文件**：
   - 陷阱：`command > /etc/passwd` 这样的命令会覆盖系统关键文件
   - 规避：使用`set -C`（禁止覆盖已存在文件）或`noclobber`选项
   ```bash
   set -o noclobber  # 启用noclobber选项
   command > file.txt  # 如果file.txt存在则失败
   command >| file.txt  # 强制覆盖（即使noclobber启用）
   ```

2. **混合使用重定向和管道**：
   - 陷阱：`command > file.txt | grep pattern` 这样的命令不会按预期工作
   - 规避：理解操作符优先级，必要时使用子shell
   ```bash
   (command | grep pattern) > file.txt
   ```

3. **后台进程输出未重定向**：
   - 陷阱：`command &` 后台进程仍会向终端输出
   - 规避：总是为重定向后台进程输出
   ```bash
   command > output.log 2>&1 &
   ```

4. **忘记恢复文件描述符**：
   - 陷阱：使用`exec`重定向后忘记恢复，影响后续命令
   - 规避：保存和恢复文件描述符
   ```bash
   exec 3>&1  # 保存stdout
   exec > output.log
   # 执行命令
   exec 1>&3  # 恢复stdout
   exec 3>&-  # 关闭文件描述符3
   ```

## 14. 底层原理剖析

### 14.1 文件描述符与重定向机制
在Linux系统中，重定向的实现依赖于文件描述符的复制和重定向系统调用。当我们执行`command > file.txt`时，shell会执行以下步骤：

1. 调用`open()`系统调用打开file.txt，获取文件描述符
2. 调用`dup2(new_fd, 1)`将标准输出（文件描述符1）重定向到新打开的文件
3. 调用`execve()`执行command命令
4. 命令的所有标准输出会写入到新的文件描述符指向的文件

### 14.2 相关系统调用

#### dup2()系统调用
`dup2()`系统调用用于复制文件描述符，是实现重定向的核心：
```c
#include <unistd.h>
int dup2(int oldfd, int newfd);
```
`dup2()`会关闭`newfd`（如果已打开），然后将`newfd`指向`oldfd`引用的文件。

示例代码（模拟`command > file.txt`）：
```c
#include <stdio.h>
#include <stdlib.h>
#include <unistd.h>
#include <fcntl.h>

int main() {
    int fd = open("file.txt", O_WRONLY | O_CREAT | O_TRUNC, 0644);
    if (fd == -1) {
        perror("open");
        exit(EXIT_FAILURE);
    }

    // 将标准输出重定向到文件
    if (dup2(fd, STDOUT_FILENO) == -1) {
        perror("dup2");
        exit(EXIT_FAILURE);
    }
    close(fd);  // 现在stdout指向文件，可以关闭原fd

    // 执行命令（这里用printf模拟）
    printf("This will be written to file.txt\n");
    return 0;
}
```

#### open()系统调用
`open()`系统调用用于打开文件并返回文件描述符：
```c
#include <fcntl.h>
int open(const char *pathname, int flags, mode_t mode);
```
在重定向中，常用的flags包括：
- `O_WRONLY`: 只写模式
- `O_CREAT`: 文件不存在则创建
- `O_TRUNC`: 截断文件（清空内容）
- `O_APPEND`: 追加模式
- `O_RDONLY`: 只读模式（用于输入重定向）

### 14.3 Shell中的重定向实现
以Bash为例，当解析到重定向操作符时，shell会：

1. 对重定向目标进行语法分析
2. 打开/创建目标文件
3. 调用`dup2()`重定向文件描述符
4. 执行命令
5. 恢复原始文件描述符

Bash源码中处理重定向的相关函数位于`redir.c`文件中，主要函数包括：
- `make_redirection()`: 创建重定向结构
- `do_redirection()`: 执行重定向
- `restore_redirection()`: 恢复重定向前的状态

### 14.4 重定向与fork/exec流程
当shell执行带重定向的命令时，完整流程如下：

1. shell调用`fork()`创建子进程
2. 在子进程中：
   a. 打开目标文件
   b. 调用`dup2()`重定向文件描述符
   c. 调用`execve()`执行命令
3. 父进程等待子进程完成

这种机制确保重定向只影响子进程，不会改变shell本身的文件描述符。
