写完上篇 MCP 协议之后，有读者问了一个好问题：**如果我有多个 Agent 要互相配合，该怎么让它们通信？**

这就是今天要聊的——Google 的 A2A（Agent-to-Agent）协议。

* * *

## 一、MCP 解决的是工具调用，A2A 解决的是 Agent 协作

先厘清一个常见误解。

MCP 是**单 Agent 调用外部工具** 的标准协议。你的本地 Ollama 模型通过 MCP 去读文件、查数据库、操作浏览器——这是一对多的星型结构，Agent 是中心，外面的工具是被调用者。

但如果你有多个 Agent，比如：

  * 一个负责研究查资料
  * 一个负责写文案
  * 一个负责审核合规

这三个 Agent 之间怎么分配任务、传递结果、保持状态？**MCP 没有解决这个** ，它只定义了工具调用，不定义 Agent 之间的协作。

这就是 A2A 存在的意义。

* * *

## 二、A2A 是什么？

A2A（Agent-to-Agent Protocol）由 Google 在 2025 年 4 月发起，2026 年 4 月 9 日发布 v1.0 稳定版，现由 Linux Foundation 托管。

**一句话定义** ：让不同框架、不同厂商构建的 AI Agent 能够互相发现、传递任务、协同工作的开放协议。

类比理解：**MCP 是 USB 接口，让你的电脑连接各种外设；A2A 是网络协议，让你的电脑和其他电脑交换文件。**

## 四、核心概念：Agent Card + Task 委托

### 4.1 Agent Card：Agent 的自我介绍

每个支持 A2A 的 Agent 都会暴露一个 **Agent Card** ——一份 JSON 描述文件，包含：
    
    
    {  
      "name":"research-agent",  
    "description":"专门负责搜索和整理信息的Agent",  
    "capabilities":["web_search","content_extraction","summarization"],  
    "skills":["fact_checking","data_analysis"],  
    "endpoint":"https://agent.example.com/a2a"  
    }  
    

其他 Agent 可以查询这个 Card，了解这个 Agent 能干什么，然后决定是否委托任务。**这是去中心化协作的关键基础设施。**

###  4.2 Task 委托：跨 Agent 任务传递

A2A 的核心机制是**任务委托** ：
    
    
    研究Agent ←→ A2A ←→ 写作Agent  
                    ↑  
               任务结果返回  
    

当研究 Agent 完成了资料搜集，它把整理好的结果通过 A2A 发给写作 Agent，并附带任务状态（处理中/已完成/失败）。写作 Agent 收到后继续处理，不需要人工介入。

### 4.3 两种通信模式

**同步** ：发起方等待结果，像函数调用，适合快速查询。 **异步（SSE）** ：发起方提交任务后立即返回，任务完成后通过 Server-Sent Events 推送结果，适合长时任务。

## 五、为什么说 A2A 和 MCP 是互补的？

这是最重要的理解点。

  
| MCP| A2A  
---|---|---  
**解决什么问题**|  Agent → 工具| Agent → Agent  
**关系结构**|  星型（Agent 为中心）| 对等（Agent 之间互传）  
**谁来发起**|  Agent 调用工具| Agent 之间互相委托  
**状态管理**|  无状态，每次独立调用| 有状态，任务有完整生命周期  
**谁在做**|  Anthropic 主推| Google 主推，Linux Foundation 托管  
  
**架构上的正确理解** ：
    
    
           ┌─────────────────────────────────────┐  
           │         多Agent系统                  │  
           │                                     │  
           │  研究Agent ←─── A2A ───→ 写作Agent   │  
           │       ↑                             │  
           │       │ MCP                         │  
           │       ↓                             │  
           │  ┌─────────┐   ┌──────────────┐     │  
           │  │ 搜索API      │   │ 文档数据库    │     │  
           │  └─────────┘   └──────────────┘     │  
           └─────────────────────────────────────┘  
    

A2A 处理 Agent 之间的协调层，MCP 处理每个 Agent 的工具访问层。**两个协议组合使用，才是完整的多 Agent 系统架构。**

##  六、生产环境怎么用？实战案例

### 场景：本地 AI 营销团队

用 Ollama 跑三个本地模型，组成一个营销团队：

  * **研究 Agent** ：用 Qwen3 8B，负责搜集竞品信息
  * **文案 Agent** ：用 Qwen2.5-Coder 7B（写作能力强），负责撰写文案
  * **审核 Agent** ：用 Gemma4 9B，负责合规检查

通过 A2A 协议串联：
    
    
    用户输入需求  
        ↓  
    研究Agent搜集资料（通过MCP调用搜索工具）  
        ↓ A2A 传递资料  
    文案Agent撰写初稿  
        ↓ A2A 传递草稿  
    审核Agent检查合规  
        ↓ A2A 返回修改意见  
    文案Agent修改  
        ↓  
    最终输出  
    

这个架构的优势：每个 Agent 可以用专门优化的模型，不需要一个模型搞定所有事。

### 本地搭建：A2A + Ollama

目前 Ollama 官方还未内置 A2A 支持，但可以通过 LangGraph 实现类似架构：
    
    
    from langgraph.prebuilt import create_react_agent  
    from langchain_mcp_adapters.tools import get_tools as get_mcp_tools  
      
    # 三个专业Agent  
    research_agent = create_react_agent(  
        model="ollama/qwen3:8b",  
        tools=get_mcp_tools("filesystem"),  # 研究需要读文件  
        state_system_message="你是一个专业的研究员"  
    )  
      
    writer_agent = create_react_agent(  
        model="ollama/qwen2.5-coder:7b",  
        tools=[],  
        state_system_message="你是一个专业文案"  
    )  
      
    review_agent = create_react_agent(  
        model="ollama/gemma4:9b",  
        tools=[],  
        state_system_message="你是一个严格的合规审核员"  
    )  
      
    # A2A 风格的任务传递  
    defrun_marketing_pipeline(topic):  
        research_result = research_agent.invoke({"messages": [topic]})  
        draft = writer_agent.invoke({  
            "messages": [f"根据以下资料写一篇营销文案：{research_result}"]  
        })  
        review_result = review_agent.invoke({  
            "messages": [f"审核以下文案并给出修改意见：{draft}"]  
        })  
        return review_result

## 七、A2A vs MCP：什么时候用哪个？

场景| 推荐协议  
---|---  
单个 Agent 查询数据库| MCP  
单个 Agent 操作浏览器| MCP  
多个 Agent 协作分工| A2A  
Agent 需要知道彼此的状态和进度| A2A  
需要跨厂商 Agent 互操作| A2A  
工具是固定能力，Agent 是使用者| MCP  
  
**简单判断** ：

  * **工具调用** → MCP
  * **任务委托** → A2A
  * **完整系统** → 两者都要

## 八、A2A 路线图展望

A2A v1.0 之后，下一步重点：

  1. **Agent Payments Protocol（AP2）** ：Agent 之间可以互相支付，用于自动化服务交易
  2. **Registry 标准化** ：Agent Card 的注册和发现机制
  3. **跨协议互操作** ：A2A 和 MCP 的更深层次整合规范
  4. **企业安全标准** ：签名 Agent Card、审计日志

## 九、我的判断

A2A 是 2026 年多 Agent 系统的事实标准，但**对普通个人用户来说，入门门槛比 MCP 高** 。

原因：MCP 可以单 Agent 独立运作，装一个 server 就能用；A2A 需要至少两个 Agent 才能体现价值，系统复杂度陡然上升。

**建议的上手路径** ：

  1. 先用 MCP 把单 Agent + 工具的架构跑通
  2. 有了 2-3 个专业 Agent 的需求，再引入 A2A
  3. 关注 LangGraph、CrewAI 这些已经内置 A2A 支持的框架
