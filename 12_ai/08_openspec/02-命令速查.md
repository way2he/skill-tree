# 02 - OpenSpec 命令速查

> Day 2 学习笔记 · 基于 v1.3.1 · 全部命令均在 Demo A 项目实测

## 📊 命令全景图

```
openspec
├── init          初始化项目
├── update        更新指令文件
├── list          列出 changes / specs        ⭐ 高频
├── view          交互式 dashboard
├── show          查看 change / spec          ⭐ 高频
├── status        查看 change 完成进度        ⭐ 神器
├── validate      验证 change / spec          ⭐ 高频
├── instructions  生成 artifact 模板          ⭐ 必用
├── archive       归档已完成 change           ⭐ 高频
├── new
│   └── change    创建新 change
├── change        (子命令组，部分已 deprecated)
│   ├── show
│   ├── list      (deprecated → openspec list)
│   └── validate
├── spec          (子命令组，已 deprecated)
│   ├── show      → openspec show
│   ├── list      → openspec list --specs
│   └── validate  → openspec validate --specs
├── config        全局配置管理
│   ├── path / list / get / set / unset / reset / edit / profile
├── schema        工作流 schema 管理（实验）
│   ├── which / validate / fork / init
├── schemas       列出可用 schemas
├── templates     查看 artifact 模板路径
├── completion    shell 自动补全
└── feedback      提交反馈
```

## 🎯 高频 7 命令（占 90% 使用场景）

### 1. `openspec init` — 初始化

```powershell
openspec init                  # 交互式
openspec init --tools none     # 不绑 AI
openspec init --tools claude   # 绑 Claude Code
openspec init --force          # 自动清理遗留文件
openspec init --profile custom # 用自定义 profile
```

支持的 AI 工具（27 个）：amazon-q, antigravity, auggie, bob, claude, cline, codex, forgecode, codebuddy, continue, costrict, crush, cursor, factory, gemini, github-copilot, iflow, junie, kilocode, kiro, opencode, **pi**, qoder, lingma, qwen, roocode, trae, windsurf

### 2. `openspec new change <name>` — 新建 change

```powershell
openspec new change add-export
openspec new change add-export --description "导出 JSON/CSV"
openspec new change add-export --schema spec-driven  # 显式指定 schema
```

只生成 `openspec/changes/<name>/.openspec.yaml`，**不生成模板文件**。

### 3. `openspec list` — 列表查询

```powershell
openspec list                  # 默认列 changes
openspec list --changes        # 显式列 changes
openspec list --specs          # 列 specs
openspec list --sort name      # 按名称排序（默认 recent）
openspec list --json           # JSON 输出
```

实测输出：
```
Changes:
  add-export     No tasks      just now

Specs:
  todo-parser     requirements 2
```

⚠️ **坑**：`list --specs --json` 不真的吐 JSON，只对 changes 生效。

### 4. `openspec status --change <name>` — 进度查询（**神器**）

```powershell
openspec status --change add-export
```

实测输出：
```
Change: add-export
Schema: spec-driven
Progress: 0/4 artifacts complete

[ ] proposal
[-] design (blocked by: proposal)
[-] specs (blocked by: proposal)
[-] tasks (blocked by: design, specs)
```

**核心价值**：清楚告诉你「**下一步该写哪个 artifact**」+ 「哪些被阻塞」。比 `validate` 更友好。

### 5. `openspec instructions <artifact> --change <name>` — 拿模板

```powershell
openspec instructions proposal --change add-export
openspec instructions specs --change add-export
openspec instructions design --change add-export
openspec instructions tasks --change add-export
openspec instructions tasks --change add-export --json  # 给 AI 用
```

输出格式：`<artifact>` XML 包裹的指令 + `<template>` 模板 + `<dependencies>` 前置文件路径。

### 6. `openspec validate [change|spec]` — 验证

```powershell
openspec validate add-parser              # 验证某个
openspec validate --all                   # 全部
openspec validate --changes               # 所有 changes
openspec validate --specs                 # 所有 specs
openspec validate --strict                # 严格模式
openspec validate --all --json            # JSON 输出（CI 友好）
openspec validate --concurrency 12        # 并发数
```

JSON 输出适合 CI 集成，关键字段：`items[].valid` / `summary.totals.failed`。

### 7. `openspec archive <change> [--yes]` — 归档

```powershell
openspec archive add-parser --yes               # 跳过确认
openspec archive add-parser --skip-specs        # 不合并 spec（文档/工具类 change）
openspec archive add-parser --no-validate       # 跳过验证（不推荐）
```

archive 默认强制 validate 通过，`--no-validate` 会二次确认。

## 🔎 查看与调试命令

### `openspec show [item-name]`

```powershell
openspec show todo-parser                    # 查看 spec（交互式选择 markdown/JSON）
openspec show todo-parser --json             # 直接 JSON
openspec show add-parser --type change       # 强制类型为 change
openspec show todo-parser --json --requirement 1   # 只看第 1 个 requirement
openspec show todo-parser --json --no-scenarios    # JSON 去掉 scenario
openspec show add-parser --deltas-only --json      # 只看 deltas（change 专用）
```

⚠️ **重要**：`show` 不支持已归档的 change（archive 后的 `2026-05-27-add-parser` 用 show 找不到，只能手动看目录）。

### `openspec view` — 交互式 dashboard

```powershell
openspec view
```

打开 TUI 风格的浏览器，键盘导航 changes 和 specs。**适合**：项目大、想快速浏览全貌。**不适合**：脚本化、CI。

### `openspec spec` (deprecated 子命令组)

```powershell
openspec spec show todo-parser     # → openspec show todo-parser
openspec spec list                  # → openspec list --specs
openspec spec validate todo-parser  # → openspec validate todo-parser
```

会打印 deprecation warning，**推荐用 verb-first 形式**。

## ⚙️ 配置管理

```powershell
openspec config path                       # 查配置文件位置
                                            # → %APPDATA%/openspec/config.json
openspec config list                       # 看当前所有设置
openspec config get delivery               # 拿单个值
openspec config set delivery both          # 设置
openspec config unset delivery             # 删除（回归默认）
openspec config reset --yes                # 全部重置
openspec config edit                       # 用 $EDITOR 打开
openspec config profile core               # 切 profile
```

实测 config list 输出：
```
featureFlags: {}
profile: core
delivery: both
telemetry:
  noticeSeen: true
  anonymousId: <uuid>

Profile settings:
  profile: core (default)
  workflows: propose, explore, apply, archive (from core profile)
```

## 🧩 Schema 管理（实验功能）

```powershell
openspec schemas                            # 列出可用 schemas（默认只有 spec-driven）
openspec schema which spec-driven           # 看某 schema 解析自哪里
openspec schema validate spec-driven        # 验证 schema 结构
openspec schema fork spec-driven my-schema  # 复制到项目本地以便定制
openspec schema init my-schema              # 创建新 schema
openspec templates                          # 看所有 artifact 模板路径
openspec templates --json                   # JSON 输出
```

实测 `templates` 输出：
```
Schema: spec-driven
Source: package
proposal: <npm-path>/templates/proposal.md
specs:    <npm-path>/templates/spec.md
design:   <npm-path>/templates/design.md
tasks:    <npm-path>/templates/tasks.md
```

这就是 `instructions` 命令背后的模板源文件，**可以直接 cat 看**。

## 🔧 其他工具命令

```powershell
openspec update                        # 更新 init 时生成的 AI 指令文件
openspec update --force                # 强制覆盖
openspec completion bash               # 生成 bash 补全脚本
openspec completion powershell         # PowerShell 补全（Win 用）

```

## 📐 典型工作流图

```
[1] openspec init [--tools <ai>]
         ↓
[2] openspec new change <name>
         ↓
[3] openspec instructions proposal --change <name>
    （拿模板 → 手动 / AI 填 proposal.md）
         ↓
[4] openspec status --change <name>        # 看下一步该写啥
         ↓
[5] openspec instructions specs --change <name>
    （写 specs/<capability>/spec.md）
         ↓
[6] openspec instructions tasks --change <name>
    （写 tasks.md）
         ↓
[7] openspec validate <name>
    （不通过则改，循环）
         ↓
[8] 写代码 + 测试通过
         ↓
[9] openspec archive <name> --yes
    （change 移到 archive/，spec merge 到 openspec/specs/）
```

**design.md 何时写**：当 change 涉及架构决策、技术选型、跨多个 capability 协作时写。简单 change 跳过即可（status 不会强制）。

## 🎯 Day 2 三大心得

### 心得 1：`status` 是被严重低估的命令
Day 1 没注意它。实际上 `status --change <name>` 是**最友好的进度追踪器**——比 validate 早一步告诉你下一步该做啥，新手必备。

### 心得 2：verb-first 形式优先
旧的 `openspec spec/change` 子命令组都打 deprecation warning。统一用 `openspec show/list/validate`，参数加 `--type` 或 `--specs` 区分。

### 心得 3：归档后的 change 是「只读化石」
`show <归档名>` 直接报 `Unknown item`。要查归档历史**必须手动浏览** `openspec/changes/archive/<日期>-<name>/`。这意味着归档前最好打个 git tag 或 commit message 注明。

## 🔑 一行流速查卡

```powershell
# 新项目启动
openspec init --tools none

# 新功能完整生命周期
openspec new change <name>
openspec status --change <name>      # 看下一步
openspec instructions proposal --change <name>
# (填 proposal.md)
openspec instructions specs --change <name>
# (填 specs/<cap>/spec.md)
openspec validate <name>
# (写代码 + 测试)
openspec archive <name> --yes

# 项目全局体检
openspec validate --all --json | jq '.summary'
openspec list && openspec list --specs
```
