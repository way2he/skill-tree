# 05 - OpenSpec 与 AI 工具集成

> Day 5 学习笔记 · 阶段 B 收官篇 · 实战耗时 2.5h
> 本篇核心：让 OpenSpec 从「独立 CLI 工具」升级为「AI 编辑器的协作底座」。

## 🎯 为什么需要 AI 工具集成？

裸用 OpenSpec CLI 的痛点：
- ❌ 写 proposal / spec 要全手敲，AI 帮不上忙
- ❌ AI 不知道 spec 存在，写代码时仍在「vibe coding」
- ❌ change 状态、依赖关系靠记忆，跨会话经常断片
- ❌ 多个 AI 工具切换时 context 丢失，每次都要重新喂

**集成的核心价值**：
1. AI 编辑器自动加载 `openspec/` 目录作为上下文（system prompt 级）
2. AI 知道当前 change 在哪一阶段，能主动推进下一步
3. AI 按 spec 的 Scenario 生成代码、写测试
4. 多人 / 多 agent 协作时共享同一份 spec，杜绝臆想

## 📋 OpenSpec 支持的 AI 工具全清单

截止 v1.3.x，OpenSpec `init --tools` 官方支持 27 个工具，可分 4 类：

### 第一梯队：原生 OpenSpec 协议支持（开箱即用）
| 工具 | 标识符 | 写入位置 | 集成深度 |
|------|--------|----------|----------|
| Claude Code | `claude` | `.claude/config.json` | ⭐⭐⭐⭐⭐ |
| Cursor | `cursor` | `.cursor/rules/openspec.mdc` | ⭐⭐⭐⭐⭐ |
| Codex CLI | `codex` | `.codex/instructions.md` | ⭐⭐⭐⭐ |
| Trae | `trae` | `.trae/rules.md` | ⭐⭐⭐⭐ |
| Windsurf | `windsurf` | `.windsurfrules` | ⭐⭐⭐⭐ |

### 第二梯队：通用规则文件注入
| 工具 | 标识符 | 写入位置 |
|------|--------|----------|
| CodeBuddy | `codebuddy` | `.codebuddy/rules.md` |
| Continue | `continue` | `.continue/config.json` |
| Cline | `cline` | `.clinerules` |
| Aider | `aider` | `.aider.conf.yml` |
| Roo Code | `roo` | `.roo/rules.md` |

### 第三梯队：IDE 插件 / Copilot 类
- GitHub Copilot：通过 `.github/copilot-instructions.md`
- JetBrains AI：通过项目级 `system_prompt.md`
- Zed AI：通过 `.zed/ai.json`

### 第四梯队：通用 LLM CLI
- `none` —— 不绑定任何工具，纯框架模式
- `all` —— 全部生成（不推荐，会污染目录）

**查看完整列表**：
```bash
openspec init --tools help
```

## 🥇 Claude Code 集成详解（强烈推荐）

Claude Code 是 OpenSpec 集成最深的工具，原因：
- 双向感知：spec 变动会立刻反映到对话上下文
- 内建 `/openspec` 斜杠命令
- 自动调用 `openspec validate`、`openspec status`

### 1. 初始化绑定
```bash
cd <your-project>
openspec init --tools claude
```

执行后会生成 / 修改：
```
.claude/
├── config.json              ← 项目级 AI 配置
└── system-prompts/
    └── openspec.md          ← 注入 OpenSpec 协议规则
openspec/
├── AGENTS.md                ← AI 行为约束（关键文件！）
└── ...
```

### 2. 核心机制：`AGENTS.md`
这是 OpenSpec 的「AI 行为契约」，告诉 AI：
- 写代码前先看 `openspec list` 当前活动 change
- 修 spec 必须用 ADDED / MODIFIED / REMOVED 三种 delta
- 写完 spec 必须跑 `openspec validate`
- archive 前必须确认代码与 spec 一致

**摘录片段**：
```markdown
## You MUST follow this workflow:
1. Before any code change, check active changes:
   `openspec list`
2. If no relevant change exists, create one:
   `openspec new change <kebab-name>`
3. Write proposal.md FIRST, get user sign-off
4. Then write specs/<capability>/spec.md
5. ONLY THEN write code
6. Run `openspec validate <name>` before commit
7. Run `openspec archive <name>` after merge
```

### 3. 实战：与 Claude 对话三段式
**❌ 反例（vibe coding）**：
> 「帮我加一个 export 功能」
> Claude 直接开始写代码 → 漏掉 JSON / CSV 区分 → 来回返工

**✅ 正例（SDD）**：
> 「我要给 mdtodo 加 export 能力，先创建 change」
> Claude → 自动 `openspec new change add-export`
> Claude → 生成 proposal.md draft，问你 review
> 你 → 「JSON 和 CSV 都要，CSV 要 UTF-8 BOM」
> Claude → 写 spec.md（含 2 个 Scenario）→ validate → 写代码 → 跑测试

### 4. 配置文件解读 `.claude/config.json`
```json
{
  "$schema": "https://claude.ai/schemas/project-config.json",
  "openspec": {
    "enabled": true,
    "auto_load_specs": true,
    "auto_validate_on_save": false,
    "active_change_in_context": true
  },
  "system_prompts": [
    "system-prompts/openspec.md"
  ]
}
```

| 字段 | 推荐值 | 说明 |
|------|--------|------|
| `auto_load_specs` | `true` | 每次对话自动注入全部 spec |
| `auto_validate_on_save` | `false` | 太频繁，手动跑更可控 |
| `active_change_in_context` | `true` | 把当前 change 的 proposal/spec 强制塞进 context |

## 🥈 Cursor / Trae / Codex 集成对比

### Cursor 集成
```bash
openspec init --tools cursor
```
生成 `.cursor/rules/openspec.mdc`，会被 Cursor 全局 system prompt 加载。

**优势**：
- Cursor `@` 引用直接补全 `@openspec/changes/<name>/proposal.md`
- Composer 模式下 spec 自动作为 multi-file context
- Cursor 0.43+ 支持 `.cursor/rules/*.mdc` 模块化规则

**踩坑**：
- ⚠️ Cursor 旧版用 `.cursorrules` 单文件，OpenSpec 0.9.x 之前的版本不会自动迁移
- ⚠️ Cursor 的 Auto 模式有时会忽略 rules，建议改用 Claude Sonnet 4.6

### Trae 集成（字节出品）
```bash
openspec init --tools trae
```
生成 `.trae/rules.md`。

**优势**：
- 国内访问稳定（不需要科学上网）
- 内置 DeepSeek V4 Pro / Qwen3-Max 等国产模型，成本低
- Builder 模式（类似 Cursor Composer）原生支持多 spec 上下文

**踩坑**：
- ⚠️ Trae 1.5 之前的版本 `.trae/rules.md` 是 beta，行为不稳定
- ⚠️ DeepSeek V4 Flash 对 SHALL/Scenario 语法理解略弱，关键 change 用 Pro 版

### Codex CLI 集成
```bash
openspec init --tools codex
```
生成 `.codex/instructions.md`，会被每次 `codex exec` 调用时注入。

**优势**：
- 命令行原生，CI/CD 友好（无需 GUI 编辑器）
- 适合「先 spec 后批量代码生成」的离线任务

**劣势**：
- 无交互式 review，spec 修改后要手动重新 codex run

### 对比矩阵
| 维度 | Claude Code | Cursor | Trae | Codex CLI |
|------|-------------|--------|------|-----------|
| spec 自动加载 | ✅ | ✅ | ✅ | ⚠️ 需指定 |
| 当前 change 主动推进 | ✅ 强 | ⚠️ 弱 | ✅ 中 | ❌ |
| 多文件 spec 引用 | ✅ | ✅ `@` 引用 | ✅ Builder | ✅ |
| 验证集成 (`validate`) | ✅ 自动调用 | ⚠️ 手动 | ⚠️ 手动 | ⚠️ 手动 |
| 国内可用性 | ❌ | ⚠️ | ✅ | ⚠️ |
| 成本（月） | $20 | $20 | ¥40 起 | 按 token |
| 适合场景 | 严肃项目 | 日常编码 | 国内 + 性价比 | CI/批处理 |

## 🎯 集成最佳实践

### 实践 1：单项目优先选 1 个主工具
多个 AI 工具同时启用会导致：
- 多套 rules 文件冲突
- 不同 AI 对同一 spec 有不同理解
- git diff 噪声大（每次都改一堆配置文件）

**推荐组合**：
- 严肃项目 / 团队：**Claude Code** + `--tools claude` 主导
- 个人快迭代 / 国内：**Trae** + `--tools trae`
- CI 流水线：**Codex** + `--tools codex`

### 实践 2：把 `openspec/AGENTS.md` 当作 SSOT
- 项目所有 AI 工具读同一份 AGENTS.md
- 不要让每个工具自定义独立的协议规则
- 改 AGENTS.md 时全员同步（git commit + 群内通知）

### 实践 3：spec 优先级 > 代码注释 > 历史对话
当 AI 回答冲突时，明确告诉它：
> 「以 `openspec/specs/<cap>/spec.md` 为准，代码和注释都过时了。」

### 实践 4：每次 commit 前手动跑 validate
即使 IDE 集成支持自动验证，也保留 git pre-commit hook：
```bash
# .git/hooks/pre-commit
#!/bin/sh
openspec validate --all || exit 1
```

### 实践 5：多 agent 协作时锁 capability
> 同一时间，同一 capability 只能有 1 个 active change。

用 git 分支 + change naming 约定避免冲突：
```
feature/add-export  → openspec change add-export
feature/fix-parse   → openspec change fix-parse-bug
```

## 🎯 Day 5 三大核心心得

### 心得 1：集成不是「装个插件」而是「契约换契约」
装工具只花 5 分钟，难的是统一团队/多 agent 的 AGENTS.md 契约。
**真正的成本**在协议对齐，而不是工具切换。

### 心得 2：spec 是 AI 的「不变量」
对话会断、context 会丢、模型会换，但 spec 永远在 git 里。
设计集成时永远问：**「如果对话 reset 一次，AI 还能继续吗？」**
答案是「能」才算集成成功。

### 心得 3：选工具优先级 = 集成深度 > 模型能力
Claude Sonnet 4.6 + 浅集成 < GLM-5.1 + 深度集成
因为前者每次都「忘记上下文」，后者每次都「拿到完整 spec」。
**SDD 时代，工具的协作能力比模型 IQ 更重要。**
