## Q1 ⭐ SDD 是什么 + 为何 AI 时代需要

### 核心答案
SDD（Spec-Driven Development，规范驱动开发）= 把「规范」当作真理来源（Source of Truth），代码沦为规范的可验证副产物。

传统开发：需求 → 代码（事实标准）→ 文档（往往过时）
SDD 开发：需求 → **规范（事实标准）**→ 代码（可重生成）→ 验证

### 为何 AI 时代突然需要
- METR 2026-02 研究：开发者用 AI 工具后实际速度 **慢了 19%**，但自我感觉「更快了」
- 根因：非结构化 prompt → AI 生成「看起来对」的代码 → 边缘 case 漏掉 → 调试循环吞噬时间
- SDD 把约束先写成结构化 spec，AI 按合约执行而非猜测

### 评分要点
- ✅ 答出「规范作为 Source of Truth」
- ✅ 指出代码是派生物 / 可验证产物
- ✅ 至少举一个 AI 痛点

### 加分项
- 提到 METR 数据 / Vibe Coding 陷阱
- 提到 spec 可作为 AI 的 system prompt

### 常见失分
- ❌ 只说「写文档驱动开发」（这是 DDD 不是 SDD）
- ❌ 把 SDD 和 TDD 混为一谈

## Q2 ⭐ SDD 的 3 个层级 + OpenSpec 位置

### 核心答案

| 层级 | 名称 | 规范角色 | 代码角色 | 典型工具 |
|------|------|----------|----------|----------|
| L1 | Spec-First | 写在前面但不强制 | 主产物 | Cursor `.cursorrules` |
| L2 | Spec-Anchored | 评审基线、变更必同步 | 主产物，需对齐 spec | **OpenSpec** / Spec-Kit |
| L3 | Spec-as-Source | 真理来源，code 可重生成 | 派生物 | Intent / Kiro |

**OpenSpec = L2 Spec-Anchored**：
- ✅ 每个 change 用 delta marker (ADDED/MODIFIED/REMOVED) 追踪
- ✅ archive 时自动合并到正式 spec
- ❌ 代码仍需手工/AI 写，不能从 spec 自动生成（不是 L3）

### 评分要点
- ✅ 三个层级名称完整
- ✅ 正确指出 OpenSpec = L2
- ✅ 说明 L2 与 L3 的关键区别（能否从 spec 自动生成代码）

### 加分项
- 引用 OpenReview 论文出处
- 给出 L1/L3 对应工具实例

## Q3 ⭐ init 到 archive 完整命令序列

### 核心答案

```bash
# 1. 初始化项目
openspec init --tools claude

# 2. 创建 change
openspec new change add-export

# 3. 查看下一步阻塞
openspec status --change add-export

# 4. 拿模板
openspec instructions proposal --change add-export
openspec instructions specs --change add-export

# 5. (手动写 proposal.md / spec.md / tasks.md)

# 6. 验证
openspec validate add-export

# 7. (实现代码 + 测试)

# 8. 归档
openspec archive add-export --yes

# 9. 验证合并
openspec list --specs
openspec show export
```

### 评分要点
- ✅ 命令顺序正确（init → new → status → instructions → validate → archive → list/show）
- ✅ 至少包含 `openspec validate`
- ✅ 注明每条用途

### 加分项
- 提到 `--yes` 跳过确认
- 提到 archive 后 change 从活跃列表消失

### 常见失分
- ❌ 漏掉 validate（直接 archive 会失败）
- ❌ 写错 `openspec change new` 顺序（正确是 `openspec new change`）

## Q4 ⭐⭐ `init --tools claude` 生成的文件及作用

### 核心答案

```
.claude/
├── config.json                 ← Claude Code 项目配置，开启 openspec 集成
└── system-prompts/
    └── openspec.md             ← 注入 OpenSpec 协议规则到 system prompt

openspec/
├── AGENTS.md                   ← AI 行为契约（最关键文件）
├── project.md                  ← 项目背景说明，AI 读它了解 domain
├── changes/
│   └── .gitkeep                ← 占位，存放进行中的 change
└── specs/
    └── .gitkeep                ← 占位，存放正式 spec 库
```

**关键作用区分**：
- 项目文件（`openspec/*`）：跨工具共享，git 提交进仓库
- AI 工具配置（`.claude/*`）：本工具专属，也建议 commit（团队同步）

### 评分要点
- ✅ 至少列 4 个文件
- ✅ 正确说出 AGENTS.md 是 AI 行为契约
- ✅ 区分 openspec 目录 vs .claude 目录

### 加分项
- 提到 `AGENTS.md` 是协议 SSOT、跨工具共用
- 说明 `project.md` 是给 AI 看的 domain knowledge

## Q5 ⭐⭐ Change 五阶段 + 典型踩坑

### 核心答案

| 阶段 | 目标 | 典型踩坑 |
|------|------|----------|
| 1. Proposal | 对齐「要不要做」 | ❌ 用「优化 / 增强」这种模糊词 |
| 2. Specs | 精确定义「做什么」 | ❌ 写实现细节（如「用正则匹配」） |
| 3. Design | 怎么实现（可选） | ❌ 小变更也写 Design 浪费时间 |
| 4. Tasks | 拆成可执行清单 | ❌ 单任务 > 2h 工作量，太粗 |
| 5. Archive | 合并 + 归档 | ❌ 没跑 validate 就 archive，导致 spec 库脏 |

### Design 何时可省略
- 单变更代码量 < 500 行
- 不改公共模块
- 不跨团队 / 不影响 DB / 不动外部 API

### 评分要点
- ✅ 五阶段名称完整且顺序正确
- ✅ 每阶段一个真实踩坑
- ✅ 答出 Design 省略条件

### 加分项
- 提到 Scenario 必须 WHEN/THEN 双段式
- 提到 archive 是单向门

## Q6 ⭐⭐⭐ 两个 change 同改 `user-auth`

### 核心答案

**方案 A：串行（推荐）**
- change-B 等 change-A archive 后再开始
- 代价：B 延迟
- 收益：避免 spec 冲突

**方案 B：合并为一个 change**
- 把 A + B 合并为一个 `change-AB-update-user-auth`
- 代价：单 change 范围变大，review 难度上升
- 收益：原子性强，不会一半合并

**方案 C（不推荐）：并行**
- 两个 change 同时跑
- 代价：archive 时 spec 文件冲突几乎必然
- 即使能合并，delta marker 也会冲突，git merge 痛苦

### 最终推荐
- 两个 change 独立 review 价值 > 合并 → 选 A 串行
- 两个 change 紧耦合（如「重构 + 加新 scenario」）→ 选 B 合并

### 评分要点
- ✅ 至少给出 2 种方案
- ✅ 分析每种代价
- ✅ 给出选择条件而非「无脑推荐」

### 加分项
- 提到「同一 capability 同一时间 1 个 active change」原则
- 提到用 CODEOWNERS / CI 防御

### 常见失分
- ❌ 直接说「并行就行，git merge 解决」（实际很痛苦）

## Q7 ⭐⭐ Cursor 集成

### 核心答案

**命令**：
```bash
openspec init --tools cursor
```

**生成文件**：`.cursor/rules/openspec.mdc`

**加载机制**：
- Cursor 0.43+ 自动加载 `.cursor/rules/*.mdc` 作为全局规则
- 注入到所有对话和 Composer 模式的 system prompt
- 支持 `@openspec/changes/<name>/proposal.md` 直接引用 spec 文件

**踩坑点**：
- ⚠️ Cursor < 0.43 用 `.cursorrules` 单文件，OpenSpec < 0.9.x 不会自动迁移到新格式
- ⚠️ Cursor 的 Auto 模式有时忽略 rules，关键 change 建议手动选 Claude Sonnet 4.6

### 评分要点
- ✅ 命令 + 文件位置正确
- ✅ 答出 mdc 模块化规则加载机制
- ✅ 至少一个踩坑

### 加分项
- 提到 `@` 引用 spec 文件
- 推荐 Sonnet 4.6 / Opus 4.7 处理复杂 change

## Q8 ⭐⭐⭐ 场景化选型（5 人 Gitea + 多 AI 工具 + 30000 行老项目）

### 核心答案：**选 OpenSpec**

### 主要理由
1. **项目在 Gitea 不在 GitHub** → Spec-Kit 直接出局
2. **3 种 AI 工具混用** → OpenSpec 27+ 工具兼容性最强
3. **30000 行老项目** → 不能一上来重构所有 spec
4. **5 人团队** → 需要协议 SSOT + git CODEOWNERS

### 渐进式接入策略（3 个月计划）

**Month 1：试点期**
- 仅 1 人在 1 个新模块用 OpenSpec
- 完成 2-3 个 change 跑通流程
- 沉淀团队 AGENTS.md 草稿

**Month 2：扩散期**
- 全员 init 自己的 AI 工具（cursor/trae/codex 各自 `--tools <x>`）
- 共用同一份 AGENTS.md
- 新需求强制走 OpenSpec，老代码不动

**Month 3：成熟期**
- 加 git pre-commit：`openspec validate --all`
- 加 PR 模板 checkbox
- 高频改动模块补 spec（按需，不强求覆盖）

### 评分要点
- ✅ 选 OpenSpec 且理由站得住
- ✅ 至少 3 个具体理由
- ✅ 给出渐进式接入而非「一次性重构」

### 加分项
- 量化 3 个月里程碑
- 提到「老代码顺其自然」而非强迫覆盖
- 提到 CODEOWNERS / CI 防御

### 常见失分
- ❌ 选 Spec-Kit（不在 GitHub 上）
- ❌ 建议「一次性补完所有 spec」（不切实际）
