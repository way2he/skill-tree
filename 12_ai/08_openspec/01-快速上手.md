# 01 - OpenSpec 快速上手

> Day 1 学习笔记 · 实战耗时 1.5h · 配套 Demo：`demo/markdown-todo-parser/`

## 一图看懂 OpenSpec

```
OpenSpec 核心理念：先写 Spec，再写 Code

传统流程：需求 → 直接写代码 → Bug → 改 → Bug → 重写
SDD 流程：需求 → 写 Spec → 评审 → AI/人按 Spec 写 → 验证

         openspec init
              ↓
       openspec/ 目录
       ├── changes/         （进行中的变更）
       ├── changes/archive/ （已归档的变更）
       └── specs/           （正式规格库）

         一个完整 Change 的生命周期：
   new → proposal → specs → tasks → 实现 → validate → archive
                                                          ↓
                                              spec 自动合并到 specs/
```

## 1. 安装

### 前置要求
- Node.js ≥ 20.19（OpenSpec 是 npm 全局工具）

```powershell
node --version    # 检查
npm install -g @fission-ai/openspec@latest
openspec --version   # 1.3.1
```

首次安装会提示 `OpenSpec collects anonymous usage stats. Opt out: OPENSPEC_TELEMETRY=0` — 想关遥测加这个环境变量。

## 2. init 初始化

```powershell
cd <你的项目目录>
openspec init                  # 交互式（会问要绑定哪些 AI 工具）
openspec init --tools none     # 非交互，不绑定任何 AI 工具（纯框架）
openspec init --tools claude   # 绑定 Claude Code（会写 .claude/config.json）
```

### 生成的目录结构
```
openspec/
├── changes/         ← 进行中的 change
├── changes/archive/ ← 已归档的 change
└── specs/           ← 正式规格库（archive 时自动合并）
```

⚠️ **重要发现**：init **不会**生成任何模板文件，只创建空目录。模板要靠 `openspec instructions <artifact>` 命令实时生成。

## 3. 五个核心命令（Day 1 够用）

| 命令 | 作用 |
|------|------|
| `openspec init` | 初始化项目结构 |
| `openspec new change <name>` | 创建一个新 change 目录 |
| `openspec instructions <artifact> --change <name>` | 输出某个 artifact 的填写指南和模板 |
| `openspec validate <change>` | 验证 change 是否合规 |
| `openspec archive <change> --yes` | 归档 change，合并 spec |

支持的 4 个 artifact：`proposal` / `specs` / `design` / `tasks`

## 4. 完整闭环实战（Demo A）

走一遍 `add-parser` change 的完整生命周期。

### Step 1: 创建 change
```powershell
openspec new change add-parser
# √ Created change 'add-parser' at openspec/changes/add-parser/
```
只生成一个 `.openspec.yaml` 元数据文件，其他要自己填。

### Step 2: 写 proposal
查模板：
```powershell
openspec instructions proposal --change add-parser
```
按模板填 `openspec/changes/add-parser/proposal.md`，核心 4 节：Why / What Changes / Capabilities / Impact。

### Step 3: 写 specs（每个 capability 一份 spec.md）

⚠️ **首次踩坑**：只写 proposal 跑 validate 会报错：
```
[ERROR] Change must have at least one delta.
```

必须在 `openspec/changes/add-parser/specs/todo-parser/spec.md` 写 delta：

```markdown
## ADDED Requirements

### Requirement: 解析 Markdown 文件提取 TODO 项
系统 SHALL 读取指定 markdown 文件，提取 `- [ ]` 和 `- [x]` 项。

#### Scenario: 文件含未完成 TODO
- **WHEN** 输入 markdown 包含 `- [ ] 写测试`
- **THEN** 返回项包含 `{status: "open", text: "写测试"}`
```

**关键格式规则**：
- 必须用 `## ADDED/MODIFIED/REMOVED/RENAMED Requirements` 开头
- 每个 `### Requirement:` 至少配 1 个 `#### Scenario:`
- Scenario 必须用**精确 4 个 `#`**，3 个会静默失败
- 用 SHALL/MUST，避免 should/may

### Step 4: 写 tasks（实施计划）
用 markdown 复选框列出步骤，按 `## 1. xxx` / `- [ ] 1.1 xxx` 格式。

### Step 5: validate + 写代码 + 测试

```powershell
openspec validate add-parser
# Change 'add-parser' is valid

py -3.12 -m pytest tests -q
# 6 passed in 0.03s
```

### Step 6: archive

```powershell
openspec archive add-parser --yes
# Specs to update: todo-parser: create
# Applying changes to openspec/specs/todo-parser/spec.md:
#   + 2 added
# Change 'add-parser' archived as '2026-05-27-add-parser'.
```

**archive 做了 3 件事**：
1. change 目录从 `changes/` 移到 `changes/archive/2026-05-27-add-parser/`
2. delta 自动合并到 `openspec/specs/todo-parser/spec.md`
3. 在文件名前加日期前缀防冲突

## 5. Day 1 三个核心心得

### 心得 1: OpenSpec 是「契约工具」，不是「代码生成器」
它不写代码，只**强制你先想清楚 What 再做 How**。代码靠你或 AI 写，但 spec 文件成为评审、测试、AI 提示的统一契约。

### 心得 2: 模板靠 `instructions` 命令拿
不像 cookiecutter 那样 init 时铺满模板。**好处**：升级 OpenSpec 不影响你已写的内容；**坏处**：第一次用容易懵，记住「想填什么 artifact 就 `openspec instructions <artifact>`」。

### 心得 3: validate 是你的安全网
每写完一个 artifact 立刻 `openspec validate <change>`，错误信息很精准。养成 TDD 式节奏。

## 6. 常见坑速查

| 现象 | 原因 | 修复 |
|------|------|------|
| `Change must have at least one delta` | 只写了 proposal，没写 specs/ | 必须创建 `specs/<capability>/spec.md` |
| Scenario 不被识别 | 用了 3 个 `#` | 改成 4 个 `####` |
| archive 后找不到 change | 移到 `changes/archive/<日期>-<name>/` 了 | 用 `openspec list` 看活跃，archive 目录手动浏览 |
| `instructions` 报 Missing artifact | 没传 artifact 名 | `openspec instructions proposal --change xxx` |
