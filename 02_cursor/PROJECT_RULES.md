# 项目规范文档

## 1. 代码规范

### 1.1 Python 代码规范
- 遵循 PEP 8 编码规范
- 使用 4 个空格进行缩进
- 行长度限制在 79 个字符以内
- 使用有意义的变量名和函数名
- 所有函数和类必须包含文档字符串（docstring）
- 使用类型注解（Type Hints）

### 1.2 命名规范
- 类名：使用 PascalCase（如 `MyClass`）
- 函数和变量：使用 snake_case（如 `my_function`）
- 常量：使用大写字母和下划线（如 `MAX_VALUE`）
- 私有成员：以下划线开头（如 `_private_var`）

## 2. 版本控制规范

### 2.1 Git 提交规范
提交信息格式：
```
<type>(<scope>): <subject>

<body>

<footer>
```

类型（type）：
- feat: 新功能
- fix: 修复 bug
- docs: 文档更新
- style: 代码格式（不影响代码运行的变动）
- refactor: 重构
- test: 测试相关
- chore: 构建过程或辅助工具的变动

### 2.2 分支管理
- main: 主分支，保持稳定
- develop: 开发分支
- feature/*: 功能分支
- hotfix/*: 紧急修复分支
- release/*: 发布分支

## 3. 文档规范

### 3.1 必要文档
- README.md：项目说明
- CHANGELOG.md：版本更新日志
- CONTRIBUTING.md：贡献指南
- LICENSE：开源协议

### 3.2 代码注释
- 关键算法和复杂逻辑必须添加注释
- 使用清晰的注释说明代码意图
- 及时更新注释，保持与代码同步

## 4. 测试规范

### 4.1 单元测试
- 测试覆盖率要求 > 80%
- 每个功能模块必须包含单元测试
- 测试用例命名清晰，表达测试目的

### 4.2 测试命名
- 测试类：`Test<被测试类名>`
- 测试方法：`test_<被测试方法名>_<测试场景>`

## 5. 安全规范

### 5.1 代码安全
- 敏感信息（如密钥、密码）必须使用环境变量或配置文件
- 避免硬编码敏感信息
- 定期更新依赖包，修复安全漏洞

### 5.2 数据安全
- 重要数据必须加密存储
- 实现适当的访问控制
- 定期备份数据

## 6. 性能规范

### 6.1 代码性能
- 避免不必要的循环和递归
- 合理使用缓存
- 优化数据库查询
- 控制内存使用

### 6.2 资源使用
- 及时释放资源（文件、数据库连接等）
- 使用上下文管理器（with 语句）
- 避免资源泄漏

## 7. 项目结构

### 7.1 目录结构
```
project/
├── src/                # 源代码
├── tests/              # 测试文件
├── docs/               # 文档
├── scripts/            # 脚本文件
├── requirements.txt    # 依赖包
└── README.md          # 项目说明
```

### 7.2 模块组织
- 相关功能放在同一模块
- 避免循环依赖
- 保持模块职责单一

## 8. 发布规范

### 8.1 版本号规范
遵循语义化版本（Semantic Versioning）：
- 主版本号：不兼容的 API 修改
- 次版本号：向下兼容的功能性新增
- 修订号：向下兼容的问题修正

### 8.2 发布流程
1. 更新版本号
2. 更新 CHANGELOG.md
3. 创建发布分支
4. 进行测试
5. 合并到主分支
6. 创建标签

## 9. 协作规范

### 9.1 代码审查
- 所有代码必须经过审查才能合并
- 审查重点：功能正确性、代码质量、性能影响
- 及时响应审查意见

### 9.2 问题跟踪
- 使用 Issue 跟踪问题和功能请求
- 清晰描述问题或需求
- 及时更新状态

## 10. 持续集成/持续部署

### 10.1 CI/CD 流程
- 自动运行测试
- 代码质量检查
- 自动构建和部署
- 环境一致性保证

### 10.2 自动化测试
- 单元测试
- 集成测试
- 端到端测试
- 性能测试
